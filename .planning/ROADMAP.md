### 🚧 v0.10.0 Multi-Agent Workspace Tooling (In Progress)

**Milestone Goal:** Make git-stacks the infrastructure layer for humans managing multiple AI agents — queryable workspace data for agent bootstrap, repo sync primitives, and composable templates.

- [ ] **Phase 37: Agent Path Discovery** - `git-stacks paths` command outputs workspace repo paths for agent CLI injection
- [ ] **Phase 38: Multi-Repo Pull** - `git-stacks pull` pulls latest across all repos in a workspace
- [ ] **Phase 39: TUI Upstream Staleness** - Dashboard shows "N behind" badge per repo with TTL-cached fetch
- [ ] **Phase 40: Template Composition** - `includes:` field and multi-`--template` ad-hoc composition
- [ ] **Phase 41: Release Prep** - v0.10.0 version bump, CHANGELOG, README

## Phase Details

### Phase 37: Agent Path Discovery
**Goal**: Users can query workspace repo paths for direct injection into agent CLI invocations
**Depends on**: Phase 36 (v0.9.0 shipped)
**Requirements**: PATH-01, PATH-02, PATH-03, PATH-04
**Success Criteria** (what must be TRUE):
  1. User can run `git-stacks paths myws` and receive one repo path per line on stdout
  2. User can pass `--prefix "--add-dir"` and receive each path prefixed with the flag string (e.g., `--add-dir /path/a`)
  3. Running `git-stacks paths` from inside a workspace worktree directory autodetects the workspace without an explicit argument
  4. Worktree repos emit `task_path`, trunk repos emit `main_path`; repos with missing task_paths are skipped with a stderr warning
**Plans**: 1 plan, 1 wave (3 tasks)

### Phase 38: Multi-Repo Pull
**Goal**: Users can pull latest commits for all repos in a workspace with a single command, with safe defaults that prevent data loss
**Depends on**: Phase 37
**Requirements**: PULL-01, PULL-02, PULL-03, PULL-04, PULL-05, PULL-06
**Success Criteria** (what must be TRUE):
  1. User can run `git-stacks pull myws` and all repos in the workspace are updated to their latest upstream commits
  2. Worktree repos pull their workspace branch; trunk repos pull their default branch
  3. Repos with uncommitted changes are skipped with a warning, and the command exits non-zero if any repo was skipped
  4. Pull uses `--ff-only`; a diverged branch produces a clear failure message identifying the repo and branch
  5. Running `git-stacks pull` from inside a workspace directory autodetects the workspace
**Plans**: TBD

### Phase 39: TUI Upstream Staleness
**Goal**: The dashboard workspace detail pane shows how many commits each repo is behind upstream, fetched non-blocking with a TTL cache
**Depends on**: Phase 38
**Requirements**: STALE-01, STALE-02, STALE-03, STALE-04, STALE-05
**Success Criteria** (what must be TRUE):
  1. Focusing a workspace in the dashboard triggers a background fetch and shows "N behind" per repo when upstream has newer commits
  2. Staleness data is cached per `main_path` and does not re-fetch within the TTL window (e.g., 5 minutes)
  3. User can press `r` on a focused workspace to force-refresh staleness, bypassing the TTL cache
  4. Repos without upstream tracking show a dash or blank — no error, no badge
  5. Network failures produce a `?` badge per affected repo and do not crash or block the TUI
**Plans**: TBD
**UI hint**: yes

### Phase 40: Template Composition
**Goal**: Templates can declare composable building blocks via `includes:`, and users can compose templates ad-hoc on `git-stacks new`
**Depends on**: Phase 39
**Requirements**: COMP-01, COMP-02, COMP-03, COMP-04, COMP-05, COMP-06, COMP-07
**Success Criteria** (what must be TRUE):
  1. A template YAML with `includes: [api, frontend]` merges repos from both referenced templates when creating a workspace
  2. `git-stacks new --template api --template frontend` creates a workspace with repos from both templates without requiring a saved meta-template
  3. When the same repo appears in multiple templates, worktree mode wins over trunk mode
  4. Hooks from included templates concatenate in include order, with the top-level template's hooks running last
  5. A circular `includes:` chain produces a clear error message and does not hang or crash
  6. Existing templates without `includes:` continue to load and work without changes
**Plans**: TBD

### Phase 41: Release Prep
**Goal**: v0.10.0 is tagged and documented with all new features covered in CHANGELOG and README
**Depends on**: Phase 40
**Requirements**: REL-01, REL-02, REL-03
**Success Criteria** (what must be TRUE):

  2. CHANGELOG contains an entry for v0.10.0 covering `paths`, `pull`, TUI staleness, and template composition
  3. README documents `paths` and `pull` with usage examples including the agent CLI injection pattern
