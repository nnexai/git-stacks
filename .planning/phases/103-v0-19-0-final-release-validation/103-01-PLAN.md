<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| release docs/package metadata -> npm consumers | Version and release notes define what users believe is being published. |
| smoke tests -> release decision | A green smoke test must represent the Phase 100-102 fixes, not just unrelated v0.18.0 workflows. |
| local release script -> git/npm state | The release script can create tags and perform publish dry-runs, so execution must separate validation from mutation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-103-01 | Spoofing | `package.json`, `CHANGELOG.md` | mitigate | Task 1 aligns package version, changelog heading, and smoke assertions on `0.19.0-rc.2`. |
| T-103-02 | Tampering | `tests/commands/release-rc.test.ts` | mitigate | Task 2 adds stable release-smoke assertions for Phase 100-102 focused coverage so unrelated smoke cannot falsely satisfy `REL-02`. |
| T-103-03 | Repudiation | `103-01-SUMMARY.md` | mitigate | Task 3 records exact focused commands, the full release gate, tag target, and publish handoff. |
| T-103-04 | Elevation of privilege | `scripts/release-rc-check.ts` tag creation | mitigate | Execution uses `--skip-tag` for verification; non-skip tag creation remains an explicit operator handoff. |
| T-103-05 | Information disclosure | `CHANGELOG.md`, `README.md` | mitigate | Release-facing docs stay user-facing and avoid maintainer-only verification details or private planning notes. |
</threat_model>
