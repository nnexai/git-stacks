# 107.3 POC Evidence

## Location and role

[`../../spikes/010-web-terminal-tabs/`](../../spikes/010-web-terminal-tabs/)

The POC is a standalone feasibility and visual harness. It does not modify or use the production service/native client, and its server/client implementation must not be copied into production without replacing the known shortcuts below.

## Run

```bash
cd .planning/spikes/010-web-terminal-tabs
bun install
bun run check
bun run build
bun server.ts
```

The root page creates three Bash-backed terminal tabs and supports adding more.

## Findings that invalidate broader POC claims

### Reconnect duplicates output

The server sends the entire retained `scrollback` array on every socket open. The client reconnects into the existing xterm without clearing it or sending a cursor. Forced reconnect doubled already-visible startup and marker output. The POC therefore proves that the PTY survives a socket loss, but **not** correct replay/reconnect.

### HTTP bootstrap is not production authentication

`GET /` grants the process-global cookie to any caller. API routes check only that cookie; they do not enforce exact Host, Origin/Fetch Metadata, principal ownership, expiry, or per-principal session scope. Adversarial requests with a different Origin/Host reached session creation during the audit. Only the WebSocket route checks Origin.

### Sessions and pressure are not bounded correctly

- One global `Map` owns every session and one global capability sees all of them.
- Session creation is unlimited; ended sessions are never pruned.
- A disconnected session has no TTL.
- `socket.send() === -1` only logs even though Bun reports queued backpressure.
- There is no `drain` recovery, ACK, queue budget, or pressure close.

### Signals are bypassed

The POC does not generate a web surface UUID/token, inject the reserved signal environment, parse authenticated OSC 9 frames, strip them from output, or publish them to the service journal. Raw relay is unsafe as production scaffolding because app-owned frames include the signal token.

### Hidden terminals are opened before they have dimensions

All xterm instances call `open()` while their pane is hidden. xterm requires a visible, dimensioned parent at open time. The audit observed inactive terminals retaining stale 24-row geometry while the active terminal resized. Production must lazy-open on first activation and fit after visibility.

### UI/lifecycle limitations

- The top status can be stale after switching tabs.
- The tab status element is styled as a 7×7 dot but contains visible words, causing overflow.
- The close affordance is a span inside a button rather than an independent keyboard action.
- Reconnect retries every 500 ms forever without backoff.
- Close removes the UI before the server confirms process cleanup.
- No accessibility, CSP, reload-state, operation, signal, or service integration exists.

## Correct verdict

**PARTIAL.** The POC answers the hard feasibility question—Bun-owned local PTYs can drive multiple xterm.js browser tabs with working input and resize. It does not validate a secure local product bridge, correct reconnect, bounded resource behavior, signal integration, or production process ownership.

The production plan keeps only these concepts:

- service-side PTY ownership for web surfaces;
- one xterm.js renderer per retained browser tab;
- WebSocket terminal data separated from HTTP/SSE control data;
- visible-pane fit/resize and explicit close.

It replaces the global cookie, raw replay, global session map, fixed reconnect loop, blind output relay, and direct Bash launch with the architecture in `107.3-ARCHITECTURE.md`.
