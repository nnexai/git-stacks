# 107.3 Architecture

## Architectural decision

The existing git-stacks service is the browser's machine authority and owns terminal sessions created for web surfaces. The browser is a thin product projection and terminal renderer. It never receives filesystem paths, launch commands, environments, service credentials, PIDs, or process-control capabilities.

This does not turn git-stacks into a general session multiplexer. Native Ghostty surfaces and external integrations retain their current ownership; web terminals are separate, service-owned processes with bounded reconnect lifetime.

## Runtime shape

```text
git-stacks web (trusted CLI)
  │ starts/discovers service using protected descriptor + official bearer
  │ POST /v1/web-pairings
  ▼
┌────────────────────────────────────────────────────────────────────┐
│ Existing Bun service — loopback only                               │
│                                                                    │
│ Shared domain/core                                                │
│ snapshot builder · launch resolver · operations · event journal   │
│ signal projection · creation catalog · workspace change monitor   │
│            ▲                         ▲                             │
│            │ direct in-process use   │                             │
│ /v1 adapter│                         │/web adapter                  │
│ bearer auth                          │browser principal + safe DTOs │
│                                      │                             │
│                                      ├─ WebPrincipalManager        │
│                                      ├─ BrowserProjection          │
│                                      ├─ WebTerminalManager         │
│                                      ├─ TerminalSignalFilter       │
│                                      └─ static asset server        │
└───────────────────────────────────────┬────────────────────────────┘
                                        │ same-origin HTTP/SSE/WS
                                        ▼
                              ┌─────────────────────────┐
                              │ Thin browser client     │
                              │ SolidJS DOM state/UI    │
                              │ xterm.js per live tab   │
                              │ IDs + intents only      │
                              └─────────────────────────┘
```

There is one listener and one service process. `/v1` stays bearer-authenticated and compatible. `/web` adds a browser-specific security and serialization boundary over the same internal objects.

## Responsibility boundary

| Concern | Service | Browser |
|---|---|---|
| Workspace/Git/filesystem truth | Owns | Displays redacted projection |
| Launch resolution | Owns argv/cwd/env/ports/provider setup | Sends IDs, revision, dimensions |
| PTY/process lifecycle | Owns spawn, group/tree cleanup, quotas, retention | Sends input/resize/close intents |
| Terminal state continuity | Owns raw delta ring and headless serialized baseline | Applies ordered frames and acknowledges processed cursor |
| Signals | Generates token/surface identity, filters OSC, publishes journal event | Displays attention and requests explicit focus/dismiss |
| Auth | Issues pairing and principal; keeps official bearer private | Receives HttpOnly cookie only |
| UI state | Supplies authoritative revisions/events | Owns selection, layout, theme, manual terminal title |
| Persistence | Existing service stores only existing domain data; web terminals are process-local | May store non-sensitive presentation preferences only |

## Shared service composition

`src/service/main.ts` should construct one shared runtime:

```text
snapshot builder
event journal -> event broker
operation registry
workspace mutation adapters
workspace change monitor
browser principal manager
web terminal manager
       │
       ├─ /v1 route adapter
       └─ /web route adapter
```

The web adapter calls these objects directly. It does not fetch `/v1` over loopback. This preserves one revision source, one operation/event identity, one set of mutation side effects, and one idle lifecycle.

The idle lifecycle gains at least `activeWebTerminals` and total browser connection accounting. A live or retained web terminal prevents idle shutdown. Service shutdown first rejects new browser work, closes web sockets/SSE, terminates tracked terminal process trees, then closes the existing broker and listener.

## Browser HTTP contracts

### Trusted pairing route

`POST /v1/web-pairings` remains behind existing official bearer authentication and rate limits.

Request:

```json
{"open_mode":"browser"}
```

Response data:

```json
{
  "pairing_id":"pair_...",
  "pairing_url":"http://127.0.0.1:43123/web/#pair=...",
  "expires_at":"2026-07-13T12:00:00.000Z"
}
```

The code has at least 192 bits of entropy, is stored hashed, expires in approximately 60 seconds, and is consumed atomically once. Pairing issuance is observable without logging the code or URL.

### Static and auth routes

| Route | Auth | Purpose |
|---|---|---|
| `GET /web/` | None or existing cookie | Serve inert bootstrap/app shell |
| `GET /web/assets/:asset` | None | Serve packaged immutable local assets |
| `POST /web/auth/exchange` | One-use pairing code | Set principal cookie and return safe bootstrap status |
| `POST /web/auth/logout` | Cookie | Revoke principal, close sockets, and begin terminal cleanup |

The pairing code is carried in the URL fragment, posted in a strict JSON body, then removed with `history.replaceState`. A page without a valid cookie or fragment remains a harmless “run `git-stacks web`” shell.

### Browser API routes

| Route | Purpose |
|---|---|
| `GET /web/api/discovery` | Browser capabilities, quotas, protocol versions |
| `GET /web/api/snapshot` | Redacted aggregate workspace model |
| `GET /web/api/events` | Browser-authenticated SSE over the shared broker |
| `GET /web/api/signals` | Current safe signal projection |
| `POST /web/api/signals/dismiss` | Dismiss a notification through existing journal logic |
| `GET /web/api/workspace-creation/catalog` | Safe creation choices |
| `POST /web/api/operations/:kind` | Allowlisted mutation submission |
| `GET /web/api/operations/:id` | Principal-owned operation status |
| `POST /web/api/operations/:id/cancel` | Cancel a principal-owned cancellable operation |
| `POST /web/api/terminal-sessions` | Create a service-owned web terminal |
| `GET /web/api/terminal-sessions` | List this principal's live/retained sessions |
| `PATCH /web/api/terminal-sessions/:id` | Update a bounded display title for an owned session |
| `POST /web/api/terminal-sessions/:id/close` | Idempotent close request |
| `GET /web/terminal/:id` | WebSocket upgrade |

Unknown routes, fields, methods, mutation names, and IDs fail closed. Existing workspace mutations currently consume a workspace name, but browser open/close requests use `{workspace_id, expected_revision}`; the service resolves the current name from the authoritative snapshot immediately before invoking the existing adapter. Browser idempotency keys are namespaced as `web:<principal-id>` before entering `OperationRegistry`. A browser operation ownership map prevents another browser principal from receiving, polling, or cancelling an operation by guessed ID.

## Browser-safe data model

The browser snapshot is a new schema, not a partial TypeScript type cast over the trusted snapshot:

```text
BrowserSnapshot
  revision, generated_at
  workspaces[]
    id, name, branch, labels
    repositories[]: id, name, mode
    commands[]: id, name, scope, repository_id?
    status[]: safe repository status counters/states
    file_status?: counters only
```

It excludes:

- repository and workspace paths;
- raw command strings and launch steps;
- cwd, environment, secret references, redacted names, and resolved ports;
- PIDs, process groups, launch diagnostics, pairing/principal identifiers;
- generic operation results/details not explicitly allowlisted.

Each event type has a browser schema. Operation projection is per mutation kind; it cannot pass through the existing open-ended `result` record. Browser SSE also filters operation events by the persisted `client_id`/operation reservation: a principal sees only operations submitted through its own `web:<principal-id>` namespace. Hidden events are consumed and acknowledged by the server-side subscription, with safe cursor checkpoints so reconnect does not repeatedly scan unrelated operations. Global snapshot-control and signal events remain visible. Tests inject recognizable sentinel secrets/paths into every trusted field and assert they are absent from serialized HTTP, SSE, logs, and terminal controls.

## Request security policy

The listener accepts only `hostname: "127.0.0.1"`; configuration cannot widen it in this phase. The canonical origin and Host are derived from the actual bound port.

| Request | Required checks |
|---|---|
| Top-level `GET /web/` | Exact Host; `Sec-Fetch-Mode: navigate` with `Sec-Fetch-Site: none` or same-origin; no state change |
| Static assets | Exact Host; same-origin destination policy; immutable known manifest path |
| `POST /web/auth/exchange` | Exact Host and Origin; same-origin Fetch Metadata; valid one-use body |
| Authenticated API `GET/HEAD` | Exact Host, principal cookie, `Sec-Fetch-Site: same-origin`; do not require Origin because browsers often omit it |
| Unsafe API methods | Exact Host and Origin, principal cookie, same-origin Fetch Metadata, JSON content type |
| WebSocket | Exact Host and Origin, principal cookie, allowed path/session ownership, protocol version |

There is no CORS response. `Origin: null`, cross-site API requests, alternate `localhost`, IPv6, DNS-rebinding Host values, and LAN addresses are rejected. API responses use `Cache-Control: no-store`; static hashed assets may be immutable.

The principal cookie is opaque, high-entropy, HttpOnly, `SameSite=Strict`, `Path=/web`, and short-lived/sliding within a hard maximum. It is not readable by JavaScript and is stored hashed server-side. Because this design serves loopback HTTP, it does not falsely depend on `Secure`; remote/HTTPS support is a separate design.

Recommended document headers:

```text
Content-Security-Policy:
  default-src 'none';
  script-src 'self';
  style-src 'self';
  connect-src 'self' ws://127.0.0.1:<port>;
  img-src 'self' data:;
  font-src 'self';
  object-src 'none';
  base-uri 'none';
  frame-ancestors 'none';
  form-action 'none'
Referrer-Policy: no-referrer
X-Content-Type-Options: nosniff
Cross-Origin-Opener-Policy: same-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

The no-inline-style target is verified in a real browser with xterm. If a pinned xterm release demonstrably requires an inline style block, use a per-response nonce or the narrowest documented exception and record the test; do not silently broaden `script-src`.

## Web terminal creation

Request:

```json
{
  "workspace_id":"uuid",
  "repository_id":"uuid",
  "command_id":"cmd_...",
  "expected_revision":"42",
  "cols":120,
  "rows":36
}
```

Creation sequence:

```text
validate principal quota + strict request
  -> call shared launch resolver with IDs/revision
  -> fail before spawn on stale/not-found/capability errors
  -> generate terminal ID, surface UUID, signal token
  -> overlay reserved service-owned env and TERM
  -> create process ownership record
  -> spawn Bun PTY with requested dimensions
  -> initialize output ring + headless terminal
  -> register session as live
  -> return only safe terminal metadata
```

The resolver's argv/cwd/environment never enter a browser DTO. Reserved variables override any user/config value:

```text
GIT_STACKS_SURFACE_ID=<service UUID>
GIT_STACKS_TAB_ID=<same UUID>
GIT_STACKS_WORKSPACE_ID=<authoritative UUID>
GIT_STACKS_REPOSITORY_ID=<authoritative UUID>
GIT_STACKS_SIGNAL_TOKEN=<32-byte secret>
GIT_STACKS_SIGNAL_TRANSPORT=osc9
GIT_STACKS_LAUNCH_KIND=shell|command
GIT_STACKS_COMMAND_ID=<when present>
TERM=xterm-256color
```

The existing resolver is POSIX-oriented for configured commands (`/bin/sh -lc`). Introduce a `TerminalLaunchPolicy`/platform adapter before claiming Windows support. Linux is the first runtime gate; macOS and Windows require actual-host launch and cleanup proof.

## Terminal state machine

```text
requested -> resolving -> starting -> live
     │           │           │         │
     └───────────┴───────────┴──error──> failed
                                      live
                                       │ socket loss
                                       ▼
                                  disconnected
                                  │ reconnect │ grace expiry
                                  └───────────┴──────> closing
                                                       │
child exit --------------------------------------------┤
explicit close / logout / shutdown --------------------┤
                                                       ▼
                                            ended | cleanup_failed
```

Session metadata includes terminal/surface IDs, owner principal, workspace/repository/command IDs, revision, title, dimensions, state, creation/last-attach times, output cursor, replay boundary, exit status, and cleanup diagnostics. Only safe fields are projected.

Ended sessions have bounded count/TTL and no PTY handle. Disconnected live sessions have a short grace TTL and continue to count against quotas and service lifetime. Principal hard expiry or logout closes attachments and starts cleanup. A newly paired principal does not inherit old sessions in the first implementation.

## Process ownership and cleanup

`TerminalAdapter` owns PTY mechanics; `ProcessControl` owns the process tree. The implementation must prove rather than assume PTY group behavior.

POSIX target:

1. establish/verify a dedicated process group for the PTY child;
2. record PID, PGID, and a birth/identity guard where available;
3. close input and send graceful termination to the owned group;
4. wait a bounded interval;
5. send forceful termination to the same verified group;
6. close the terminal and verify the group no longer exists.

Windows target uses a tested process-tree/job abstraction and follows Bun's documented ConPTY teardown ordering. If safe ownership cannot be established on a target, web-terminal capability is unavailable there rather than falling back to unscoped PID killing.

Abrupt service death cannot run ordinary cleanup. Before release, a Linux crash fixture must prove that closing the PTY master removes the owned process group, including descendants. If that proof fails for ignored-HUP/stubborn descendants, add a small ownership guardian that monitors a parent pipe and kills only the verified PGID on EOF. This guardian provides cleanup, not session persistence or reclaim. Apply the equivalent tested mechanism on each supported platform; no platform may advertise web terminals while abrupt owner death leaves a known owned process tree behind.

## Terminal signal filter

PTY output first enters an incremental byte-state machine:

```text
PTY bytes
  -> detect bounded OSC (BEL or ST terminator) across chunks
  -> if body starts git-stacks-signal:
       consume frame
       validate token/provider/session/state
       derive surface/workspace/repository from TerminalSession
       publish Signal through shared journal/broker
       record bounded diagnostic on invalid frame
     else
       preserve bytes unchanged
  -> sanitized bytes feed headless terminal, replay ring, and sockets
```

The app-owned candidate frame is consumed even when invalid, preventing token-bearing bytes from reaching the browser. The parser caps candidate size and recovers without buffering arbitrary output. Chunk-boundary, malformed UTF-8, BEL/ST, token mismatch, oversized body, and ordinary OSC tests are mandatory.

## Terminal WebSocket protocol

The subprotocol is versioned, for example `git-stacks-terminal.v1`. Text frames are strict JSON control messages. Output is binary to preserve bytes efficiently.

Binary frames:

```text
kind 0x01 — live or delta output
  byte 0      0x01
  bytes 1..8  unsigned big-endian output cursor
  bytes 9..N  sanitized PTY bytes

kind 0x02 — serialized snapshot chunk
  byte 0      0x02
  bytes 1..8  unsigned big-endian snapshot-through cursor
  bytes 9..12 unsigned big-endian chunk index
  bytes 13..N serialized UTF-8 VT state
```

Client controls:

```json
{"v":1,"type":"attach","last_applied_cursor":"42","cols":120,"rows":36,"takeover":false}
{"v":1,"type":"ack","cursor":"47"}
{"v":1,"type":"input","data":"..."}
{"v":1,"type":"resize","cols":120,"rows":36}
{"v":1,"type":"ping"}
```

Server controls:

```json
{"v":1,"type":"ready","session_id":"term_...","state":"live","cursor":"47"}
{"v":1,"type":"replay_start","after":"42","through":"47"}
{"v":1,"type":"replay_end","through":"47"}
{"v":1,"type":"snapshot_start","through":"47","reset":true}
{"v":1,"type":"snapshot_end","through":"47"}
{"v":1,"type":"history_unavailable","cursor":"47"}
{"v":1,"type":"exit","exit_code":0,"signal":null}
{"v":1,"type":"error","code":"...","message":"...","retryable":false}
```

Input may remain JSON for v1 because browser keyboard data is small; it is byte-limited before decoding/writing. Resize is range-checked and rate/debounce limited. The socket URL contains only the opaque terminal ID, never a pairing capability or service token.

### Ordering and continuity

- Every sanitized PTY chunk receives one monotonically increasing cursor and is stored as `{cursor, bytes}` in a bounded ring.
- The service captures a replay upper bound, queues later live frames, sends replay/snapshot through the bound, emits the end control, then flushes queued live frames in cursor order.
- The browser advances `lastAppliedCursor` only in xterm's write callback, then sends a coalesced ACK. Socket receipt alone is not application.
- A soft reconnect requests delta after `lastAppliedCursor`; already-applied output is never resent.
- A fresh page requests a serialized headless baseline. The browser resets xterm, applies ordered `0x02` snapshot chunks between `snapshot_start`/`snapshot_end`, acknowledges the snapshot-through cursor, then receives deltas after that cursor.
- If neither delta nor snapshot is valid, the process may stay live but the pane clearly reports history unavailable; it never silently displays a partial stream as complete.

### Backpressure

PTY draining must not depend on one browser. Output always advances the bounded ring/headless state. Each attachment has a bounded pending budget and last ACK.

When `send()` reports backpressure, stop adding live frames to that socket. Resume from its ACK in `drain` if retained; otherwise send a snapshot/resync. Configure a small explicit `backpressureLimit`, `closeOnBackpressureLimit`, payload limit, idle timeout, and retryable close code. A slow attachment may be disconnected, but it cannot pause the PTY or grow global memory.

Only one writable attachment is allowed per terminal. A same-principal explicit takeover closes the prior attachment; unrelated principals can never attach. Multiple independent terminals/tabs are supported.

## Browser client architecture

The browser uses `solid-js/web`, never `@opentui/solid`. Its stores hold browser DTOs only:

- `controlStore`: discovery, snapshot revision, SSE cursor, reconnect/gap refresh, operations, signals;
- `terminalStore`: safe session metadata and tab/pair organization;
- `presentationStore`: active workspace/repository/tab, grouping, browser-local tab order, theme, and density. Pin membership and numeric workspace priority are authoritative workspace-definition metadata, not client presentation state;
- `connectionStore`: authenticated/disconnected/incompatible/recovering state.

xterm ownership stays inside `TerminalPane`:

1. create terminal/addons when the pane is first activated;
2. call `open()` only after the pane is visible and measured;
3. retain the mounted terminal while its live tab is hidden;
4. debounce `ResizeObserver`, call `fit()`, and send exact resulting dimensions;
5. update replay cursor only after xterm's write callback;
6. dispose only after server-confirmed close/ended removal.

Terminal titles, OSC titles, signal details, and errors are rendered with text nodes. No raw `innerHTML`, automatic URL link addon, or terminal-triggered navigation is enabled initially. Clipboard actions require explicit user gestures and visible feedback.

Product behavior mirrors the native semantics: workspace-first navigation, exact workspace/repository tab collections, browser-local tab reorder, workspace-backed pinning and priority, service-persisted bounded manual terminal titles, command-only launcher with scope labels, explicit signal focus, creation operation progress, and snapshot invalidation without terminal teardown.

## Build and packaging

- `tsconfig.web.json` includes DOM libraries and sets browser Solid JSX; it excludes service/TUI sources.
- `scripts/build-web.ts` runs an explicit browser-target bundle for JS and CSS and emits a deterministic asset manifest.
- Production assets live under a package-included path such as `dist/web`; `package.json.files` and `prepublishOnly` include/build them.
- Development uses `bun run web:build` before `git-stacks web`; missing/stale assets fail with an actionable message rather than an implicit runtime CDN fetch.
- `npm pack`/installed-package tests verify HTML, xterm CSS, JS, source maps policy, MIME types, and no `@opentui/solid` import.
- No service worker in the initial version; stale terminal/auth code is a worse failure mode than reloading assets.

## Capacity defaults to decide and test

The exact numbers remain configurable implementation constants, but every dimension is bounded:

- pairing codes globally and per official client;
- browser principals and lifetime;
- HTTP rate/body sizes and SSE connections per principal/total;
- live and retained terminals per principal/total;
- terminal input frame, title, dimensions, and resize rate;
- replay bytes/chunks, headless scrollback, pending live bytes, and ACK lag;
- ended-session retention and disconnected grace;
- close/force-kill deadlines.

Discovery exposes safe client-relevant limits so the UI can disable actions before capacity errors. Diagnostics expose counts and states, never terminal content or secrets.

## Cross-platform posture

The protocol and adapter interfaces are portable; runtime support is earned per host:

1. Linux production slice and process-tree tests.
2. macOS actual-host shell/PTY/resize/cleanup/package proof.
3. Windows actual-host launch-policy, ConPTY, process-tree, resize, Unicode/IME, and package proof.

Until a platform passes, `/web/api/discovery` reports terminal capability unavailable there while non-terminal browser control features may remain available.
