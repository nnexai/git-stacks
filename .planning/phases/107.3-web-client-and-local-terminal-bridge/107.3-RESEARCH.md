# 107.3 Research

## Research question

Can git-stacks expose the native client's workspace behavior in a browser while keeping the browser thin and connecting terminal tabs to real local processes safely?

## Conclusion

Yes. The appropriate boundary is an in-process browser adapter on the existing Bun service. The browser should not become an official `/v1` client and should not receive the official bearer credential. It receives a purpose-built, redacted API and terminal protocol under `/web`; the service reuses its existing domain components directly and adds only browser authentication, projection, terminal ownership, signal filtering, and static assets.

The terminal transport itself is feasible. The existing POC proves multiple PTYs, xterm.js rendering, input, tab retention, and resize. It does **not** prove production reconnect or browser security: reconnect currently replays the whole retained buffer into an existing terminal, HTTP routes accept the cookie without Host/Origin policy, sessions are unbounded, slow-client handling only logs, and signal OSC frames are not integrated.

## What already exists and should be reused

| Existing component | Current responsibility | Browser use | Required addition |
|---|---|---|---|
| `startManagedService()` / descriptor / official credential | Starts or discovers one protected loopback service | `git-stacks web` uses these from the CLI only | Bearer-authenticated pairing issuance; no descriptor or bearer in the page |
| `createSnapshotBuilder()` | Produces consistent revisions and workspace projections | Direct in-process source for browser state and launch resolution | Explicit browser DTO that removes paths, commands, cwd, environment, references, redaction metadata, and ports unless intentionally exposed |
| `resolveNativeLaunch()` | Resolves IDs and revision into server-authoritative argv/cwd/env | Internal input to web-terminal creation | Generalize the name or wrap it; add web surface identity and platform/process policy after resolution |
| `EventJournal` + `EventBroker` | Durable event order, replay, SSE backpressure, signal projection | Source for browser SSE and terminal signal publication | Browser-principal quotas and sanitized event/operation payloads |
| `OperationRegistry` | Idempotent operations and lifecycle events | Shared operation executor | Principal-scoped idempotency namespace and persisted ownership lookup; filter SSE/poll/cancel by owner; per-operation browser result schemas |
| Workspace creation catalog/adapters | Catalog and mutation execution | Shared browser creation flow | Browser schemas and safe progress/error projection |
| Workspace change monitor | Snapshot invalidation | Shared browser refresh trigger | Browser SSE adapter; live terminals survive projection refresh |
| Agent integration manager + signal contracts | Installs provider hooks and defines authenticated OSC 9 payloads | Same hook output inside web PTYs | Service-side incremental OSC recognizer, token validation/stripping, surface attribution, and journal publication |
| Idle lifecycle | Stops an unused managed service | Shared service lifetime | Count web sockets, live terminals, retained terminals, and active principals |

This is not an HTTP proxy architecture. Browser routes and `/v1` routes are two adapters over the same in-memory components. Calling `/v1` from `/web` would duplicate serialization/authentication, complicate deadlines, and make operation ownership harder to enforce.

## Browser-safe projection is mandatory

The current workspace snapshot is intentionally useful to trusted native/official clients, but it contains machine information that a thin browser does not need:

- repository absolute `path`;
- launch step `command` and `cwd`;
- launch and step environments;
- secret reference names and redaction metadata;
- resolved ports;
- generic operation `result` and error `details`, which may acquire path-bearing fields over time.

The browser snapshot should contain display and intent fields only: revision/time, workspace identity/name/branch/labels, repository identity/name/mode, safe status summaries, command identity/name/scope, and capability/limit data. Launch resolution remains entirely server-side. Operation results need per-kind schemas rather than a pass-through `Record<string, unknown>`. The event broker is global, so browser SSE must also filter operation events through the registry's persisted client/operation reservation; otherwise one paired browser principal could observe official-client or other-principal operations.

## PTY and terminal evidence

[Bun's child-process documentation](https://bun.sh/docs/runtime/child-process) documents a real PTY through `Bun.spawn(..., { terminal })`, including input, data callbacks, resize, drain, close, POSIX `openpty()`, and Windows ConPTY. It also documents platform differences: ConPTY output is semantically equivalent but not byte-identical, resize signaling differs, and older Windows builds need child termination before pseudo-console close. These differences require actual-host tests; Linux success is not Windows cleanup proof.

[xterm.js](https://github.com/xtermjs/xterm.js/) provides the browser renderer, a headless package, and a serialization addon. Its documented headless/serialize use case is server-side state tracking and restoration after reconnect. [The Terminal API](https://xtermjs.org/docs/api/terminal/classes/terminal/) also requires the parent to have dimensions when `open()` is called. Production tabs must therefore lazy-open while visible and fit after activation; opening all hidden panes at `0×0` is not valid.

The compatible POC pair is `@xterm/xterm` 6.0.0 with `@xterm/addon-fit` 0.11.0. Omitting xterm's stylesheet exposed measurement helpers as comma-like text and broke fit; production packaging needs an automated asset/style smoke check.

## Transport evidence and corrected reconnect model

[Bun WebSockets](https://bun.sh/docs/runtime/http/websockets) support binary frames, payload/idle limits, `drain`, `backpressureLimit`, and `closeOnBackpressureLimit`. Terminal output should use binary frames with an explicit cursor header; JSON remains suitable for bounded control messages.

The POC's reconnect is incorrect: every socket open sends the complete retained buffer, while the client keeps the existing xterm buffer. A reconnect therefore duplicates prior output. A production protocol needs:

1. a monotonically increasing output cursor attached to every binary frame;
2. a client acknowledgement after xterm has processed the frame;
3. delta replay from the last applied cursor for a short disconnect;
4. an ordered replay barrier so live output cannot interleave ahead of replay;
5. a server-side headless xterm snapshot for a fresh page or raw-ring gap;
6. an explicit `history_unavailable` state if headless restoration is unavailable or fails.

Raw output retention alone is not terminal state. A bounded ring is useful for deltas, while headless serialization supplies a coherent baseline.

## Signal integration finding

The current native launch path generates a surface UUID and a 32-byte signal token, injects `GIT_STACKS_SURFACE_ID`, `GIT_STACKS_TAB_ID`, workspace/repository IDs, `GIT_STACKS_SIGNAL_TOKEN`, and `GIT_STACKS_SIGNAL_TRANSPORT=osc9`, then parses provider-owned OSC 9 payloads against the surface token.

Web terminals need the same attribution semantics on the service side. Sending raw PTY bytes directly to the browser can expose the token inside app-owned OSC frames and leaves web attention disconnected from the shared journal. The service must scan across arbitrary chunk boundaries, consume only the bounded `git-stacks-signal:` OSC payload, validate it against the session token, derive workspace/repository/surface identity from server state, publish a normal service signal event, and strip the private frame before feeding replay/headless/browser output. Other terminal bytes remain byte-preserving and untrusted.

## Browser authentication and request policy

A browser WebSocket cannot set the service's `Authorization` header, and the official bearer is too powerful to expose to JavaScript. The safe bootstrap is:

1. `git-stacks web` starts/discovers the service and calls bearer-authenticated `POST /v1/web-pairings`.
2. The service returns a one-use, short-lived URL whose code is in the URL fragment: `/web/#pair=...`.
3. Fragments are not sent in the HTTP request. The static bootstrap reads it, posts it to `/web/auth/exchange`, clears the fragment with `history.replaceState`, and receives an HttpOnly `SameSite=Strict; Path=/web` cookie.
4. The one-use code and principal secrets are stored hashed in memory and die on service restart.

An exchange-on-GET design is rejected because prefetchers, scanners, and link expansion can consume a one-use capability. Query parameters also leak more easily through logs/history/referrers than fragments.

The original “exact Origin on all browser API requests” rule is also incorrect. [MDN documents that same-origin `GET` and `HEAD` commonly omit `Origin`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Headers/Origin). The policy should be:

- exact `Host` on every `/web` request and WebSocket upgrade;
- exact served `Origin` on unsafe methods and WebSocket upgrades;
- cookie plus a [Fetch Metadata resource-isolation policy](https://developer.mozilla.org/en-US/docs/Web/HTTP/Guides/Fetch_metadata) for browser API reads;
- no CORS, no `Origin: null`, and no alternate localhost/IPv6/LAN aliases;
- static top-level navigation allowed from `Sec-Fetch-Site: none`, while authenticated API calls require same-origin browser context.

Because the first implementation uses loopback HTTP, the cookie cannot depend on `Secure` until an HTTPS/secure-loopback deployment is explicitly designed and tested. HttpOnly, Strict SameSite, narrow Path, opaque entropy, expiry, revocation, Host/Origin checks, and local-only binding are therefore all required together.

## Frontend/build finding

The root `tsconfig.json` and `bunfig.toml` are configured for `@opentui/solid`, not browser DOM JSX. Browser code needs a separate `tsconfig.web.json`, explicit DOM libraries and browser Solid JSX settings, and a dedicated build entry that does not import `@opentui/solid`. The package currently publishes only `src`, so built web assets need an explicit package path and `npm pack`/installed-CLI test.

The recommended client is SolidJS DOM plus xterm.js. “Thin” means it has no system authority or duplicated business logic; it does not require an unstructured frontend. Solid is already a dependency, while a separate build boundary prevents OpenTUI preload/JSX leakage. No CDN, dynamic remote code, or service worker should be used in the initial release.

## Options

| Option | Advantages | Problems | Decision |
|---|---|---|---|
| Browser attaches to native Ghostty surfaces | One live session universe | Requires a new attachable native PTY/surface protocol and couples native/web lifecycle | Reject for 107.3 |
| Existing Bun service owns web PTYs | Thin browser, direct reuse of launch/domain logic, one local process | Requires robust terminal/process/auth components | Chosen |
| Separate Node/`node-pty` sidecar | Mature ecosystem | Extra service, native addon packaging, duplicate lifecycle/auth | Fallback only if Bun PTY fails a required target |
| tmux-backed web sessions | Mature persistence/attach | Makes a specific external integration mandatory and changes product semantics | Defer |
| Browser-only terminal | Trivial deployment | Cannot interact with local sessions | Reject |

## Feasibility verdict

**Feasible with partial POC validation.** There is no fundamental blocker to service-owned local PTYs rendered in browser tabs. Production readiness depends on new service components, particularly pairing, browser-safe DTOs, cursor/snapshot continuity, slow-client bounds, process-tree cleanup, and signal filtering. The standalone POC should remain evidence and a visual harness, not the production codebase.

## Validation architecture

| Risk | Earliest proof | Production gate |
|---|---|---|
| Sensitive state leaks through browser DTOs | Schema/source test with sentinel paths/env/secrets | Every browser response and SSE event parses through explicit browser schemas; negative fixture contains no sentinel |
| Pairing/CSRF/WS hijack | Route tests with Host, Origin, Fetch Metadata, cookie, expiry, replay | Real-browser cross-origin adversarial page cannot read/mutate/upgrade; official `/v1` behavior unchanged |
| PTY cleanup leaves descendants | Linux real-process fixture with child/grandchild and stubborn process | Normal close, timeout escalation, service shutdown, and crash-recovery policy leave no tracked process group; actual-host gates later |
| Reconnect duplicates or loses state silently | Deterministic cursor/replay test plus forced browser socket close | Marker appears once after reconnect; reload restores serialized screen or displays explicit history loss |
| Slow browser grows memory | Flood PTY with paused/slow socket | Per-socket/ring/headless/session memory remains within configured limits; client closes/reconnects predictably |
| Signal token leaks or attribution can be forged cross-surface | Chunk-boundary/fuzz tests around OSC 9 parser | Valid frame publishes one attributed journal event and is absent from browser bytes; wrong token never publishes |
| Hidden panes mismeasure | Browser resize/tab-switch test | xterm opens only in visible measured pane; `stty size` matches after switching and viewport resize |
| Packaging misses assets/config | `npm pack` fixture and installed CLI smoke | `git-stacks web --no-open` serves hashed local assets with no OpenTUI imports or network fetches |
