# 107.3 Implementation Plan

> **Execution result (2026-07-13):** Implemented as one production vertical run without promoting the provisional phase into the shared roadmap/state. The slice design below remains the traceability baseline; completion evidence is in `107.3-SUMMARY.md` and `107.3-VERIFICATION.md`.

## Status and planning mode

This was authored as an implementation-ready blueprint rather than standard GSD plan files. Execution preserved its deliberate constraint: `ROADMAP.md`, `REQUIREMENTS.md`, and `STATE.md` were not modified while native work was active. The production implementation and verification records now close the blueprint on Linux.

## Goal

Deliver a local browser client with the same workspace/control semantics as the native client, while keeping all machine authority in the existing git-stacks service and all browser terminal processes owned and bounded by that service.

## Delivery principles

1. Build vertical slices against the real service; do not create a mocked browser backend that must later be replaced.
2. Preserve `/v1` behavior while adding a separate `/web` adapter over the same in-process domain objects.
3. Introduce an explicit browser-safe model before exposing snapshots/events/operations.
4. Include signal-token stripping and safe process ownership in the first production PTY slice.
5. Treat cursor replay, headless restoration, backpressure, and quotas as terminal correctness, not optional polish.
6. Earn platform claims on actual hosts.

## Proposed waves

| Wave | Slice | Depends on | User-visible result |
|---|---|---|---|
| 1 | 01 Secure web shell + snapshot | Existing service | `git-stacks web` opens a paired, read-only workspace browser |
| 2 | 02 One real service-owned terminal | 01 | One safe interactive shell/command terminal works end to end |
| 3 | 03 Multi-tab continuity + pressure | 02 | Multiple terminals survive switching/reconnect/reload without duplicate output |
| 4 | 04 Shared control plane + attention | 01, 03 | SSE, operations, creation, signals, and explicit terminal focus work |
| 5 | 05 Product-parity workspace UX | 04 | Native workspace hierarchy and command workflows are usable in browser |
| 6 | 06 Packaging + diagnostics + platform gates | 05 | Installed-package Linux release candidate and truthful macOS/Windows capability status |
| 7 | 07 Independent security/reliability closure | 06 | Adversarial and lifecycle gaps are closed before roadmap completion |

Wave 1 is intentionally read-only but uses production pairing, route policy, DTOs, assets, and the real snapshot. Wave 2 is the first terminal walking slice and must not reuse the POC's bootstrap or replay implementation.

## Slice 01 — Secure web shell and redacted snapshot

### Objective

`git-stacks web` starts or discovers the managed service, obtains a one-use pairing URL through authenticated `/v1`, opens/prints the loopback app, exchanges the fragment capability for an HttpOnly principal, and renders the real browser-safe workspace snapshot.

### Likely files

- `src/commands/web.ts`
- `src/lib/cli-program.ts`
- `src/lib/service/contract.ts` only for the trusted pairing route schema if appropriate
- `src/service/server.ts`, `src/service/main.ts`
- `src/service/web/contract.ts`, `projection.ts`, `principal-manager.ts`, `policy.ts`, `routes.ts`
- `src/web-client/index.html`, `app.tsx`, `api/http.ts`, `state/control-store.ts`
- `scripts/build-web.ts`, `tsconfig.web.json`, `package.json`
- focused tests under `tests/service/web/`, `tests/web-client/`, CLI/completion tests

### Implementation

- Refactor server dispatch so `/v1` still passes through the existing admission/rate/deadline behavior while `/web` passes through browser policy. Keep one `Bun.serve` listener.
- Add bearer-authenticated `POST /v1/web-pairings`; generate hashed, one-use, 60-second in-memory codes and fragment URLs.
- Add `git-stacks web` with `--no-open` and `--json`; use the official credential only inside the CLI. If this invocation creates the service, keep it foregrounded until SIGINT/SIGTERM; if a healthy service already exists, issue/open the pairing and exit.
- Serve only manifest-listed local assets. Add strict MIME, no-sniff, CSP, COOP, Referrer-Policy, Permissions-Policy, and cache policy.
- Implement fragment-to-POST exchange, immediate fragment removal, opaque HttpOnly Strict cookie, principal idle/hard expiry, logout, and in-memory revocation.
- Enforce exact Host everywhere, exact Origin for exchange/unsafe requests, Fetch Metadata for API reads, and no CORS.
- Define browser discovery and snapshot schemas. Project only safe workspace/repository/command/status fields; reject unknown fields.
- Use `solid-js/web` behind a dedicated web TypeScript/build config. Prove no `@opentui/solid` import and no CDN/network dependency.
- Update CLI completion inventory for the new top-level command.

### Acceptance

- `git-stacks web --no-open --json` returns a loopback fragment URL and expiry but never prints the official bearer or principal cookie.
- A valid code exchanges once; replay, expiry, wrong Host/Origin/Fetch Metadata, malformed JSON, and cross-site API calls fail closed.
- Same-origin browser `GET /web/api/snapshot` succeeds even though it may omit `Origin`.
- A sentinel trusted snapshot containing `/secret/path`, command text, cwd, environment values, references, redaction names, and ports produces none of those values in HTTP, page state, logs, or built assets.
- Existing `/v1` service integration tests and official credential behavior remain byte/contract compatible.
- The real browser shows current workspace/repository/command names and disconnected/incompatible/error states without reading local files.
- `bun run typecheck`, browser typecheck/build, focused tests, dependency-cycle check, and completion audit pass.

### Artifacts this slice produces

- `webCommand`
- pairing request/response schemas and `WebPrincipalManager`
- browser request policy and security-header builder
- browser discovery/snapshot DTO schemas and projection functions
- `/web` route adapter and initial web client bundle
- `tsconfig.web.json`, `web:build`, `web:typecheck`, asset manifest

## Slice 02 — One real service-owned terminal

### Objective

From a selected workspace/repository, create one real shell or configured command through the shared launch resolver and interact with it in xterm.js, with service-owned process cleanup and authenticated signal filtering from the first byte.

### Likely files

- `src/lib/service/snapshot.ts` and new `launch-policy.ts`
- `src/service/web/terminal/{adapter,manager,process-control,replay,signal-filter,protocol}.ts`
- `src/service/web/contract.ts`, `routes.ts`, `src/service/main.ts`
- `src/web-client/terminal/pane.tsx`, `api/terminal-socket.ts`, terminal store/components
- `tests/service/web/terminal-*`, real PTY fixtures, browser smoke

### Implementation

- Define strict create/list/close metadata schemas and versioned WS controls. Terminal creation accepts IDs, expected revision, cols, and rows only.
- Wrap/generalize `resolveNativeLaunch()` as the sole launch source. Keep argv/cwd/environment server-only and overlay reserved web surface/signal variables after resolution.
- Add `TerminalLaunchPolicy` so the current POSIX shell contract is explicit; report terminal capability unavailable on unsupported targets rather than pretending `/bin/sh` is portable.
- Implement a Bun `TerminalAdapter` and deterministic fake. Initialize session ownership and limits before reporting live.
- Implement POSIX process-group ownership, bounded TERM/KILL escalation, terminal close, child exit, shutdown cleanup, and observable `cleanup_failed`. Verify PGID assumptions with real process fixtures. Add an abrupt-owner-death fixture; if PTY-master close does not remove stubborn descendants, add a parent-pipe ownership guardian before the capability can ship.
- Implement the incremental OSC 9 filter before any output store/socket: consume token-bearing app frames, validate the session token, publish attributed service signals, and preserve all other bytes.
- Create a bounded cursor ring and headless xterm/serialize state from sanitized bytes. Even before reconnect UI, output must be cursor-framed and bounded.
- Add one writable same-principal WebSocket attachment, strict Origin/Host/cookie/session ownership, binary output cursor header, input/resize/control limits, and idempotent close.
- In the client, lazy-open xterm only in a visible measured pane, load pinned xterm CSS/FitAddon, debounce fitting, send dimensions, and dispose only after close confirmation.

### Acceptance

- Shell and configured command launch use stable IDs/revision; stale revision, missing scope, extra `argv`/`cwd`/`env`, and unsupported platform fail before spawn.
- A real shell roundtrip, Unicode/ANSI, `stty size`, and one full-screen program smoke pass in a browser.
- Valid provider OSC split across arbitrary chunks publishes exactly one service signal with the web surface identity and is absent from browser/replay/headless bytes. Wrong token/oversize/malformed frames do not publish or leak the candidate payload.
- Close, child exit, SIGINT/SIGTERM service shutdown, abrupt service death, and a stubborn child/grandchild fixture leave no owned process group on Linux.
- Input, output, replay ring, headless scrollback, sessions, and sockets have asserted limits; no global unbounded `Map` or byte array remains.
- The xterm stylesheet is present in the packaged build; no measurement-helper comma stream or zero-dimension open occurs.

### Artifacts this slice produces

- browser terminal HTTP/WS schemas and `git-stacks-terminal.v1`
- `WebTerminalManager`, `TerminalAdapter`, `TerminalLaunchPolicy`, `ProcessControl`
- cursor output frame encoder/decoder and bounded ring
- `TerminalSignalFilter`
- web terminal pane/store and first real terminal browser flow

## Slice 03 — Multiple tabs, reconnect, reload restoration, and backpressure

### Objective

Support several independent service-owned terminals with exact pair scoping, correct soft reconnect, coherent reload restoration, explicit history-loss fallback, and bounded slow-client behavior.

### Implementation

- Add terminal tab collections keyed by exact workspace/repository pair and terminal ID; hiding a pane preserves its xterm/process.
- Implement `attach(last_applied_cursor)`, xterm-write-completion ACKs, replay barriers, queued live output, and monotonic cursor assertions.
- Use the raw ring for delta replay. Use headless serialization for first attach/reload/gap: reset the browser terminal, apply one baseline through cursor N, then ordered deltas after N.
- If headless serialization is unavailable/corrupt, keep process state explicit and display `history_unavailable`; do not silently append from an arbitrary retained cursor.
- Add exponential reconnect with jitter, online/offline awareness, bounded retries, session-state refresh, and explicit user retry. Do not use a fixed infinite 500 ms loop.
- Configure WebSocket payload, idle, backpressure, and close limits. Track last ACK and per-attachment queued bytes; stop sending to a pressured socket and resume from ACK on drain or reconnect/snapshot.
- Enforce one writer. Support explicit same-principal takeover; reject unrelated or accidental second writers.
- Add disconnected grace, principal expiry/logout handling, ended-session retention/pruning, total/per-principal quotas, and service idle integration.
- Make close transactional in the UI: show closing, await server result/event, then remove; expose retry/cleanup failure.

### Acceptance

- At least three independent shells retain cwd/environment/process state while tabs switch; closing one does not affect the others.
- Forced socket reconnect leaves a unique marker exactly once and preserves process identity.
- Full page reload reconstructs the declared screen/scrollback baseline and then live output; if the headless path is intentionally disabled, the UI shows an explicit history-loss banner.
- Output produced during replay arrives after replay in cursor order. Property tests cover chunk boundaries, ring eviction, ACK coalescing, cursor overflow bounds, and duplicate frames.
- A flood process plus paused/slow browser stays within configured memory/queue budgets; the PTY keeps draining and only the slow attachment is resynced/disconnected.
- Resize after hidden-tab activation and browser viewport changes yields matching `stty size`; inactive panes are never fit at zero dimensions.
- Principal/session quotas and TTLs are deterministic under fake time; service does not idle-exit while a retained live terminal exists.

### Artifacts this slice produces

- ACK/replay/snapshot protocol controls and close codes
- headless snapshot coordinator and replay barrier
- attachment backpressure state and diagnostics
- terminal retention/quota policies
- multi-tab pair-scoped UI and reconnect/history-loss states

## Slice 04 — Shared control plane, operations, creation, and attention

### Objective

Connect the thin browser to the existing event journal, operation registry, workspace creation, invalidation, and signal projection without exposing trusted payloads or tearing down live terminals.

### Implementation

- Add principal-authenticated browser SSE using the shared `EventBroker`; retain its replay-gap semantics and add browser connection quotas/lifecycle accounting. Filter operation events through the registry's persisted client/operation ownership seam, consume hidden events, and advance with safe cursor checkpoints so official/other-principal operations never leak or cause replay loops.
- Sanitize every event. Add per-kind browser operation result/error schemas, browser operation ownership, namespaced idempotency, polling/reconnect, and cancel rules.
- Project the existing creation catalog and submit `workspace.create` through the existing adapter/registry.
- Expose allowlisted `workspace.open`/`workspace.close` operations and operation progress. Browser requests carry `workspace_id` plus `expected_revision`; resolve the current workspace name server-side immediately before reusing the existing name-based mutation adapter. Do not add arbitrary mutation names.
- On `snapshot_invalidated`, refetch and reconcile authoritative workspace state while preserving live web terminals as linked/orphaned presentation state according to product semantics.
- Surface journaled provider signals and browser-terminal OSC signals in one attention projection. Focus is explicit and resolves a live web surface when owned; otherwise use the documented repository/workspace fallback.
- Dismiss notifications through existing signal dismissal. Activity lifecycle remains provider/session/surface attributed.

### Acceptance

- SSE reconnect and replay-gap behavior refreshes state without duplicate operation/signal application.
- Another principal cannot receive, poll, or cancel an operation or attach/focus a terminal by guessing an ID; operations created by official `/v1` clients are absent from browser SSE.
- Sentinel generic operation result/details never leak; each allowlisted mutation returns only its declared browser result.
- Workspace creation from template and selected repositories shows accepted/running/succeeded/failed/cancelled states and refreshes the snapshot through normal invalidation.
- External workspace changes update browser navigation but do not kill live terminals; invalid pairs become explicit stale/orphan states.
- Signal focus never steals focus asynchronously, and a web terminal's OSC lifecycle event is visible through the same service event stream.

### Artifacts this slice produces

- browser event and per-operation schemas/projections
- principal operation ownership/idempotency adapter
- browser SSE bridge and control-store event reducer
- workspace creation and signal/attention browser flows

## Slice 05 — Product-parity workspace UX and accessibility

### Objective

Deliver the native client's locked workspace-first interaction semantics in a responsive browser implementation without copying GTK/Ghostty internals.

### Implementation

- Workspace-first sidebar with browser-local pinned manual order, active/grouped views, single-repository collapse, and explicit repository rows for multi-repository workspaces. Persist only opaque IDs and safe display preferences.
- Exact workspace/repository terminal tab collections with keyboard/drag reorder; meaningful shell/command titles, safe OSC-title handling, service-confirmed bounded manual rename, ended predecessor state, and relaunch as a new terminal identity.
- Command-only launcher with recent-first order, exact workspace/repository scope labels, disabled/capacity states, and no generic command-palette expansion.
- Contextual workspace/repository actions and creation UI driven only by browser capabilities/catalog.
- Hierarchical attention badges/inbox and explicit focus/fallback behavior.
- Disconnected, stale, incompatible, capability-unavailable, empty, loading, operation-failure, cleanup-failure, and history-loss states.
- Keyboard tab switching, roving focus, focus restoration, screen-reader names/live regions, 200% text, high contrast, reduced motion, dark/light/system themes, and narrow desktop layout.
- Explicit clipboard controls and safe terminal metadata rendering; no automatic URL activation.

### Acceptance

- A browser UAT matrix covers the corresponding native workspace, launcher, creation, tab, and attention scenarios with the same semantic outcomes.
- Keyboard-only users can reach, activate, rename, close, and switch terminal tabs and operate launcher/creation/signal inbox with visible focus.
- Screen-reader roles/names/status changes are verified; contrast, reduced motion, zoom, and narrow viewport checks pass.
- Terminal focus remains stable across SSE events and snapshot reconciliation; only explicit selection/focus actions move it.
- No browser component imports filesystem/Git/launch modules or trusted service models.

### Artifacts this slice produces

- complete browser workspace shell, launcher, creation, signals, contextual actions, and state views
- accessibility/browser UAT suite
- shared browser presentation preferences containing no machine paths or terminal content

## Slice 06 — Packaging, diagnostics, and platform gates

### Objective

Ship the web client in the installed package, provide safe diagnostics, and advertise terminal capability only on platforms that pass real launch/PTY/cleanup/browser tests.

### Implementation

- Build deterministic hashed assets into `dist/web`, include them in `package.json.files`, and build before publish.
- Add `npm pack` and installed-CLI smoke that starts `git-stacks web --no-open`, loads all assets/CSS, pairs, fetches snapshot, and opens a terminal on supported hosts.
- Add diagnostics for principal/session/socket counts, session states, replay range/gaps, ACK lag, backpressure closes, and cleanup failures. Never log codes, cookies, env, paths, terminal bytes, input, or signal token.
- Add Linux long-run/stress gates and release documentation for local-only security, TTL/restart behavior, separate native/web sessions, and recovery limitations.
- Run actual-host macOS and Windows launch-policy, PTY, process-tree, resize, Unicode/IME, browser, and package tests. Capability discovery remains false until each platform passes.
- Verify latest supported Chrome/Edge, Firefox, and Safari according to xterm's browser policy; document unavailable CI coverage honestly.

### Acceptance

- Tarball inventory contains all and only expected web assets; installed package has no source-tree dependency and makes no network requests.
- Existing package CLI/TUI/native service entry points still work with browser assets present.
- Linux stress repeatedly creates/uses/reconnects/closes terminals with bounded RSS and no owned descendants.
- Platform capability matrix is generated from actual test evidence, not inferred from Bun documentation.
- Support docs state local-only scope, pairing model, data exposure boundary, quotas, reconnect/history behavior, service-restart loss, and separate live-session ownership.

### Artifacts this slice produces

- production web asset package/manifest and publish scripts
- installed-package and browser compatibility tests
- safe web-terminal diagnostics
- platform capability matrix and operator/security documentation

## Slice 07 — Independent security and reliability closure

### Objective

Re-audit the integrated production path independently and close gaps before treating 107.3 as complete.

### Required review

- Adversarial HTTP/WS page exercises DNS-rebinding Host, wrong/missing/null Origin, cross-site forms/fetch, Fetch Metadata variations, cookie replay/expiry/logout, pairing race/replay/prefetch, oversized/unknown payloads, and terminal ID enumeration.
- Terminal fuzz/property tests exercise arbitrary byte/chunk boundaries, OSC ambiguity, malformed UTF-8, cursor/replay ordering, snapshot races, resize storms, output flood, and connection churn.
- Process review exercises close during spawn/replay, child exit during shutdown, stubborn descendants, failed TERM/KILL, principal expiry, and service stop ordering.
- Data-flow audit traces every `/web` response/event/control/log field back to an explicit browser schema and checks sentinel secrets.
- Browser review uses the real installed app and real PTYs; synthetic unit tests are not sufficient for final interaction acceptance.

### Acceptance

- No open high-severity browser-auth, process-ownership, secret-exposure, unbounded-memory, signal-token, or terminal-ordering finding remains.
- All medium findings are fixed or explicitly deferred with user-visible limitations and roadmap ownership.
- Root typecheck, focused/unit/integration/browser tests, dependency cycle check, `verify:gates`, package smoke, Linux process cleanup, and interactive UAT pass.
- A final verification document maps every locked decision and success criterion to current evidence.

## Cross-cutting threat model

| Threat | Mitigation | Verification owner |
|---|---|---|
| Malicious website controls local terminal | Strict cookie principal, exact Host/unsafe Origin/WS Origin, Fetch Metadata, no CORS, one-use CLI pairing | Slices 01, 07 |
| XSS gains service authority | No bearer in browser, strict CSP/assets, text-only metadata, scoped principal and operations | Slices 01, 05, 07 |
| Browser supplies arbitrary shell launch | IDs/revision-only strict schema and internal resolver | Slice 02 |
| Snapshot/operation leaks machine secrets | Explicit browser DTOs and sentinel negative tests | Slices 01, 04, 07 |
| Slow client exhausts service | Bounded socket/ring/headless/session stores and pressure disconnect/resync | Slice 03 |
| Reconnect corrupts terminal view | Cursor-framed output, write-completion ACK, replay barrier, headless baseline | Slice 03 |
| Terminal output forges cross-surface attention or leaks token | Per-session token, service-derived attribution, private OSC consume/filter | Slice 02 |
| Close kills unrelated process | Verified process group/tree identity and bounded platform adapter | Slices 02, 06, 07 |
| Service idle exit kills active work | Live/retained terminal lifecycle accounting | Slice 03 |

## Global completion criteria

- Browser code has no direct filesystem, Git, descriptor, credential, launch, PID, or process-tree access.
- Existing `/v1`, CLI/TUI, and native service clients remain compatible.
- Multiple real browser terminals pass input/output/resize/tab/reconnect/reload/close and no-orphan checks.
- Every queue/store/lifetime is bounded and observable without logging content or secrets.
- Signals from web terminals are authenticated, attributed, stripped from terminal bytes, and shared through the existing journal.
- The installed package serves all assets locally and passes real-browser interactive UAT.
- Local-only and separate-session limitations are explicit; macOS/Windows claims match actual-host evidence.
