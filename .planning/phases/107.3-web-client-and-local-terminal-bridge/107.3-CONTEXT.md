# 107.3 Context

This captures the decisions for the provisional web-client package. It does not register 107.3 in the live roadmap or start execution.

<decisions>

## Locked decisions

- **D-01 — Service authority:** The existing Bun/TypeScript git-stacks service is the only browser-facing machine authority. It performs filesystem, Git, workspace, operation, launch-resolution, PTY, process-lifecycle, and signal-ingress work.
- **D-02 — Thin browser:** The browser renders browser-safe state, sends stable IDs and user intents, and hosts xterm.js. It never reads git-stacks files, runs Git, resolves paths, constructs launch commands, or receives service credentials and raw launch environments.
- **D-03 — Reuse before addition:** Browser support adapts the existing snapshot builder, launch resolver, event journal/broker, operation registry, workspace-change monitor, creation catalog, and signal contracts in process. It does not create a second workspace engine or proxy back through the service's own HTTP API.
- **D-04 — Web-only terminal ownership:** The service may own PTYs created for browser terminal surfaces. This is a narrow browser adapter, not a change to native Ghostty ownership or to git-stacks' integration-neutral stance for external terminal/session products.
- **D-05 — Separate live sessions:** Native terminal surfaces, external integration sessions, and browser terminal sessions remain separate process identities. Cross-client live PTY sharing and durable multiplexer behavior are not part of the first implementation.
- **D-06 — Stable launch intents:** Browser terminal creation accepts only `workspace_id`, `repository_id`, optional `command_id`, `expected_revision`, and dimensions. The service resolves all executable, cwd, environment, port, and provider-integration details.
- **D-07 — Local-only trust boundary:** The web server binds only to `127.0.0.1`. LAN/public hosting, remote access, and hosted authentication require a separate threat model and are out of scope.
- **D-08 — Paired browser principal:** `git-stacks web` obtains a one-use pairing capability through the existing bearer-authenticated `/v1` service. Browser JavaScript receives only an HttpOnly, scoped web-principal cookie after a same-origin exchange.
- **D-09 — Service-owned continuity:** Short reconnects use cursor-based delta replay. Fresh page loads and replay gaps use bounded service-side headless terminal state when supported; otherwise the UI must show an explicit history-loss state. Duplicate or silently partial output is forbidden.
- **D-10 — Signals stay authenticated:** Each web terminal gets a service-generated surface ID and signal token. The service incrementally recognizes, validates, strips, and publishes app-owned OSC 9 signal frames before terminal bytes reach the browser. The token never crosses the web boundary.
- **D-11 — Product semantics:** The web client targets the same workspace-first behavior as the native client: exact workspace/repository tab scoping, command-only launcher, creation/operation progress, signal attention, external invalidation, and explicit focus. GTK/Ghostty implementation details are not shared.
- **D-12 — Separate browser build:** Browser code uses a dedicated DOM build/typecheck boundary and production-packaged assets. It must not inherit the root OpenTUI JSX configuration or rely on a CDN/service worker.

## The agent's discretion

- Exact internal module names and whether browser route composition is split from `src/service/server.ts` immediately or after the first walking slice.
- Final numeric quotas and retention durations, provided tests prove all queues, sessions, principals, and replay stores are bounded.
- Whether headless serialization ships in the first terminal slice or the immediately following continuity slice. No reload-continuity claim may precede it.
- Visual styling details beyond preserving the locked native product hierarchy and accessibility behavior.

## Deferred

- Native/web shared PTY attachment.
- Durable browser terminals across service restart or machine reboot.
- Multi-user collaboration or multiple concurrent writers to one terminal.
- LAN/public/remote mode.
- Mobile terminal editing as a primary target; narrow layouts should remain usable, but desktop browsers are the initial acceptance surface.

</decisions>

## Planning boundary

The implementation can proceed independently of minor native signal-integration fixes as long as shared `/v1`, signal, or launch contracts do not change. Before execution, re-read those live contracts and absorb any compatible changes; do not modify the native terminal renderer for this work.
