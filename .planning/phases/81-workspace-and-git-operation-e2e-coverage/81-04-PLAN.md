**Import setup:**
```typescript
import { describe, test, expect, beforeEach, afterEach } from "bun:test"
```

Standard beforeEach/afterEach with tmpDir, git env isolation, and a workspace fixture backed by `makeRepoWithRemote`.

**describe("status --fetch")** — Proves that `--fetch` updates remote state.

1. **"status --fetch updates behind count after remote gets new commits"** — Build fixture. Run `status --json` first, verify `behind: 0`. Then push a new commit to main on origin (from mainPath: `git checkout main`, add file, commit, push, `git checkout -`). Run `status --json --fetch`. Verify the repo now shows `behind > 0` (or at least `behind` field is present as a number).

2. **"status without --fetch does NOT update behind count from remote"** — Same setup: push new commit to origin. Run `status --json` (NO `--fetch`). Verify `behind` is still 0 (stale — hasn't fetched yet).

3. **"status --fetch succeeds even when remote has no changes"** — Build fixture. Run `status --json --fetch`. Verify exit code 0 and `behind: 0`.

4. **"status --fetch updates ahead count for local commits"** — Build fixture. Make a commit on the workspace branch in taskPath (committed but not pushed). Run `status --json --fetch`. Verify `ahead > 0`.

5. **"status --fetch handles workspace with multiple repos"** — Build fixture with 2 repos (both remote-backed). Push new commit to one repo's origin but not the other. Run `status --fetch --json`. Verify different behind counts for each repo.

**Implementation notes:**
- The key insight is that `status` without `--fetch` uses cached/local tracking refs, while `--fetch` actually fetches from the remote first. The test must prove this difference.
- Use the `--json` flag for structured assertions on `ahead` and `behind` fields.
- Creating "behind" state: push a commit to origin/main that isn't in the workspace branch. The workspace branch was created from main, so any new commit on main makes the workspace branch behind.
- Creating "ahead" state: make a commit on the workspace branch in taskPath but don't push.
- Read `status --json` output shape carefully from existing `status-json.test.ts` pattern.
