# Phase 1.1: File and Folder Copy/Symlink Support - Research

**Researched:** 2026-03-17
**Domain:** Node.js/Bun filesystem operations, glob expansion, Zod schema extension
**Confidence:** HIGH

## Summary

This phase extends `src/lib/files.ts` and `src/lib/config.ts` with folder support, glob expansion, two config levels, absolute/home-relative source paths, and a loud-fail three-case error policy. All required primitives are available natively in Bun without adding any third-party dependencies.

The existing codebase provides nearly every building block needed: `cpSync` (recursive folder copy), `symlinkSync` (directory symlinks), `Bun.Glob` (glob expansion), and `expandHome` from `paths.ts`. The two-level config model (workspace-instance and per-repo) maps cleanly onto the existing schema structure. The main work is schema additions, a new orchestration layer, and upgrading `applyFileOperations` from void-returning with silent skips to returning a discriminated union result with loud failure.

**Primary recommendation:** Use `Bun.Glob` with `dot: true` for glob expansion (no new packages), `cpSync` with `recursive: true` for folder copy, and `symlinkSync` for directory symlinks. Add `files` to `StackSchema`, `WorkspaceSchema`, and `WorkspaceRepoSchema` with `.optional()` or `.default([])`. Upgrade `applyFileOperations` return type to `{ ok: true } | { ok: false; error: string }`.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- `copy` and `symlink` only — hardlinks deferred to a future phase
- Files AND folders in the same arrays — no separate `copy_dirs`/`symlink_dirs` keys
- Glob patterns supported in all arrays (workmux-style) — e.g., `.env.*`, `secrets/**`
- Two distinct config levels:
  - **Level 1 — Workspace-instance level:** `files:` on `StackSchema` and `WorkspaceSchema`, applied once to workspace instance root `{tasksDir}/{wsName}/`, does NOT apply to each repo
  - **Level 2 — Per-repo level:** `files:` on `StackRepoSchema` (existing) and `WorkspaceRepoSchema` (new), applied to that specific repo's `task_path`
- Merge semantics: within each level, stack definition and workspace definition merge additively; two levels are independent
- Source path resolution:
  - Workspace-instance level: relative → resolved relative to workspace instance root; absolute → as-is; `~/` → `expandHome()`
  - Per-repo level: relative → resolved relative to repo's `main_path`; absolute → as-is; `~/` → `expandHome()`
- Three-case error rule (both levels, every resolved path after glob expansion):
  1. Destination exists → skip (no error, source not checked)
  2. Destination missing, source missing → fail loud; surface error to user
  3. Destination missing, source exists → apply (copy or symlink)
- Replaces current `// silently skip` pattern in `applyFileOperations`
- Applies on both `new` (workspace creation) and `open` (workspace re-open) using same three-case logic

### Pattern 2: Three-Case Apply Logic

The three-case rule replaces all `if (existsSync(src))` + silent-skip patterns in the current `applyFileOperations`.

```typescript
// Pseudocode — actual implementation in files.ts
function applyEntry(
  op: "copy" | "symlink",
  src: string,
  dst: string
): { ok: true } | { ok: false; error: string } {
  // Case 1: destination exists — skip silently
  if (existsSync(dst)) return { ok: true }  // lstatSync handles symlinks too
  // Case 2: destination missing, source missing — fail loud
  if (!existsSync(src)) return { ok: false, error: `Source not found: ${src}` }
  // Case 3: destination missing, source exists — apply
  if (op === "copy") {
    mkdirSync(dirname(dst), { recursive: true })  // ensure parent for symlink case below
    cpSync(src, dst, { recursive: true })
  } else {
    mkdirSync(dirname(dst), { recursive: true })  // symlinkSync requires parent to exist
    symlinkSync(src, dst)
  }
  return { ok: true }
}
```

**Critical:** `existsSync` follows symlinks (returns true for existing symlink pointing to existing target, false for dangling). Use `lstatSync` only if you need to distinguish file-vs-symlink. For the three-case check, `existsSync` is correct — an existing symlink (dangling or not) at dst is treated as "destination exists, skip."

**Critical for symlinks:** `symlinkSync` throws `ENOENT` if the parent directory does not exist. Always `mkdirSync(dirname(dst), { recursive: true })` before calling `symlinkSync`. `cpSync` does not require this (it creates parents automatically), but do it for both for consistency.

### Anti-Patterns to Avoid

- **Silent skip on source missing:** Current code `if (existsSync(src)) { ... }` silently ignores missing sources. Replace with three-case logic that fails loud on Case 2.
- **`existsSync` for destination check on dangling symlinks:** `existsSync` returns `false` for dangling symlinks (target missing). Use `lstatSync` wrapped in try/catch, or check if `lstatSync` returns without throwing — this tells you "something is at dst path even if it's a broken symlink." For this phase's purpose (skip if dst present), the correct check is: try `lstatSync(dst)` — if it doesn't throw, dst exists, skip.
- **Glob without `dot: true`:** Verified that `Bun.Glob('.env.*').scanSync({ cwd })` returns empty array without `dot: true`. This would silently fail to match dotfiles — a primary use case for this feature.
- **`symlinkSync` without parent mkdir:** Verified that `symlinkSync` throws `ENOENT` when the parent directory does not exist. Always `mkdirSync(dirname(dst), { recursive: true })` before `symlinkSync`.
- **Calling `applyFileOperations` for workspace-instance level ops:** The existing function only handles per-repo ops. The workspace-instance level is a new, separate execution path.

## Common Pitfalls

### Pitfall 1: Dotfile Glob Returns Empty Without `dot: true`
**What goes wrong:** `new Bun.Glob('.env.*').scanSync({ cwd: base })` returns `[]` even when `.env.local` and `.env.prod` exist.
**Why it happens:** Bun.Glob defaults to `dot: false`, which excludes dotfiles (files beginning with `.`) from all glob expansions — even when the pattern explicitly starts with `.`.
**How to avoid:** Always pass `dot: true` when calling `g.scanSync({ cwd, dot: true })`. This is not optional for this use case.
**Warning signs:** Test with `.env.*` pattern against a directory containing `.env.local`; expect non-empty results.

### Pitfall 2: `symlinkSync` Fails When Destination Parent Missing
**What goes wrong:** `symlinkSync(src, dst)` throws `ENOENT: no such file or directory` when `dirname(dst)` does not exist.
**Why it happens:** Unlike `cpSync`, `symlinkSync` does not create parent directories automatically.
**How to avoid:** Always call `mkdirSync(dirname(dst), { recursive: true })` before `symlinkSync`.
**Warning signs:** Tests with deeply nested destination paths failing with ENOENT.

### Pitfall 3: `existsSync` Returns False for Dangling Symlinks
**What goes wrong:** If Case 1 uses `existsSync(dst)` to check "destination already exists," a broken/dangling symlink at dst returns `false`, causing the code to try re-applying the operation and potentially failing or overwriting.
**Why it happens:** `existsSync` follows symlinks and returns `false` if the symlink's target does not exist.
**How to avoid:** For the "destination exists" check in Case 1, use `lstatSync(dst)` inside a try/catch — if it does not throw, something is at `dst` (file, directory, or symlink regardless of target health) → skip.
**Warning signs:** Idempotent re-apply breaking on workspaces that have dangling symlinks.

### Pitfall 4: Glob Result Paths Are Relative — Must Join with `cwd`
**What goes wrong:** `Bun.Glob.scanSync({ cwd })` returns relative paths like `"subdir/file.txt"`, not absolute paths.
**Why it happens:** By design — scanSync returns paths relative to `cwd`.
**How to avoid:** `const absolutePath = join(cwd, relativeResult)` for every glob result before passing to `applyEntry`.
**Warning signs:** `ENOENT` errors on paths that look correct but are missing the leading directory.

### Pitfall 5: Workspace-Instance Level `files` Applied to Wrong Target
**What goes wrong:** Workspace-instance-level file ops are applied to each repo's `task_path` instead of the workspace root `{tasksDir}/{wsName}/`.
**Why it happens:** Confusion between the two levels; easy to accidentally reuse per-repo logic.
**How to avoid:** Workspace-instance ops use `join(getTasksDir(config.workspace_root), wsName)` as their target directory, not any repo path. This is a single execution, not per-repo.
**Warning signs:** Files appearing inside repo worktrees instead of at the workspace root.

### Pitfall 6: Stack Files Not Merged with Workspace Files (Additive)
**What goes wrong:** Stack-defined `files.copy` is completely replaced by workspace-defined `files.copy` instead of being merged.
**Why it happens:** Naive `{ ...stack.files, ...workspace.files }` spread replaces array fields rather than concatenating them.
**How to avoid:** Explicit additive merge: `copy: [...(stack.files?.copy ?? []), ...(ws.files?.copy ?? [])]`.

### Pitfall 7: Source Relative Path Resolved Against Wrong Base
**What goes wrong:** Per-repo level relative source paths are resolved against `task_path` instead of `main_path`. This matters when the worktree doesn't have the large binary (it was never checked in).
**Why it happens:** The intent is "source lives in the main clone"; only the destination is the task/worktree path.
**How to avoid:** Per-repo source resolution uses `main_path` as the base for relative paths. Per CONTEXT.md: "Relative path → resolved relative to that repo's `main_path`".

### Three-Case Apply Entry
```typescript
// Source: derived from CONTEXT.md + verified primitives
import { cpSync, symlinkSync, mkdirSync } from "fs"
import { dirname, join } from "path"

type ApplyResult = { ok: true } | { ok: false; error: string }

function applyEntry(op: "copy" | "symlink", src: string, dst: string): ApplyResult {
  // Case 1: destination exists → skip
  try { lstatSync(dst); return { ok: true } } catch { /* dst absent, continue */ }
  // Case 2: destination missing, source missing → fail loud
  try { lstatSync(src) } catch {
    return { ok: false, error: `Source not found: ${src}` }
  }
  // Case 3: apply
  mkdirSync(dirname(dst), { recursive: true })
  if (op === "copy") {
    cpSync(src, dst, { recursive: true })
  } else {
    symlinkSync(src, dst)
  }
  return { ok: true }
}
```

## Open Questions

1. **Zero-match glob handling: warn or error?**
   - What we know: CONTEXT.md says "leaning toward warn since glob zero-match is often intentional"
   - What's unclear: Where/how to surface the warning — `onProgress` callback? `console.warn`? Part of result type?
   - Recommendation: Return a warning string in the result (e.g., `{ ok: true; warnings?: string[] }`) so callers can display it. Upgrade the `{ ok: true } | { ok: false; error: string }` type to include an optional `warnings` array.

2. **`openWorkspace` access to stacks for file ops**
   - What we know: `loadWorkspaceStacks()` is already called later in `openWorkspace` (line ~466 in workspace-ops.ts) for post_open hooks and env. `applyFileOpsForRepo` needs the stack to access `stackRepo.files`.
   - What's unclear: Whether to call `loadWorkspaceStacks` earlier in `openWorkspace`, or pass stacks through as a parameter.
   - Recommendation: Move the `loadWorkspaceStacks(workspace)` call to the top of `openWorkspace`, before the worktree recreation block. It's a cheap YAML read and this consolidates stack access.

3. **Error handling surface in workspace-wizard.ts**
   - What we know: `workspace-wizard.ts` uses `p.log.error()` + `process.exit(1)` for hook failures. File op errors should follow the same pattern.
   - What's unclear: Whether a single missing source should abort the entire workspace creation, or just log and continue (since workspace creation already succeeded at that point).
   - Recommendation: Abort on loud-fail errors (consistent with the hook failure behavior). The three-case rule makes Case 2 an explicit error — treat it like a failed hook.
