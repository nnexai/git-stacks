# Phase 1.1: File and Folder Copy/Symlink Support - Context

**Gathered:** 2026-03-17
**Status:** Ready for planning

<domain>
## Phase Boundary

Extend the existing file operations system (`src/lib/files.ts`) to support: folder operations alongside files (same arrays, no separation), glob pattern expansion, absolute and `~/`-relative source paths, two config levels (stack/workspace-level for all repos, per-repo level for individual repos), loud failure on missing sources, and idempotent re-application on `open`.

Hardlinks and cross-repo source paths are explicitly out of scope for this phase.

</domain>

<decisions>
## Implementation Decisions

### Operations supported
- `copy` and `symlink` only — hardlinks deferred to a future phase
- Files AND folders in the same arrays — no separate `copy_dirs`/`symlink_dirs` keys
- Glob patterns supported in all arrays (workmux-style) — e.g., `.env.*`, `secrets/**`

### Config levels
Two levels, both additive (not override):

**Level 1 — Workspace-wide (new):**
- `files:` block at the top of `StackSchema` → merged with `files:` on `WorkspaceSchema`
- Combined list applied to ALL repos in the workspace
- Useful for: `.env` shared across all repos, `node_modules` shared build cache, workspace-wide large binaries

**Level 2 — Per-repo (extends existing):**
- `files:` on `StackRepoSchema` (already exists) → merged with `files:` on `WorkspaceRepoSchema` (new)
- Combined list applied to that specific repo only

**Merge semantics:** Both levels are additive — stack-level + workspace-level run together; stackRepo + workspaceRepo run together. Stack/workspace-level applies to each repo in addition to that repo's own `files:` block.

### Source path resolution
Consistent at all levels:
- Relative path → resolved relative to the repo's `main_path` (each repo's main clone)
- Absolute path → used as-is
- `~/`-relative path → expanded to user's home directory

For workspace-wide ops with relative paths, each repo's `main_path` serves as the source base — the same relative path is applied to every repo. Use absolute/`~/` paths when sharing a single source across repos (e.g., `~/.shared/secrets/.env`).

### Error behavior
Fail loud — a missing source path (after glob expansion) is an error surfaced to the user, not a silent skip. Current `// silently skip` pattern in `applyFileOperations` is removed.

### When to apply
- **`new`**: Apply all file ops during workspace creation (existing behavior)
- **`open`**: Re-apply idempotently — only apply ops where the destination does not yet exist. Do not overwrite existing destinations.

### Claude's Discretion
- Glob library choice (fast-glob, micromatch, or Bun native — check what's already in dependencies)
- Exact merge function signature (how merged lists are passed into `applyFileOperations`)
- Whether `WorkspaceRepo.files` is optional with `.optional()` default (yes — backward compat required)
- How folder copy is implemented (Node `cpSync` with `recursive: true` vs shell fallback)
- How to handle a glob pattern that matches zero files (warn or error — leaning toward warn since glob zero-match is often intentional)

</decisions>

<specifics>
## Specific Ideas

- Design inspiration: workmux's `files:` block — same arrays handle files and folders, glob expansion, two config levels that merge additively
- Use cases driving the design: `.env` files in a central place, `node_modules` or `.pnpm-store` shared via symlink, large model/binary files shared via symlink from a `~/models/` directory
- Source paths with `~/` must expand via the existing `expandHome()` utility in `paths.ts`

</specifics>

<deferred>
## Deferred Ideas

- Hardlinks (`hardlink:` array) — deferred; filesystem limitation (dirs can't be hardlinked) makes design non-trivial
- Global config level `files:` — deferred; applying to every repo everywhere conflicts with loud-fail semantics
- Cross-repo source paths (referencing another repo's path as source) — explicitly out of scope; use absolute/`~/` paths instead
- `"<global>"` merge token (workmux-style include marker) — could allow workspace-level to opt into stack-level list; deferred for now, additive merge is sufficient

</deferred>
