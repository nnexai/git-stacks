## Decisions Made

### 1. Isolation Strategy
- **Options presented:** Custom bun test runner, Vitest migration, Bun subprocess isolation
- **Selected:** Custom bun test runner
- **Rationale:** No new deps, stays on bun:test, just orchestrates processes. Unit tests together, integ tests isolated.

### 2. DI Object Cleanup
- **Initial selection:** Remove all, use mock.module in unit tests
- **User revisited:** Raised concern about testing logic in modules that wrap external programs (niri, glab, etc.). Mocking the whole module loses the ability to test the binding logic.
- **Revised options:** Keep external program seams, remove internal hacks; Keep all _exec; Remove all; Keep all
- **Final selection:** Keep external, remove internal
- **Rationale:** _exec/_detect objects are legitimate testing seams at the external program boundary. _cwdDetect/_resolveWorkspaceDeps are internal isolation hacks that process isolation makes unnecessary.

### 3. Cache-Busting Removal
- **Options presented:** Remove all, Remove most keep for config
- **Selected:** Remove all
- **Rationale:** Process isolation eliminates the need entirely.

### 4. mock.module Future
- **Options presented:** Keep with factory helpers, Minimize to paths only, Eliminate entirely
- **Selected:** Keep with complete factory helpers
- **Rationale:** mock.module works correctly when contamination is controlled. Complete factories prevent the partial-export footgun structurally.

## Corrections Applied

- DI objects: "Remove all" → "Keep external program seams, remove internal hacks"
  - User insight: _exec objects for external CLIs are a legitimate testing pattern, not an isolation hack. Removing them would prevent testing the logic layer that wraps those CLIs.
