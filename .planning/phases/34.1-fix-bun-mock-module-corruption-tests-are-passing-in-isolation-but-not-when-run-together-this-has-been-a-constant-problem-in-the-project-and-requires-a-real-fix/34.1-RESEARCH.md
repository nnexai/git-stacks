# Phase 34.1: Fix Bun Mock Module Corruption - Research

**Researched:** 2026-03-25
**Domain:** Bun test runner — `mock.module` cross-file contamination
**Confidence:** HIGH

## Summary

Bun's `mock.module()` API replaces module entries in a **global, process-wide module registry** that persists across all test files in the same `bun test` run. When a test file calls `mock.module("@/lib/config", factory)` with a factory that returns only a subset of the real module's exports, that partial mock remains active for all subsequently-loaded test files. Any module that is **freshly loaded** (e.g., via a cache-busting `await import("@/lib/X?query-string")`) and transitively imports the partially-mocked module will fail with a static ESM linker error: `SyntaxError: Export named 'X' not found in module '...'.`

This is not a runtime "undefined" error — it is a **module linking error** that fires when Bun links a fresh module's static import bindings against a mock that doesn't advertise the needed export. The error message names the export and the mocked module, but the root cause is always an incomplete mock factory in a **different test file** that ran earlier.

The project already recognises this problem and uses the **injectable deps pattern** (`_exec`, `_cwdDetect`, `_resolveWorkspaceDeps`, `_detect`) to avoid `mock.module` cross-file contamination in critical modules. The fix for this phase is to audit every `mock.module` call across all 64 test files and ensure each mocked module is given a **complete export surface** — every export the module actually ships must appear in the mock factory, even if it's just a no-op stub.

**Primary recommendation:** Audit all `mock.module` factories. For each mocked module, compare the factory's exports against the real module's `export` statements and add stubs for anything missing. Centralise the complete mock shapes in `tests/helpers.ts` using `makeConfigMock(overrides?)`, `makeWorkspaceOpsMock(overrides?)`, `makeForgeUtilsMock(overrides?)`, and `makeIssueUtilsMock(overrides?)` factory helpers so future tests can't accidentally create a partial mock.

## Root Cause Analysis (HIGH confidence)

### The Bun `mock.module` Mechanism

`mock.module(path, factory)` in Bun replaces the module at `path` in the **process-global module registry** with the object returned by `factory`. This replacement:

1. Persists for the entire `bun test` process — there is no per-file reset.
2. Affects modules loaded via any path that resolves to the same file (`@/lib/config`, `../../src/lib/config`, `../../../src/lib/config` all refer to the same entry in Bun's registry).
3. Does NOT get reset by `mock.restore()` — the official docs explicitly state: *"Doing so does not reset the value of modules overridden with `mock.module()`."*
4. Causes an ESM **link-time error** (not a runtime error) when a freshly-loaded module static-imports a named export that the mock factory did not include.

### Why Cache-Busting Does Not Fully Protect

Query-string cache-busting (`await import("@/lib/X?query")`) forces a fresh load of `X` from disk. However, if `X.ts` itself has a static `import { Y } from "./dependency"`, that dependency is **not** cache-busted — Bun resolves it against the module registry and may find a partial mock. The fresh `X` module then fails to link because `Y` is not in the mock's export list.

### Verified Failure Chains (confirmed by bisection)

**Chain 1: `@/lib/config` partial mock → `forge-utils` link failure**

`forge-utils.ts` statically imports `{ workspaceExists, readWorkspace, readRegistry }` from `"../config"`. Several tests mock `@/lib/config` with only one or two exports. When a freshly cache-busted module that depends on `forge-utils` is loaded, Bun tries to link `forge-utils.ts` against the partial config mock — the missing `readWorkspace`/`readRegistry` exports cause a link failure. The error surfaces as `"Export named 'resolveForgeRepoAnyMode' not found in forge-utils.ts"` on the importing side because `forge-utils` itself failed to produce its exports.

**Chain 2: `@/lib/config` partial mock → `workspace-ops` link failure**

`workspace-ops.ts` statically imports `WorkspaceSchema`, `readGlobalConfig`, `workspacePath`, `readRegistry`, and ~10 other exports from config. Tests that mock config with only `{ workspaceExists }` corrupt workspace-ops in the same way. Manifest as `"WorkspaceSchema not found in config.ts"`.

**Chain 3: `@/lib/workspace-ops` partial mock → `issue-utils` link failure**

`issue-utils.ts` statically imports from `../workspace-ops`. Six test files mock `@/lib/workspace-ops` without including `detectWorkspaceFromCwd`. When `issue-utils?unit-test` is loaded (cache-busted), it resolves `workspace-ops` without a query string, gets the partial mock, and the linker fails: `"detectWorkspaceFromCwd not found in workspace-ops.ts"`.

**Chain 4: `@/lib/integrations/forge-utils` partial mock → `gitea/github/gitlab` link failure**

`integration-commands.test.ts` mocks `forge-utils` without `resolveForgeRepoAnyMode` and `resolveRepoCwd`. Later tests that load fresh copies of `gitea.ts`, `github.ts`, or `gitlab.ts` (which statically import both exports) see the partial mock and fail.

**Chain 5: `@/lib/tmux` partial mock → `integrations/tmux` link failure**

`artifacts.test.ts` mocks `@/lib/tmux` without `killTmuxSession`, `tmuxSessionExists`, `_exec`, and `createTmuxSession`. `integrations/tmux.ts` statically imports all of these; the cache-busted load of `@/lib/integrations/tmux?artifacts-test` fails to link.

**Chain 6: `@/lib/integrations/issue-utils` partial mock → `jira` link failure**

`integration-commands.test.ts` mocks `issue-utils` without `resolveWorkspaceArg`. `jira.ts` statically imports `resolveWorkspaceArg`; any subsequent fresh load of `jira.ts` fails.

## Architecture Patterns

### The Injectable Deps Pattern (already in use — extend it)

The codebase already solves the hardest cross-file contamination by making shell/fs operations injectable:

```typescript
// In src/lib/tmux.ts
export const _exec = {
  run: async (args: string[]): Promise<CmdResult> => { /* real impl */ },
}

// In test:
const { _exec, killTmuxSession } = await import("@/lib/tmux?tmux-test")
_exec.run = mock(async () => ({ exitCode: 0, stdout: "" }))
```

For the remaining problem modules (`config`, `workspace-ops`, `forge-utils`, `issue-utils`, `tmux`), the fix is NOT to add more injectable deps to production code — those modules are already structured correctly. The fix is to make the `mock.module` factories in the test files **return complete module shapes**.

### The Complete Mock Factory Pattern (the fix)

Add factory helpers to `tests/helpers.ts` that return a fully-populated mock object for each problem module. Tests override only the exports they care about:

```typescript
// tests/helpers.ts

import { mock } from "bun:test"

/** Returns a complete mock of @/lib/config. Override what you need. */
export function makeConfigMock(overrides: Record<string, unknown> = {}) {
  return {
    // Value exports
    WorkspaceSchema: { parse: mock(() => ({})), safeParse: mock(() => ({ success: true, data: {} })) },
    WorkspaceRepoSchema: { parse: mock(() => ({})), safeParse: mock(() => ({ success: true, data: {} })) },
    TemplateSchema: { parse: mock(() => ({})), safeParse: mock(() => ({ success: true, data: {} })) },
    RepoRegistryEntrySchema: { parse: mock(() => ({})), safeParse: mock(() => ({ success: true, data: {} })) },
    GlobalConfigSchema: { parse: mock(() => ({})), safeParse: mock(() => ({ success: true, data: {} })) },
    // Function exports
    formatZodError: mock(() => ""),
    readGlobalConfig: mock(() => ({ workspace_root: "/tmp", integrations: {} })),
    writeGlobalConfig: mock(() => {}),
    workspacePath: mock((name: string) => `/tmp/workspaces/${name}.yml`),
    workspaceExists: mock(() => false),
    readWorkspace: mock(() => { throw new Error("not configured") }),
    writeWorkspace: mock(() => {}),
    listWorkspaces: mock(() => []),
    readRegistry: mock(() => []),
    writeRegistry: mock(() => {}),
    listRegistryEntries: mock(() => []),
    templatePath: mock((name: string) => `/tmp/templates/${name}.yml`),
    templateExists: mock(() => false),
    readTemplate: mock(() => { throw new Error("not configured") }),
    writeTemplate: mock(() => {}),
    listTemplates: mock(() => []),
    expandBranchPattern: mock((p: string) => p),
    expandHome: mock((p: string) => p),
    ...overrides,
  }
}
```

Usage in test files:

```typescript
import { makeConfigMock } from "../helpers"

const workspaceExistsMock = mock(() => false)

mock.module("@/lib/config", () => makeConfigMock({
  workspaceExists: workspaceExistsMock,
  readWorkspace: mock(() => myWorkspaceFixture),
}))
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead |
|---------|-------------|-------------|
| Per-file mock isolation | Custom mock registry / process spawning | Complete mock factories in helpers.ts |
| Mock cleanup between files | `afterAll(() => mock.restore())` | Complete mocks (restore doesn't work for mock.module) |
| Production code injectable deps for already-injectable modules | More `_exec`-style objects | Just fix the partial mock factory |

**Key insight:** The injectable deps pattern is already in place for shell operations. The problem is purely in test files passing incomplete module shapes to `mock.module`. No production code changes are required.

## Common Pitfalls

### Pitfall 1: Adding only the missing export that caused the failure

**What goes wrong:** A developer sees "WorkspaceSchema not found in config.ts", adds `WorkspaceSchema` to the mock factory, ships it, and the tests fail again next week with a different missing export.

**Why it happens:** The partial mock is still partial — only one more export was added. The next test file to be added will likely need another export.

**How to avoid:** When fixing any mock factory, audit the FULL export list of the target module (`grep "^export" src/lib/config.ts`) and ensure ALL exports are present in the factory. Use `makeConfigMock()` from helpers.ts.

**Warning signs:** The error message says "Export named X not found" and X is in the real module but not in the mock factory.

### Pitfall 2: Assuming `mock.restore()` fixes mock.module

**What goes wrong:** A developer adds `afterAll(() => mock.restore())` expecting it to undo `mock.module` calls, but the module mock persists.

**Why it happens:** Bun's official docs state: *"Doing so does not reset the value of modules overridden with `mock.module()`."*

**How to avoid:** Never rely on `mock.restore()` for module cleanup. Use complete mock factories instead.

### Pitfall 3: Cache-busting with different queries creates duplicate module instances

**What goes wrong:** A test does `await import("@/lib/forge-utils?test-A")` and another does `await import("@/lib/forge-utils?test-B")`. Both load real module code, but if they share mutable state (`_detect` object), mutations from one don't affect the other.

**Why it happens:** Each query string creates a separate module instance. This is intentional for isolation — but breaks tests that need to share injectable state.

**How to avoid:** Use a CONSISTENT query string across a test file (one per file), or use `_exec`/`_detect` style injection (mutate the exported object, don't re-import).

### Pitfall 4: Using `../../src/lib/config` vs `@/lib/config` — they ARE the same

**What goes wrong:** A developer believes that using relative paths for imports instead of `@/` aliases bypasses mock.module, so they think direct imports in `config.test.ts` are safe.

**Why it happens:** Bun normalises both paths to the same absolute file path before looking up the module registry. `mock.module("@/lib/config", ...)` DOES affect `import { X } from "../../src/lib/config"`.

**How to avoid:** Always assume a `mock.module` call affects ALL import paths that resolve to the same file.

### Pitfall 5: The "error test file" is not the "cause test file"

**What goes wrong:** The test file reported in the error output is not the one with the broken mock. Developer looks at `issue-utils.test.ts` which reports "WorkspaceSchema not found in config.ts", fixes `issue-utils.test.ts`'s config mock, but the problem was actually in `github.test.ts` running earlier.

**Why it happens:** The CAUSE is the partial mock set by the FIRST test file to mock the module with incomplete exports. The SYMPTOM appears in the NEXT test file that cache-busts something depending on that module.

**How to avoid:** Fix ALL partial mocks for a given module in a single pass. The `grep -rn "mock.module.*@/lib/config"` command lists all 16 affected files simultaneously.

## State of the Art

| Old Approach | Current Approach | Impact |

## Open Questions

1. **Should `makeConfigMock` stub schema `.parse`/`.safeParse` with real Zod objects or simple mocks?**
   - What we know: Config tests (`config.test.ts`) import schemas directly from `../../src/lib/config` (not mocked) and call `.parse()`. Other tests never call `.parse()` on the mock schemas.
   - What's unclear: Using real Zod objects in the mock would be more faithful but requires importing `src/lib/config` in helpers.ts, which may cause circular deps or side effects.
   - Recommendation: Use `{ parse: mock((x: any) => x), safeParse: mock((x: any) => ({ success: true, data: x })) }` as a simple structural stub. Tests that care about schema validation use direct imports, not mock.module.

2. **Are there other partially-mocked modules not in the current failure set?**
   - What we know: 20 failures, all traced to 5 modules. The 64-file test suite has 95 `mock.module` calls total.
   - What's unclear: Other partial mocks may exist that don't yet cause failures because no test has been added that loads a cache-busted module depending on them.
   - Recommendation: After Wave 2 (all config mocks fixed), do a final audit of remaining `mock.module` calls with `grep -rn "mock.module" tests/` and compare each against the real module's exports.
