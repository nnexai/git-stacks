## Summary

Bun's `mock.module()` API replaces module entries in a **global, process-wide module registry** that persists across all test files in the same `bun test` run. This causes link-time errors when a mock factory omits exports that downstream modules statically import.

The current codebase has **20 failing tests** in the full suite (`bun test tests/`). All 20 failures trace to three root causes:

1. `tests/lib/integration-commands.test.ts` mocks `@/lib/integrations/forge-utils` without `resolveForgeRepoAnyMode` and `resolveRepoCwd` — this is the most impactful root cause, contaminating 15+ downstream files that cache-bust-import anything depending on forge-utils.
2. `tests/lib/integration-commands.test.ts` mocks `@/lib/integrations/issue-utils` without `resolveWorkspaceArg` — contaminates `jira.ts` imports.
3. Multiple test files mock `@/lib/config` without `listTemplates`, `expandBranchPattern`, `WorkspaceSchema` (and other schema exports) — contaminates dashboard and wizard tests that need the full config API.
4. Multiple test files mock `@/lib/workspace-ops` without `detectWorkspaceFromCwd` — contaminates `issue-utils.ts` imports.
5. `tests/lib/integrations/artifacts.test.ts` mocks `@/lib/tmux` without `killTmuxSession`, `tmuxSessionExists`, `createTmuxSession`, `_exec`.

The fix per CONTEXT.md is a **two-track approach**: (1) build `scripts/test-runner.ts` that runs unit tests together and integ tests in per-file isolated processes, AND (2) add complete mock factory helpers to `tests/helpers.ts` so partial mocks are structurally impossible. Additionally, remove all 23 cache-busting query-string imports and remove the isolation-only DI objects `_cwdDetect` and `_resolveWorkspaceDeps` from production code.

**Primary recommendation:** Execute in three waves: Wave 0 (test runner + helpers.ts factories), Wave 1 (surgical mock completions for the 5 root causes), Wave 2 (cache-busting removal + DI object removal). The test runner alone does not fix the unit test failures — the mock factories are required for unit tests sharing a process.

## Architecture Patterns

### Pattern 1: Test Runner Script (`scripts/test-runner.ts`)

**What:** A TypeScript script run with Bun that orchestrates two test modes.

**Implementation approach:**

```typescript
// scripts/test-runner.ts
import { join, relative } from "path"
import { readdirSync, statSync } from "fs"

const ROOT = join(import.meta.dir, "..")
const TESTS_DIR = join(ROOT, "tests")

function globTestFiles(dir: string, pattern: RegExp): string[] {
  const results: string[] = []
  for (const entry of readdirSync(dir, { withFileTypes: true })) {
    const full = join(dir, entry.name)
    if (entry.isDirectory()) {
      results.push(...globTestFiles(full, pattern))
    } else if (pattern.test(entry.name)) {
      results.push(full)
    }
  }
  return results
}

// D-02: Unit test directories
const unitFiles = [
  ...globTestFiles(join(TESTS_DIR, "lib"), /\.test\.[jt]sx?$/),
  ...globTestFiles(join(TESTS_DIR, "tui", "dashboard"), /^(?!integ-).+\.test\.[jt]sx?$/),
]

// D-02: Integration test directories
const integFiles = [
  ...globTestFiles(join(TESTS_DIR, "commands"), /\.test\.[jt]sx?$/),
  ...globTestFiles(join(TESTS_DIR, "tui"), /\.test\.[jt]sx?$/),
  ...globTestFiles(join(TESTS_DIR, "tui", "dashboard"), /^integ-.+\.test\.[jt]sx?$/),
]
```

**Key design decisions (Claude's discretion):**
- Unit tests: single `Bun.spawn(["bun", "test", ...unitFiles])` call passes all files explicitly
- Integ tests: each file gets its own `Bun.spawn(["bun", "test", filePath])`, optionally in parallel
- The runner passes `--preload @opentui/solid/preload` to match bunfig.toml behavior (NOTE: bunfig.toml preload applies automatically; verify it propagates to child spawns)
- Exit code: 0 only if all spawns exit 0; non-zero otherwise
- Output: pipe and stream child process stdio to parent; prefix integ output with filename

**Bunfig.toml compatibility:** The `bunfig.toml` preload is defined under `[test]` section — Bun applies it automatically when `bun test` is invoked in the project root. Child spawns of the runner will also pick this up because they're invoked from the same directory. No additional `--preload` flags needed in the runner.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead |
|---------|-------------|-------------|
| Per-file mock isolation for unit tests | Manual mock registry, complex reset logic | Complete mock factories in helpers.ts |
| Mock cleanup between unit test files | `afterAll(() => mock.restore())` | Complete mocks (mock.restore() doesn't work for mock.module) |

## Common Pitfalls

### Pitfall 1: Runner spawns `bun test tests/lib/` but this re-enters subdirectories

**What goes wrong:** `bun test tests/lib/` recurses into `tests/lib/integrations/` too. The runner must pass explicit file lists, not directory paths, to control exactly which files run together.

**How to avoid:** Use explicit file arrays: `Bun.spawn(["bun", "test", ...unitFiles])` where `unitFiles` is the computed array. Don't pass directory strings.

### Pitfall 2: Runner child processes don't inherit bunfig.toml preload

**What goes wrong:** If the runner spawns `bun test` with `cwd` set to the project root and bunfig.toml is there, preload is inherited automatically. But if `cwd` is different, the OpenTUI preload breaks SolidJS dashboard tests.

**How to avoid:** Always set `cwd: ROOT` (project root) in `Bun.spawn` options for child test processes.

### Pitfall 3: Fixing one mock factory import in a test file but missing others

**What goes wrong:** A developer fixes `makeConfigMock()` usage in `jira.test.ts` but doesn't update the `makeIssueUtilsMock()` call — the file still passes a partial issue-utils mock.

**How to avoid:** When converting a test file to use factory helpers, audit ALL `mock.module()` calls in that file, not just the one causing the immediate failure.

### Pitfall 4: `detect-workspace-cwd.test.ts` still uses old DI pattern after D-06 removal

**What goes wrong:** After `_cwdDetect` and `_resolveWorkspaceDeps` are removed from production code, `detect-workspace-cwd.test.ts` fails to compile because it references these removed objects.

**How to avoid:** Refactor this test file as part of the same task that removes the DI objects. The test file needs the most extensive rewrite — plan it explicitly.

### Pitfall 5: `workspace-ops.test.ts` mock contamination loop

**What goes wrong:** `workspace-ops.test.ts` currently does `mock.module("@/lib/lifecycle", () => realLifecycle)` to reset the lifecycle mock. After cache-busting is removed, if complete mock factories aren't used by the time `workspace-ops.test.ts` runs, lifecycle will be the partial mock from another file.

**How to avoid:** Ensure `makeLifecycleMock()` is used in ALL files that mock lifecycle before removing cache-busting from `workspace-ops.test.ts`.

### Pitfall 6: Partial `@/lib/paths` mock in `useIsolatedConfig()` causes link failures

**What goes wrong:** `useIsolatedConfig()` in helpers.ts also calls `mock.module("@/lib/paths", ...)` with a complete map but WITHOUT a `makePathsMock()` factory — this is acceptable since paths.ts only exports constants and 2 functions, and `useIsolatedConfig()` already provides all of them. Do not break `useIsolatedConfig()` while adding `makePathsMock()`.

**How to avoid:** Add `makePathsMock()` as a NEW export alongside `useIsolatedConfig()`. Don't refactor `useIsolatedConfig()` — tests that need real config isolation still need it.

### Pitfall 7: `mock.restore()` does not work for `mock.module()`

**Why it happens:** Bun's official docs state: *"Doing so does not reset the value of modules overridden with `mock.module()`."* Issue #7823 confirms this.

**How to avoid:** Never add `afterAll(() => mock.restore())` expecting it to clear module mocks. Complete factories are the only solution for shared-process unit tests.

## Open Questions

1. **Does `lifecycle.test.ts` require two cache-busted imports by design?**
   - What we know: It imports `@/lib/lifecycle?lifecycle-real` for real-shell tests and `?lifecycle-exec-test` for `_exec` injection tests. The two query strings ensure the real-shell instance isn't contaminated by the `_exec` mock.
   - What's unclear: After `workspace-ops.test.ts` uses `makeLifecycleMock()`, the lifecycle mock.module call will be complete — but `lifecycle.test.ts` still needs a fresh instance to test real shell execution.
   - Recommendation: Keep `lifecycle.test.ts`'s cache-busting (it's genuinely testing real shell commands, not just mocking). This is an exception to D-07. Or, `lifecycle.test.ts` can use `_exec` injection for all tests and use plain import — the `runHooks`/`runHooksCaptured` functions use `_exec.spawn` internally, so `_exec.run = mock(...)` pattern works without cache-busting.

2. **How should `detect-workspace-cwd.test.ts` test `detectWorkspaceFromCwd` after `_cwdDetect` is removed?**
   - What we know: Currently it injects `_cwdDetect.listWorkspaces = () => listWorkspaces()` to connect the detection logic to the isolated config dir. After removal, `detectWorkspaceFromCwd` will call `listWorkspaces()` from `config.ts` directly.
   - What's unclear: Whether the test should use `useIsolatedConfig()` + real module import (no mock.module on workspace-ops), or use mock.module with a factory that returns a custom `listWorkspaces`.
   - Recommendation: Use `useIsolatedConfig()` + plain `import { detectWorkspaceFromCwd } from "@/lib/workspace-ops"` (no mock.module on workspace-ops itself, no cache-busting). The `useIsolatedConfig()` call sets up paths.ts and config.ts will use the isolated dir. The `mock.module("@/lib/workspace-ops", ...)` from OTHER unit test files won't interfere because those files run before this one alphabetically... but wait — that's the contamination problem. Better approach: use `mock.module("@/lib/config", () => makeConfigMock({ listWorkspaces: () => [...myWorkspaces] }))` and import workspace-ops normally.

3. **Does `workspace-ops.test.ts` cache-bust lifecycle to avoid contamination from `integration-commands.test.ts`?**
   - What we know: `workspace-ops.test.ts` line 20 does `mock.module("@/lib/lifecycle", () => realLifecycle)` to reset the lifecycle mock. This is because `integration-commands.test.ts` mocks lifecycle with a partial mock first.
   - Recommendation: After Wave 1 fixes `integration-commands.test.ts` to use `makeLifecycleMock()`, this reset in `workspace-ops.test.ts` becomes unnecessary. Remove it in Wave 2.
