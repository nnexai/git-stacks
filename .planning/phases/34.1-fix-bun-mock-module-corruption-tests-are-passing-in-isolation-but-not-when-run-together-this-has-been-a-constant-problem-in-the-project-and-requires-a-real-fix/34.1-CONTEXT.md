# Phase 34.1: Proper Test Isolation Framework - Context

**Gathered:** 2026-03-25
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the current hack-on-hack test isolation approach (mock.module partial exports, cache-busting query-string imports, internal DI objects) with a robust framework. Unit tests run together in one process. Integration tests each run in their own process. Production code is cleaned of test-only indirection. The partial-export footgun is eliminated via complete mock factory helpers.

This phase does NOT add new test coverage or change what is tested тАФ it fixes HOW tests are isolated from each other.

</domain>

<decisions>
## Implementation Decisions

### Test Runner Architecture
- **D-01:** Build a custom bun test runner script (`scripts/test-runner.ts`) that orchestrates two modes: (a) run all unit tests together in one `bun test` process, (b) run each integration test file in its own `bun test` process
- **D-02:** File convention determines test category:
  - `tests/lib/*.test.ts` тЖТ unit tests (run together)
  - `tests/lib/integrations/*.test.ts` тЖТ unit tests (run together тАФ these mock external CLIs via `_exec`)
  - `tests/commands/*.test.ts` тЖТ integration tests (run isolated per-file)
  - `tests/tui/*.test.ts` тЖТ integration tests (run isolated per-file)
  - `tests/tui/dashboard/integ-*.test.tsx` тЖТ integration tests (run isolated per-file)
  - `tests/tui/dashboard/*.test.tsx` (non-integ) тЖТ unit tests (run together)
- **D-03:** Package.json scripts: `test` runs the custom runner (full suite), `test:unit` runs unit tests only, `test:integ` runs integration tests only
- **D-04:** The runner aggregates results and exits non-zero if any test fails. Output should clearly show which mode each file ran in

### DI Object Policy
- **D-05:** KEEP all `_exec` objects that mock external program calls тАФ these are legitimate testing seams at the boundary between our logic and external CLIs:
  - `src/lib/niri.ts` `_exec` тЖТ mocks `niri msg`
  - `src/lib/tmux.ts` `_exec` тЖТ mocks `tmux`
  - `src/lib/cmux.ts` `_exec` тЖТ mocks `cmux`
  - `src/lib/lifecycle.ts` `_exec` тЖТ mocks `Bun.spawn`
  - `src/lib/integrations/gitea.ts` `_exec` тЖТ mocks `tea`
  - `src/lib/integrations/github.ts` `_exec` тЖТ mocks `gh`
  - `src/lib/integrations/gitlab.ts` `_exec` тЖТ mocks `glab`
  - `src/lib/integrations/jira.ts` `_exec` тЖТ mocks `curl`
  - `src/lib/integrations/forge-utils.ts` `_detect` тЖТ mocks `which`, `git remote`, `tea pulls ls`
- **D-06:** REMOVE all internal isolation hack DI objects тАФ these only exist to avoid mock.module cross-file contamination, which process isolation now handles:
  - `src/lib/workspace-ops.ts` `_cwdDetect` тАФ remove, tests mock at module level instead
  - `src/lib/integrations/issue-utils.ts` `_resolveWorkspaceDeps` тАФ remove, tests mock at module level instead

### Cache-Busting Removal
- **D-07:** Remove ALL 27 cache-busting query-string imports across test files. Process isolation for integ tests eliminates the need. Unit tests use mock.module (safe in shared process since factory helpers ensure complete exports)
- **D-08:** Replace `await import("@/lib/X?query")` with either static `import` or plain `await import("@/lib/X")` depending on whether mock.module is used in the file

### Mock.Module Strategy
- **D-09:** Keep mock.module as the primary mocking mechanism тАФ it works correctly when: (a) tests share a process without cross-file contamination concerns (unit tests), or (b) tests run in isolated processes (integ tests)
- **D-10:** Add complete mock factory helpers to `tests/helpers.ts` for every module that gets mocked in 3+ test files. Each factory returns ALL exports of the target module with sensible stubs. Tests spread overrides. This makes partial mocks structurally impossible
- **D-11:** Required factory helpers (minimum): `makeConfigMock(overrides?)`, `makeWorkspaceOpsMock(overrides?)`, `makeForgeUtilsMock(overrides?)`, `makeIssueUtilsMock(overrides?)`, `makePathsMock(overrides?)`, `makeLifecycleMock(overrides?)`
- **D-12:** Schema stubs in mock factories use simple structural objects (`{ parse: mock((x) => x), safeParse: mock((x) => ({ success: true, data: x })) }`), not real Zod objects. Tests that need real schema validation import directly, not through mock.module

### Claude's Discretion

- Which test files need to move directories to match the unit/integ convention
- Whether `useIsolatedConfig()` in helpers.ts needs refactoring or just the mock factory additions
- Order of operations during migration (runner first vs cleanup first)

</decisions>

## Existing Code Insights

### Reusable Assets
- `tests/helpers.ts` `useIsolatedConfig()` тАФ already encapsulates paths mock + temp dir creation; extend pattern for other mock factories

### Established Patterns
- `_exec` object pattern is consistent across 9 modules тАФ well-understood, tests mutate `.run` property after cache-busted import. With cache-busting removal, tests will mock the whole module or import normally and mutate `_exec.run`
- Dashboard integ tests follow a consistent pattern: mock config + workspace-ops + git + lifecycle at top level, render SolidJS component, assert behavior
- Unit tests for forge integrations follow: mock `_exec.run`, import module, call function, assert output

- The test runner script needs to detect file categories and invoke `bun test` appropriately
- CI (if any) needs to use the new `bun run test` entry point
