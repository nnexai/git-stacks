## Summary

Phase 50.1 refactors `src/lib/completion-generator.ts` to eliminate manual `DYNAMIC_COMPLETIONS` maintenance. The key insight is that Commander.js argument names are metadata that can be matched against a naming convention to auto-derive completion types. Commands like `open <workspace>` or `template edit <template>` carry enough information in the argument name to infer the right completion without a separate map entry.

The change has two interdependent parts: (1) command argument renames in the command definition files so that argument names match the conventions (`<name>` â†’ `<workspace>` in workspace commands, `<name>` â†’ `<repo>` in repo subcommands, etc.), and (2) updates to `buildNode()` and all three shell generators to read `registeredArguments` by position rather than a single `dynamic` field.

The `integration` type is new and requires a static ID list emitted at generation time. Since integration IDs are registered at startup (not user-configurable), they can be hardcoded into the generated shell script as a space-separated word list.

**Primary recommendation:** Rename command argument names first (as a prerequisite), then update `buildNode()` and generators, then shrink `DYNAMIC_COMPLETIONS` to only the remaining edge cases.

## Architecture Patterns

### Commander.js Argument API (HIGH confidence â€” verified against installed 14.0.3)

Commander.js `Argument` objects are accessible via `cmd.registeredArguments: Argument[]`.

Each `Argument` exposes:
- `a.name(): string` â€” the argument name without `<>` or `[]` brackets (e.g. `"workspace"`, `"repo"`)
- `a.required: boolean` â€” `true` for `<arg>`, `false` for `[arg]`
- `a.argChoices: string[] | undefined` â€” populated when `.choices(['a','b','c'])` is called; `undefined` otherwise

```typescript
// Commander 14.0.3 â€” verified
const arg = cmd.registeredArguments[0]
arg.name()      // "workspace" (from "<workspace>" or "[workspace]")
arg.required    // true for <required>, false for [optional]
arg.argChoices  // string[] | undefined â€” from .choices()
```

### Convention Map (the lookup table for D-01)

```typescript
// Source: synthesis of D-01 decisions + arg audit
const NAME_TO_COMPLETION_TYPE: Record<string, DynamicCompletion> = {
  workspace: "workspace",
  template:  "template",
  repo:      "repo",
  shell:     "shells",
  integration: "integration",
}
```

Convention matching should be **exact match** (not contains/regex). The argument name `name`, `old`, `source`, `path`, `dir`, `text`, `new-name` do not match any key â€” they correctly produce no inference, leaving the path-based override layer to handle them. This is the cleanest approach.

### ArgCompletion Type Design

```typescript
// New type â€” replaces dynamic: DynamicCompletion | null + firstArgRequired: boolean
type ArgCompletion = {
  name: string                // Argument name (from Argument.name())
  type: DynamicCompletion     // Inferred or overridden type
  required: boolean           // From Argument.required
  choices?: string[]          // Only present when source is D-04 (.argChoices)
}

// Updated CommandNode interface
interface CommandNode {
  path: string
  name: string
  description: string
  options: OptionInfo[]
  subcommands: CommandNode[]
  argCompletions: ArgCompletion[]   // replaces: dynamic + firstArgRequired
}

// Convenience accessors used by generators:
// argCompletions[0]?.type       â†’ replaces dynamic
// argCompletions[0]?.required   â†’ replaces firstArgRequired
```

Storing only completable args (not all args) is safe because: position index equals the array index when null positions are excluded â€” provided they are filtered correctly. However, to support sparse multi-arg commands where only arg[1] has a completion (e.g., `cmd <freetext> [workspace]`), include the position index or store all args. For the current command set this is not an issue, but the cleanest approach is to **only store args that have a non-null type** and track no index (position == array index is always true for current commands).

### buildNode() Transformation

```typescript
// OLD
const firstArg = cmd.registeredArguments[0]
return {
  ...
  dynamic: DYNAMIC_COMPLETIONS[path] ?? null,
  firstArgRequired: firstArg?.required === true,
}

// NEW â€” conceptual skeleton
function buildNode(cmd: Command, parentPath: string): CommandNode {
  const path = ...
  const argCompletions: ArgCompletion[] = []
  
  for (const arg of cmd.registeredArguments) {
    // Priority chain (D-02):
    // 1. Path-based override from DYNAMIC_COMPLETIONS (first arg only â€” see note below)
    // 2. Convention inference from arg name
    // 3. Commander .argChoices extraction
    // 4. null â†’ skip (don't add to argCompletions)
    
    const overrideType = DYNAMIC_COMPLETIONS[path]   // only applies to arg[0]
    const inferredType = NAME_TO_COMPLETION_TYPE[arg.name()]
    const choicesType  = arg.argChoices?.length ? "choices" : undefined
    
    const type = overrideType ?? inferredType ?? choicesType ?? null
    if (!type) continue
    
    argCompletions.push({
      name: arg.name(),
      type,
      required: arg.required,
      ...(type === "choices" ? { choices: arg.argChoices } : {}),
    })
  }
  
  return { path, name, description, options, subcommands, argCompletions }
}
```

**Note on DYNAMIC_COMPLETIONS scope:** The override map key is a command path (`"completion"`, `"clone"`), not a `path:argIndex` key. It overrides the first positional argument type only. Arguments beyond index 0 are always inference-only.

### Integration ID Completion (D-03)

Integration IDs are static (registered at startup). The generator should import the `integrations` array and emit IDs as a hardcoded word list at generation time â€” not read from disk at completion time.

```typescript
// In completion-generator.ts
import { integrations } from "./integrations/index"

// Current IDs: vscode intellij cmux tmux niri aerospace github gitlab gitea jira
const INTEGRATION_IDS = integrations.map(i => i.id).join(" ")

// In bashDynamicLookup for type === "integration":
// COMPREPLY=($(compgen -W "vscode intellij cmux tmux niri aerospace github gitlab gitea jira" -- "$cur"))

// In zsh:
// _values 'integration' vscode intellij cmux ...

// In fish:
// complete -c git-stacks ... -a 'vscode intellij ...'
```

This avoids runtime I/O and keeps completions fast. New integrations added to the registry automatically appear in the next `git-stacks completion` invocation.

### Shell Generator Updates

#### Pattern for First-Arg (most commands)
Generators currently check `node.dynamic`. After migration, they check `node.argCompletions[0]?.type`. This is a mechanical substitution â€” the generated shell code structure stays the same.

#### Pattern for Multi-Arg Position Dispatch (D-06)
Commands with 2 completable args (`run`, `cd`, `repo rename`, `template clone`, `rename`):

**Bash** â€” add a second `elif` branch checking `COMP_CWORD` against arg position 2:
```bash
# For command 'run <workspace> [repo]'
if [[ ${COMP_CWORD} -eq 2 ]]; then
  # workspace completion
elif [[ ${COMP_CWORD} -eq 3 ]]; then
  # repo completion
fi
```

**Zsh** â€” use `_arguments` with two positional specs:
```zsh
_arguments \
  ':workspace:_git_stacks_workspaces' \
  '::repo:_git_stacks_repos'
```

**Fish** â€” position detection via `__fish_seen_subcommand_from` plus argument count:
```fish
# Position 1 (2nd word after subcommand): workspace
complete -c git-stacks -f -n '__fish_seen_subcommand_from run; and test (count (commandline -opc)) -eq 2' \
  -a "(__git_stacks_workspaces)"
# Position 2 (3rd word): repo  
complete -c git-stacks -f -n '__fish_seen_subcommand_from run; and test (count (commandline -opc)) -eq 3' \
  -a "(__git_stacks_repos)"
```

For the planner's awareness: multi-arg dispatch only affects a small set of commands. The generator changes for this are additive â€” the common single-arg path uses the same code structure as today.

### DYNAMIC_COMPLETIONS After Refactor

After all arg renames, remaining entries in DYNAMIC_COMPLETIONS:

| Entry | Why it stays |
|-------|-------------|
| `clone: "workspace"` | Only if `clone [source]` keeps `[source]` arg for semantic clarity (cloning an existing workspace, not creating new). If arg renamed to `[workspace]`, this entry can be deleted too. |

The map shrinks from ~40 entries to 0-1 entries. The planner should decide whether `clone [source]` gets its arg renamed or stays in the override map.

### Anti-Patterns to Avoid

- **Regex/contains matching for arg names:** D-01 specifies exact convention mapping. `repo` matches `repo`, not `repository` or `repo-name`. Keep the lookup as an exact key check.
- **Runtime filesystem reads for integration IDs:** Integration IDs are static at generation time. Don't emit shell code that reads integration IDs from disk â€” hardcode them at generation time.
- **Leaving both `dynamic` and `argCompletions` on CommandNode during migration:** The interface change should be atomic â€” no compatibility shim. All generators must be updated in the same PR.
- **Adding a `DynamicCompletion | "choices"` union:** The `choices` case is different enough to warrant a separate field on ArgCompletion (`choices?: string[]`) rather than polluting the type union. Generators check `if (ac.choices)` separately.

## Common Pitfalls

### Pitfall 1: `<name>` Ambiguity Without Arg Renames
**What goes wrong:** If command files are not updated and arg names stay as `<name>`, the convention inference produces no match, `argCompletions` is empty for workspace/repo/template commands, and completion breaks entirely.
**Why it happens:** `name` is not in the NAME_TO_COMPLETION_TYPE map â€” it was only handled by the old DYNAMIC_COMPLETIONS map.
**How to avoid:** Arg renames in command files are a prerequisite to convention inference working. The test program in the test file must also use the new arg names.
**Warning signs:** Completion tests pass but real `git-stacks open <TAB>` produces no completions.

### Pitfall 2: Forgetting to Update `buildTestProgram()` in Tests
**What goes wrong:** Tests pass because `buildTestProgram()` uses the same old arg names as before, so both the test and the production code are internally consistent. But real commands have been renamed and the production path is different.
**Why it happens:** The test program is a synthetic recreation, not the real program tree.
**How to avoid:** Update `buildTestProgram()` to use the new arg names alongside the command file changes. Add a test that verifies a new command (with no DYNAMIC_COMPLETIONS entry) gets correct completion from convention alone.
**Warning signs:** All tests green but new integration subcommands added in future phases don't auto-complete.

### Pitfall 3: DYNAMIC_COMPLETIONS Override Applies to All Args
**What goes wrong:** `DYNAMIC_COMPLETIONS["rename"] = "workspace"` is treated as overriding ALL args of `rename`, so arg[1] (new name) is also typed as workspace.
**Why it happens:** The path key in the override map doesn't specify a position.
**How to avoid:** The override from `DYNAMIC_COMPLETIONS` applies only to `registeredArguments[0]`. Args at index >= 1 are always inference-only. This must be explicit in `buildNode()` logic.

### Pitfall 4: `argChoices` Is Undefined, Not Null
**What goes wrong:** Check `if (arg.argChoices)` passes on empty array `[]` (truthy in JS but no choices to offer) OR fails on undefined (falsy) correctly.
**Why it happens:** Commander 14 sets `argChoices: undefined` (not `null`) when `.choices()` is never called.
**How to avoid:** Check `if (arg.argChoices && arg.argChoices.length > 0)` before treating as choices type.

### Pitfall 5: Fish Position Detection for Multi-Arg Commands
**What goes wrong:** Fish position detection using `count (commandline -opc)` counts differently than expected for deeply nested subcommands.
**Why it happens:** Fish's `commandline -opc` returns the current token list excluding the command itself.
**How to avoid:** Multi-arg position dispatch in fish is only needed for top-level commands (not nested subcommands). For nested subcommands under `integration.*`, the `emitFishSubcommands` recursive logic handles position. Apply positional dispatch only in the top-level command fish generation path.

## Code Examples

### buildNode() â€” Core Logic

```typescript
// Convention inference map
const NAME_TO_COMPLETION_TYPE: Record<string, DynamicCompletion> = {
  workspace:   "workspace",
  template:    "template",
  repo:        "repo",
  shell:       "shells",
  integration: "integration",
}

function buildNode(cmd: Command, parentPath: string): CommandNode {
  const name = cmd.name()
  const path = parentPath ? `${parentPath}.${name}` : name
  const options = cmd.options
    .filter(opt => opt.long !== undefined && opt.long !== "--help" && opt.long !== "--version")
    .map(opt => ({ long: opt.long!, description: opt.description }))
  const subcommands = cmd.commands.map(sub => buildNode(sub, path))

  // D-02 priority chain for first arg (override layer)
  const firstArgOverride = DYNAMIC_COMPLETIONS[path]

  const argCompletions: ArgCompletion[] = cmd.registeredArguments.flatMap((arg, index) => {
    // Priority 1: path-based override (first arg only)
    if (index === 0 && firstArgOverride) {
      return [{ name: arg.name(), type: firstArgOverride, required: arg.required }]
    }
    // Priority 2: convention inference
    const inferred = NAME_TO_COMPLETION_TYPE[arg.name()]
    if (inferred) {
      return [{ name: arg.name(), type: inferred, required: arg.required }]
    }
    // Priority 3: .argChoices extraction
    if (arg.argChoices && arg.argChoices.length > 0) {
      return [{ name: arg.name(), type: "choices" as DynamicCompletion, required: arg.required, choices: arg.argChoices }]
    }
    // Priority 4: null â€” no completion for this position
    return []
  })

  return { path, name, description: cmd.description(), options, subcommands, argCompletions }
}
```

## Open Questions

1. **Should `clone [source]` be renamed to `clone [workspace]`?**
   - What we know: `[source]` is semantically accurate (cloning an existing workspace as the source). `[workspace]` is consistent with the convention.
   - What's unclear: Whether `clone [workspace]` reads as confusing (you're creating a NEW workspace FROM the source, not opening the source itself).
   - Recommendation: Rename to `[workspace]` â€” the completion inference benefit outweighs the minor semantic ambiguity. If kept as `[source]`, add one entry to DYNAMIC_COMPLETIONS.

2. **Should `DynamicCompletion` type be renamed?**
   - What we know: CONTEXT.md leaves this to Claude's Discretion.
   - Recommendation: Keep `DynamicCompletion` as the type name. It's used only in `completion-generator.ts` and has no external callers. Renaming adds churn with no benefit.

3. **How to handle the `new [name]` workspace creation command?**
   - What we know: `new [name]` takes a new workspace name as free-form input â€” no completion is appropriate (you're naming something that doesn't exist yet).
   - What's unclear: Nothing â€” `name` does not match any convention key, so no completion is inferred. This is the correct behavior.
   - Recommendation: No action needed. Confirm this remains the intent.
