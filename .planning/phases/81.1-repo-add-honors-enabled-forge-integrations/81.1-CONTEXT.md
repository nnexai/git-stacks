# Phase 81.1: Repo add honors enabled forge integrations - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Update `git-stacks repo add` so forge detection and forge-selection prompts only consider forge integrations that are globally enabled in git-stacks config.

This phase is about repo-registry behavior only. It should make `repo add` align with the existing global integration enablement model, avoid prompting for disabled forges, register git repos without a forge when no enabled forge matches, and keep ambiguity handling limited to enabled matches. It does not expand into workspace/template override semantics, broader integration-runner changes, or the larger Phase 82 E2E coverage work.

</domain>

<decisions>
## Implementation Decisions

### Forge eligibility source
- **D-01:** `repo add` should use global integration config only as the source of truth for which forge detectors are allowed to run.
- **D-02:** A forge that is not globally enabled must be hidden from both auto-detection and prompt options.

### No-match behavior
- **D-03:** If no globally enabled forge matches a git repo, `repo add` should register the repo with `forge` unset and should not prompt.

### Ambiguity handling
- **D-04:** If multiple globally enabled forge detectors match the same repo, `repo add` should prompt only among those enabled matching forges plus `None`.

### the agent's Discretion
- Exact helper extraction and whether `repo add` filters detectors directly or builds on top of a shared allowlist helper, as long as it reuses the existing global enablement model.
- Exact prompt wording and logging text, as long as disabled forges stay invisible and the command remains consistent with existing CLI output style.
- Exact test split between repo command E2E coverage and lower-level forge-detection unit coverage updates.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `resolveEnabledGlobally()` in `src/lib/integrations/types.ts` already models the exact global-only enablement lookup this phase needs.
- Forge integrations in `src/lib/integrations/github.ts`, `gitlab.ts`, and `gitea.ts` already declare `enabledByDefault: false`, so the allowlist semantics are already encoded in plugin metadata.
- `src/commands/integration.ts` already exposes enabled/configured state to users, giving this phase an existing UX model for what “enabled” means.

### Established Patterns
- Non-workspace commands use global config rather than workspace overrides when no workspace context exists.
- Repo command handlers follow a simple “detect, maybe prompt, write registry, print one summary line” pattern.
- Forge detection is currently separated from registry persistence, making it feasible to insert enablement filtering without redesigning registry schema or integration plugins.

### Integration Points
- `src/commands/repo.ts` is the primary behavior change site for filtering detector execution and prompt options.
- `src/lib/integrations/forge-utils.ts` is the likely seam for detector filtering or allowlist-aware aggregation.
- `tests/commands/` will consume this behavior in later Phase 82 subprocess tests, so this phase should leave `repo add` deterministic enough for those tests.
- `tests/lib/integrations/forge-utils.test.ts` and any new repo command tests are the main verification surfaces for the behavior change.

</code_context>

<specifics>
## Specific Ideas

- Treat global forge enablement as an allowlist for `repo add`, not just as an informational display in other commands.
- Disabled forges should be completely invisible to `repo add`; the command should behave as if they do not exist.
- The fix is meant to remove accidental prompt branches from normal repo registration when the user has not enabled any forge integrations.

</specifics>
