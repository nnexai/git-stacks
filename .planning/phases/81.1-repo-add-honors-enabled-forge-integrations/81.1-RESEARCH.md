## Summary

Phase 81.1 is a focused behavioral fix: `repo add` currently ignores global integration enablement entirely when running forge detection and building prompt options. Because all three forge integrations (`github`, `gitlab`, `gitea`) have `enabledByDefault: false`, a user who hasn't explicitly enabled any forge will still be prompted to pick one on every `repo add` ÔÇö a noisy and incorrect UX.

The fix is small and self-contained. `resolveEnabledGlobally(id, enabledByDefault, config)` already exists in `src/lib/integrations/types.ts` and encodes exactly the global-only enablement lookup that `repo add` needs. The forge integration objects in `src/lib/integrations/index.ts` already expose `id` and `enabledByDefault`. The only missing piece is wiring these into `repo add`'s detection and prompt logic.

The cleanest seam is to add a `detectForgeForRepoEnabled(repoPath, config)` function in `forge-utils.ts` that runs only the detectors for globally-enabled forges, then update `repo add` to call this variant and remove the current fallback that presents all three forge options when detection returns zero results.

**Primary recommendation:** Add `detectForgeForRepoEnabled` to `forge-utils.ts` (filtering against `resolveEnabledGlobally`) and update the `repo add` action in `repo.ts` to read `GlobalConfig` and use the filtered variant. Tests extend `forge-utils.test.ts` for allowlist logic and add a thin `repo add` test in `tests/commands/`.

## Architecture Patterns

### Existing Pattern: Global-only enablement lookup

```typescript
// Source: src/lib/integrations/types.ts
export function resolveEnabledGlobally(
  id: string,
  enabledByDefault: boolean,
  config: GlobalConfig
): boolean {
  const result = enabledSchema.safeParse(config.integrations[id])
  return result.success ? result.data.enabled : enabledByDefault
}
```

### Anti-Patterns to Avoid
- **Reimplementing enablement logic in `repo.ts` directly:** Don't copy the `enabledSchema.safeParse` pattern into the command. Use `resolveEnabledGlobally` via import.
- **Importing from `src/lib/integrations/index.ts` inside `forge-utils.ts`:** The `integrations` array includes all plugins; creating a dependency in `forge-utils.ts` on `index.ts` risks a circular import (index.ts imports the forge plugins which would then depend on forge-utils which imports index.ts). Instead, hardcode the three forge ID/default pairs as a module-level constant in `forge-utils.ts`.
- **Keeping the "no detection Ôćĺ show all three" fallback:** The current code in `repo.ts` shows all three forge options when `forgeSuggestions.length === 0`. Under the new model, no-match means no prompt ÔÇö this fallback branch must be removed entirely.

### New function in `forge-utils.ts`

```typescript
// ADD to src/lib/integrations/forge-utils.ts
import type { GlobalConfig } from "../config"
import { resolveEnabledGlobally } from "./types"

// Forge ID Ôćĺ default pairs for the three forge integrations
// Hardcoded to avoid circular import through index.ts
const FORGE_DEFAULTS: Array<{ id: NonNullable<ForgeType>; enabledByDefault: boolean }> = [
  { id: "github",  enabledByDefault: false },
  { id: "gitlab",  enabledByDefault: false },
  { id: "gitea",   enabledByDefault: false },
]

/**
 * Run forge detection only for globally enabled forges.
 * Returns matched forge IDs ÔÇö only for forges the user has explicitly enabled.
 */
export async function detectForgeForRepoEnabled(
  repoPath: string,
  config: GlobalConfig
): Promise<NonNullable<ForgeType>[]> {
  const results: NonNullable<ForgeType>[] = []
  for (const { id, enabledByDefault } of FORGE_DEFAULTS) {
    if (!resolveEnabledGlobally(id, enabledByDefault, config)) continue
    if (id === "github" && await detectGitHubForge(repoPath)) results.push("github")
    if (id === "gitlab" && await detectGitLabForge(repoPath)) results.push("gitlab")
    if (id === "gitea"  && await detectGiteaForge(repoPath))  results.push("gitea")
  }
  return results
}
```

**Note on `FORGE_DEFAULTS`:** Hardcoding the three forge IDs and their defaults avoids importing from `index.ts`. This mirrors the `enabledByDefault: false` already declared in each plugin file. [VERIFIED: all three have `enabledByDefault: false`]

## Common Pitfalls

### Pitfall 1: Circular import via `index.ts`
**What goes wrong:** If `forge-utils.ts` imports from `src/lib/integrations/index.ts` to access the `integrations` array for `enabledByDefault` values, it creates a circular dependency. `index.ts` imports `githubIntegration` from `github.ts`, which imports from `forge-utils.ts`.
**How to avoid:** Hardcode the three forge ID/default pairs directly in `forge-utils.ts` as a module-level constant.
**Warning signs:** TypeScript circular import error or `undefined` values at runtime.

### Pitfall 2: Forgetting to remove the all-three-forges fallback
**What goes wrong:** The current `repo.ts` has a branch: when `forgeSuggestions.length === 0`, it shows `[None, github, gitlab, gitea]`. Under D-03, zero enabled matches Ôćĺ no prompt, forge unset. If this branch survives the refactor, D-03 is violated.
**How to avoid:** The new `else` (zero-match Ôćĺ `forge = undefined`) replaces the old ternary entirely. Verify with a test where `detectForgeForRepoEnabled` returns `[]` and confirm `p.select` is never called.
**Warning signs:** Test for D-03 sees prompt mock called.

### Pitfall 3: Missing `readGlobalConfig` import in `repo.ts`
**What goes wrong:** `repo.ts` currently imports `readRegistry` and `writeRegistry` but not `readGlobalConfig`. The new code needs it.
**How to avoid:** Add `readGlobalConfig` to the existing import from `"../lib/config"`.

### Pitfall 4: Test mock doesn't set `integrations` in config
**What goes wrong:** `makeConfigMock` returns `{ integrations: {} }` ÔÇö zero forges enabled (all default to `false`). Tests that want an enabled forge must override to `{ integrations: { github: { enabled: true } } }`.
**How to avoid:** Set `integrations` explicitly in per-test config mock values. Follow the `_detect` reset pattern in `forge-utils.test.ts` (per-`beforeEach` resets).

### Pitfall 5: `forge-utils.test.ts` mock.module override
**What goes wrong:** `forge-utils.test.ts` already uses `mock.module("@/lib/integrations/forge-utils", ...)` to override the real implementations. New `detectForgeForRepoEnabled` tests must either be added inside that mock (inlining the real implementation) or placed in a separate test file that imports the real module.
**How to avoid:** The simplest approach is to add a separate test file (`tests/commands/repo-add.test.ts`) that mocks at the `forge-utils` level and tests `repo add` behavior end-to-end rather than extending the existing mock-module override. Low-level `detectForgeForRepoEnabled` unit tests can be added directly to `forge-utils.test.ts` within the existing mock.module block.
