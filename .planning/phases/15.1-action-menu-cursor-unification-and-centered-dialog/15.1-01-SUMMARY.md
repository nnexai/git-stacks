## Decisions Made

- `widths` record typed as `Record<Size, \`${number}%\`>` (not `Record<Size, string>`) to satisfy TypeScript's strict template literal type for OpenTUI `width` prop

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] TypeScript width type mismatch in CenteredDialog**
- **Found during:** Task 1 (Create CenteredDialog wrapper component)
- **Issue:** `widths` record typed as `Record<Size, string>` but OpenTUI box `width` prop expects `number | "auto" | \`${number}%\`` — TypeScript rejected `string`
- **Fix:** Changed record type to `Record<Size, \`${number}%\`>` — values already matched the template literal pattern
- **Files modified:** src/tui/dashboard/CenteredDialog.tsx
- **Verification:** `bun run typecheck` passes
- **Committed in:** 11c8a7a (Task 1 commit)

**2. [Rule 3 - Blocking] Integration tests failing: CenteredDialog inside detail pane clipped content**
- **Found during:** Task 2 (Add cursor navigation — GREEN phase)
- **Issue:** When ActionMenu was wrapped in CenteredDialog and rendered inside the detail pane (`<box flexGrow={2} minHeight={10}>`), the `height="100%"` of the dim layer resolved to the pane's height (not full terminal). The vertical centering of the inner dialog box caused overflow clipping — `[o] Open` was clipped at top; only last few items visible
- **Fix:** Promoted all three action menu Show blocks from inside the detail pane to App root level (parallel to HelpOverlay/MessageOverlay). Updated split pane Show condition to `!helpOpen() && !messagesOpen() && view().view !== "action-menu" && view().view !== "repo-action-menu"` so action menus get full terminal height allocation
- **Files modified:** src/tui/dashboard/App.tsx
- **Verification:** `bun test tests/tui/dashboard/` — 79 pass, 0 fail
- **Committed in:** 73719c5 (Task 2 commit)

## Issues Encountered

- OpenTUI flex layout distributes zero height to children added after full `flexGrow` containers — root-level overlay dialogs must either appear before the split pane or the split pane condition must exclude their states. Followed the HelpOverlay/MessageOverlay precedent (hide split pane when dialog is active).
