# Phase 107.2: Beautify Workspace and Signal Presence - Research

<user_constraints>
## User Constraints (from CONTEXT.md)

### Locked Decisions

- Use the source-derived rules in `107.2-UI-SPEC.md` and both supplied Supacode screenshots as visual references.
- Adapt semantics to GTK symbolic icons and libadwaita theme tokens; do not copy macOS-only assets or fixed colors.
- Show default/main branch, pinned state, Git additions/removals, agent presence/awaiting state, running work, unread notifications, remote/missing state, and PR/check state when the service exposes it.
- Pinned and Active are global relevance sections; remaining rows keep label/repository organization.
- Agent activity and unread notifications are distinct signals and distinct markers.
- Completed activity is history, not unread attention.
- Selection remains the dominant row treatment; decorative status colors neutralize when necessary for contrast.
- Reduced motion, 200% text, high contrast, light/dark, inactive window, keyboard, mouse, and screen reader behavior are release gates.
- Phase 107.2 consumes Phase 107.1 signal projections and does not introduce a new attention transport or provider integration.

## Summary

Most of Phase 107.2 is ready to plan now. The GTK shell, workspace identity/pin/selection model, label/repository organization, exact pair and surface routing, symbolic-icon/CSS machinery, accessibility helpers, and native test runners already exist. The UI specification fixes the visual hierarchy, spacing, deterministic Active ordering, compression rules, motion fallback, inbox copy, and acceptance matrix. Those are stable presentation contracts and can be decomposed into executable plans without waiting for all Phase 107.1 implementation work. [VERIFIED: `107.2-CONTEXT.md`, `107.2-UI-SPEC.md`, `native/linux/app.zig`, `native/linux/workspace_view.zig`]

The dependency on 107.1 is narrow but real. Current in-progress code still carries additive `attention`/`signal` compatibility, native `Attention` names, and an incomplete signal decoder. Plans 107.1-03 and 107.1-04 own the atomic transport/native cutover and normalized provider/OSC identity; 107.2 must not bind its production projection tasks to today's transitional field names. Separately, the service snapshot already exposes branch, existence, dirty, ahead, and behind data, but the native aggregate model currently retains only workspace/repository names and labels; additions/removals, default-branch identity, remote/degraded detail, and PR/check state are not represented end-to-end. These are shared-model/service projection gaps, not reasons to postpone planning. [VERIFIED: live diff; `src/lib/service/contract.ts`; `src/lib/service/snapshot.ts`; `native/core/model.zig`; 107.1 plans 03-04]

**Primary recommendation:** Plan the phase in independent projection-first waves now, but give the signal adapter/inbox integration an explicit dependency on completed 107.1-03/04 and give final row rendering an explicit dependency on a service-to-native workspace metadata enrichment task.

## Planning Readiness and Dependency Boundary

### Stable enough to plan immediately

1. **Pure presentation contracts:** Define `WorkspaceRowProjection`, section classification, relevance tuple, provider-badge grouping, overflow, compression tiers, accessible descriptions, and inbox grouping as pure Zig functions with golden tests. These contracts use domain meanings, not transitional transport names. [VERIFIED: UI spec]
2. **GTK structure and styling:** Refactor the monolithic row/inbox construction in `native/linux/app.zig` around projection outputs; use existing `setAccessible`, `setAccessibleSelected`, GTK symbolic images, CSS classes, and libadwaita token names. [VERIFIED: native source]
3. **Identity-safe navigation:** Keep `model.PairKey`, workspace IDs, repository IDs, surface IDs, and existing actions as the selection/focus keys. Sorting and hoisting must never derive selection from visible row indexes. [VERIFIED: `workspace_view.zig`, `application.zig`]
4. **Responsive and accessible acceptance:** The UI spec already locks wide/medium/narrow compression, 200% text behavior, motion-off fallback, RTL logical placement, theme coverage, and blocking production UAT. [VERIFIED: UI spec]
5. **Pinned/origin behavior:** Existing persisted pins and grouping mode are stable. Plan Active as a derived global section and compact origin summaries without changing pin persistence. [VERIFIED: `native/core/model.zig`, `native/core/persistence.zig`, `app.zig`]

### Execution gates that must wait for 107.1

1. **Signal projection adapter:** Do not finalize against current `model.Attention`, `.attention_received`, `AttentionProvider`, or the additive decoder. Execute after 107.1-03 establishes the sole native Signal model/reducer/ABI and 107.1-04 establishes normalized source/session/surface identity. [VERIFIED: live diff; 107.1 plans 03-04]
2. **Dismiss/focus inbox wiring:** Execute against 107.1's final service-authoritative notification dismissal and exact signal focus actions. Do not add local dismissed IDs or presentation-owned lifecycle mutation. [VERIFIED: 107.1 research/resolved decisions]
3. **Provider badges and activity keys:** Badge instances require final provider session identity and activity/notification discrimination. The visual grouping algorithm can be built first against fixtures; the production adapter waits. [VERIFIED: 107.1 SIG-01/SIG-02 plan]
4. **Final cross-phase verification:** Production scenarios must run only after 107.1 replay/reconnect, capacity, provider/OSC ingress, and dismissal gates pass. [VERIFIED: 107.1 plan 07]

### Independent prerequisite discovered for 107.2

Add a bounded workspace-presentation projection to the existing service snapshot/shared native model before final row rendering. It should expose, per workspace/repository pair, the service-known branch, default-branch boolean/name, existence/remote/degraded state, additions/removals (not merely dirty), and optional PR/check summary. Optional unavailable PR/check state must remain absent rather than trigger direct provider calls from GTK. This enrichment belongs to 107.2 because it is presentation data and is not part of the 107.1 signal transport. [VERIFIED: current snapshot and native model inventory]

### Pattern 1: Pure semantic projection before widgets

Create bounded structs for row identity, leading icon semantic, primary/secondary text, Git counters, optional PR/check, sorted provider badges, activity/unread markers, accessible description, section membership, and compression visibility. Test these without GTK. Widget code binds the result and owns no ordering or lifecycle decisions. [VERIFIED: existing `attention_view.present/project` establishes this seam]

### Pattern 2: Stable identity, derived ordering

Classify each workspace once: Pinned first; otherwise Active if it has unread, awaiting, provider presence, or running work; otherwise ordinary. Sort Active by the exact locked priority tuple and then case-stable workspace name, with ID as a final deterministic tie-break. Store selection as `PairKey`, never list position. [VERIFIED: UI spec and existing state model]

### Pattern 3: Capability-shaped optional presentation

Decode optional PR/check data only when the service exposes it. Absence means omit the badge, not degraded service. Missing workspace/repository state is explicit and redundant. This preserves the scope fence against new provider integration. [VERIFIED: locked decisions]

### Pattern 4: Separate activity, presence, and notification

Provider badges represent active sessions; activity ping represents background running work; solid orange marker represents unread notification. Awaiting input modifies the relevant provider badge/ring and accessible copy. Completed/idle activity belongs to Recent activity and never increments unread. [VERIFIED: UI spec and 107.1 resolved semantics]

### Anti-Patterns to Avoid

- Rendering directly from the transitional 107.1 decoder or keeping `attention` compatibility aliases for UI convenience.
- Calling Git, GitHub, or another provider from GTK.
- Reusing unread counts as agent presence or activity.
- Sorting widgets after construction or using row indexes as identity.
- Encoding pin with a literal star in the primary title; section placement and accessible state are authoritative.
- Hard-coded RGB colors, macOS assets, custom font metrics, or animation timers that ignore GTK's animation setting.
- Hiding awaiting/unread state before lower-priority Git/PR details during compression.
- Mutating read/dismiss state merely because a row was rendered or activity completed.

## Don't Hand-Roll

| Problem | Don't build | Use instead | Why |
|---------|-------------|-------------|-----|
| Theme colors | Fixed light/dark palettes | libadwaita/GTK semantic tokens | Automatically tracks theme, contrast, selection, and accent. |
| Icons | Copied Supacode/SF assets | GTK symbolic icons and documented local fallbacks | Preserves licensing, recoloring, scale, and portability. |
| Motion preference | App-specific preference | GTK/libadwaita animation setting | Matches the normative platform control. |
| Git state | GTK subprocess polling | Service snapshot/shared model | Avoids duplicated authority, blocking UI work, and races. |
| Signal coalescing/dismissal | Widget-local caches | 107.1 reducer and service-authoritative actions | Replay/reconnect and multi-client convergence already belong there. |
| Responsive truth | Ad hoc child hiding in callbacks | Deterministic compression projection + width classes | Makes priority and tests repeatable. |

## Common Pitfalls

### Planning against transitional 107.1 names

The live diff accepts both `attention` and `signal` and maps notification to synthetic `waiting`. This contradicts the final 107.1 contract. Plans should name domain semantics and put the production adapter after 107.1-03/04, not freeze the temporary compatibility shape. [VERIFIED: live diff]

### Assuming service data already reaches GTK

The TypeScript snapshot contains branch/status facts, but `aggregateState()` currently copies only identity/name/labels/repositories into `model.Workspace`. A row renderer cannot display facts that were discarded at decoding. Add contract, capacity, fixture, decoder, model, and projection tests together. [VERIFIED: service/native source]

### Confusing dirty/ahead/behind with additions/removals

The current snapshot status is insufficient for `+N -N`; define exactly whether counts are changed lines or changed files and derive them service-side from one authoritative status operation. The UI spec's `DiffStatsContent` wording implies additions/removals, so the planner must make the contract explicit rather than label dirty state as line counts. [VERIFIED: current contract gap; UI spec]

### Duplicate rows after hoisting

Pinned and Active are global sections. Ordinary label/repository groups must exclude hoisted workspaces and show only compact origin summaries. Test multi-label workspaces, pinned+active overlap, grouping changes, and selection persistence. [VERIFIED: UI spec]

### Widget rebuild loses state or handlers

Current `refreshProjection()` clears and rebuilds the list, and `RowBinding.unbind()` exists specifically to clear recycled identity/CSS/drag state. Preserve exact identity and ensure no stale CSS classes, duplicate signal handlers, or focus loss across refresh. [VERIFIED: `app.zig`, `workspace_view.zig`, existing test]

### Visual acceptance substituted by contracts

Pure tests can prove projection and accessibility metadata but cannot prove 200% text, clipping, high contrast, inactive selection, animation, or real screen-reader behavior. Keep screenshot evidence and blocking production GTK UAT as separate final gates. [VERIFIED: UI spec]

## Open Questions

1. **What exactly do Git `+N -N` count?**
   - What we know: the UI reference calls them additions/removals and the current service exposes only dirty/ahead/behind.
   - What's unclear: changed lines versus changed files is not locked in context.
   - Recommendation: planner should lock line additions/removals in the new service projection unless existing product terminology says otherwise; include a contract test.
2. **Where does default-branch identity come from?**
   - What we know: creation catalog exposes repository `default_branch`; workspace snapshot exposes current branch but not default branch.
   - What's unclear: multi-repository workspaces may have different defaults.
   - Recommendation: project default-branch status per repository pair, not once per workspace.
3. **Is PR/check data currently available from any authoritative service seam?**
   - What we know: no field appears in the current snapshot/native model and new provider integration is out of scope.
   - Recommendation: keep the UI slot and optional decoder contract; do not block VIS-01 when the service legitimately lacks the capability, matching “when available.”
