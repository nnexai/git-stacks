# Phase 107.2 UI Specification

## Design Direction

Dense and calm rather than dashboard-like. A workspace row is the primary operational summary. Use GTK symbolic icons and libadwaita semantic colors; do not copy SF Symbols, branded artwork, or hard-coded macOS palette values.

Use a 60/30/10 allocation: roughly 60% neutral content/background, 30% elevated or selected surfaces and structural chrome, and at most 10% semantic accent. Selection and keyboard focus always outrank decorative status color.

Typography uses at most four GTK/libadwaita roles and two weights: body semibold for primary titles, body regular for actions, caption regular for metadata/counters, and heading semibold for section labels. Git digits use the caption size with monospaced digits. Preserve the native role metrics with a minimum 20px body line box and 16px caption line box; do not tighten line height through custom CSS.

All metrics use the approved spacing set: 4px micro gap, 8px internal gap and row horizontal inset, and 16px section rhythm. Target row height is 44px with a secondary line and 36px without one.

## Sidebar Information Hierarchy

1. Global `Pinned` section, always expanded when nonempty.
2. Global `Active` section for unpinned workspaces with signals, agents, or running work.
3. Remaining workspaces in the selected label/repository organization.

Active relevance order:

1. unread + awaiting input + running
2. unread + awaiting input
3. unread + agent + running
4. unread + agent
5. unread + running
6. awaiting input + running
7. awaiting input
8. agent + running
9. agent present
10. running

Alphabetical workspace name is the stable tie-break. A workspace appears once even when it qualifies for multiple sections.

## Workspace Row Anatomy

- Leading 16px symbolic icon: repository/branch, folder, remote, missing, creating, archiving, deleting, or error.
- Primary title: workspace or user override; one line, ellipsized.
- Secondary line when useful: repository, branch, `Default`, remote host, or concise status; never repeat the title.
- Trailing fixed-width status cluster, in order:
  - Git `+N -N` counters with monospaced digits.
  - Pull-request/check badge when available.
  - Up to three overlapping provider badges, then `+N`.
  - Activity ping and/or unread notification marker.
- Pin is expressed by section placement plus accessible state; a subtle star may appear on hover/focus or single-repository rows but must not crowd the status cluster.

## Accent Semantics

- Default/main branch: warning/yellow semantic accent plus text label `Default` where ambiguity exists.
- Pinned: orange semantic accent, primarily through section/header or small marker.
- Ordinary: neutral/tertiary.
- Git additions/removals: success green / error red with explicit `+` and `-` text.
- Awaiting input/unread notification: orange; never use the same animation as background work.
- Running/activity: provider/script tint with expanding ring; reduced motion uses a static outlined/solid dot.
- Missing/degraded: warning icon and redundant status text.
- Selected rows reduce decorative colors toward high-contrast foreground/secondary tokens so selection remains dominant.

No meaning relies on color alone. Every indicator has an accessible label and tooltip; combined row descriptions enumerate the meaningful states.

## Provider Presence

- Circular 14–16px provider badge using available provider mark or a stable letter fallback.
- Provider artwork must be locally owned, redistributable, and documented. Otherwise use a symbolic or letter fallback; never copy Supacode assets.
- One badge per active provider session; awaiting-input sessions sort first.
- Awaiting input uses a contrast inversion/ring treatment, not a provider-color replacement.
- Badge group accessible label includes provider counts and awaiting state, for example: `3 agent sessions: 2 Codex, 1 Copilot; 1 awaiting input`.

## Activity and Notification Markers

- Activity: 6px center dot plus expanding one-pixel ring, approximately one-second ease-out cycle.
- Multiple activity tints may cycle slowly; Reduce Motion selects the highest-priority static tint.
- Notification: separate solid orange 6px dot/button. When activity and notification coexist, show the notification center above the activity ring.
- Clicking an unread marker opens the signal inbox scoped to that workspace; keyboard activation provides the same action.

Supacode source drives its ping from running-script tints; agent presence is rendered separately as provider badges. Git-stacks deliberately adapts the ring to unified background activity from Phase 107.1 signals. This is a git-stacks semantic adaptation, not a copied Supacode meaning.

Normative motion control is the GTK/libadwaita animation setting. When animations are disabled, render a static outlined/solid marker with identical accessible state.

## Signal Inbox

- Separate `Needs attention` and `Recent activity` groups.
- Row hierarchy: provider + lifecycle chip, title, one-line detail, exact workspace/repository/terminal location, relative time.
- Primary actions: focus exact context and dismiss notification. Activity is not manually dismissed unless projected as a notification.
- Completed/idle activity is history, not unread attention.
- Empty state explains that hooks/OSC/ACP and automation signals appear here; integration health is a secondary link, not persistent chrome.

Exact state copy and recovery actions:

- Needs-attention empty: `No signals need attention` / `Waiting and failed agent sessions and unread notifications will appear here.`
- Recent-activity empty: `No recent activity` / `Agent and automation activity for this workspace will appear here.`
- Filtered empty: `No signals match this filter` with `Clear filters`.
- Loading: `Loading signals…`.
- Reconnecting: `Signals are reconnecting` / `Existing activity is retained while git-stacks reconnects.` with `Retry signals`.
- Integration degraded: `Some agent integrations need attention` with `Open integration health`.
- Load failure: `Signals could not be loaded` with `Retry signals` and `Connection details`.
- Focus failure: `That terminal is no longer available` with `Open workspace` when a fallback survives.
- Dismiss failure: `Notification could not be dismissed` with `Retry dismissal`; keep the row visible.
- Primary row actions are `Focus terminal` and `Dismiss notification`.

## Deterministic Compression

- Wide: show secondary line, Git counters, PR/check, up to three session badges plus overflow, activity, and unread.
- Medium: remove the secondary line; keep every trailing semantic group.
- Narrow or 200% text: collapse PR/check into one accessible icon, collapse agents to the highest-priority badge plus `+N`, then hide Git counters. Awaiting-input and unread markers never disappear.
- If space is still insufficient, move Git/PR details into the row tooltip/details popover; never overlap or clip markers.
- Long names ellipsize before trailing state compresses.

## Section Origin and Discovery

Rows hoisted into Pinned or Active appear only once. Their original label/repository group shows a compact summary such as `+1 pinned` or `+2 active`; activation focuses the corresponding global section.

## Row Interaction States

- Active-window selected: libadwaita selected background and high-contrast foreground; decorative accents become secondary except unread/error markers.
- Inactive-window selected: neutral elevated background with secondary outline; no accent fill.
- Keyboard focus: one distinct focus ring outside the selected fill; never a second selection border.
- Hover: subtle neutral elevation only.
- Attention does not recolor the whole row; it uses the explicit unread/awaiting marker.

## Themes and Accessibility

- Validate libadwaita dark, light, high-contrast, inactive-window selection, and system accent changes.
- Text and icons remain legible at 200% scaling without hiding Git counters behind overlapping badges.
- Narrow mode progressively hides secondary metadata, then Git counters, but retains title, awaiting/unread marker, and accessible full description.
- Visible keyboard focus is distinct from selection and never creates a double outline.
- Animations respect the native GTK/libadwaita animation setting.

## Supacode Source Findings Applied

- `WorktreeAccent`: main yellow, pinned orange, ordinary tertiary; selected rows neutralize decorative colors.
- `DiffStatsContent`: compact green/red monospaced counters.
- `AgentBadgeView` and `AgentAvatarGroupView`: overlapping circular provider marks with awaiting-input contrast inversion and overflow count.
- `SidebarPingDots`: source reference for a 6px running-script ring/dot, multiple tints, and Reduce Motion fallback; git-stacks intentionally adapts it to unified activity signals.
- `SidebarActiveClassification`: priority-derived Pinned/Active hoisting.
- `SidebarItemView`: fixed trailing cluster combining Git, PR, providers, scripts, and notifications.

## Evidence Matrix

- Golden projection tests for every row state and relevance ordering.
- GTK contract tests for CSS classes, symbolic icons, accessible names/descriptions, focusability, and action targets.
- Screenshot evidence in dark/light/high contrast at normal/narrow/200% text.
- Motion evidence with animation enabled and reduced.
- Real Codex, Copilot, Claude, and OpenCode presence/awaiting/completed scenarios.
- Deterministic overflow: four or more sessions, duplicate providers, simultaneous PR/check + Git + agents + activity + unread, long names, inactive-window selection, and failed focus/dismiss actions.
- RTL is normative: use logical start/end placement so leading identity and trailing status clusters mirror correctly. Any upstream GTK defect must be documented with a focused regression/limitation rather than disabling RTL acceptance.
- Blocking human UAT using dense multi-workspace fixtures.
