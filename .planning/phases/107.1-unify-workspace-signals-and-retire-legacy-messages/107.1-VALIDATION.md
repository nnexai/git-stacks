# Phase 107.1 Validation Architecture

## Contract Loop

- Every signal schema/state-machine task starts with failing behavior tests for valid and invalid activity/notification combinations, coalescing, dismissal, and completed-state readability.
- TypeScript contract fixtures and Zig golden fixtures use the same canonical examples.
- A generated/checked capacity manifest covers signal counts and every bounded string field.

## Migration Loop

- No committed wave may leave a managed hook, OSC adapter, ACP adapter, CLI command, service route, journal event, or native decoder targeting a removed contract.
- Service/journal/SSE/CLI/provider producer cutover, native discovery/decoder/model/ABI consumer cutover, and obsolete publication removal run in one atomic task and commit. This intentional file-scope exception prevents any committed dual protocol or incompatible producer/consumer state.
- After each migration wave, focused TypeScript tests, native decoder/model tests, typecheck, and dependency-cycle checks must pass.

## Stateful Verification

- Activity coalesces by provider session and surface through working, waiting, failed, completed, and idle transitions.
- Completed activity is not unread/actionable unless accompanied by a discrete notification.
- Notification dismissal is authenticated, service-authoritative, journaled, replayable, and converges after reconnect/restart across two clients.
- A dismissal never clears unrelated activity or another notification.
- Replay gaps rebuild an equivalent projection from authoritative service state.

## Integration Verification

- Phase 107 owned integrations emit normalized signals through exact-surface OSC without relying on terminal scraping.
- App-owned ACP fixtures normalize into the same signal contract; ordinary PTY commands are never labeled ACP-owned.
- CLI automation publishes both activity and notification signals through the sole endpoint.
- TUI and native inboxes show the same service-authoritative projection.

## Removal Gates

- Repository-wide forbidden-symbol checks cover legacy message storage/commands, `/v1/attention`, `service attention publish`, `StructuredAttentionEvent`, legacy `{code,message}` journal payloads, and transport-named native actions.
- README, help snapshots, completions, acceptance docs, fixtures, and tests must contain only the unified signal terminology except historical planning records.

## Final Gates

- `bun run typecheck`
- `bun run test:deps`
- focused signal/service/journal/replay/TUI suites
- native service-client/model/ABI/attention-projection suites
- `bun run native:verify`
- `bun run verify:gates`
- full `bun run test`
- blocking production UAT for hook/OSC, automation, replay/restart, cross-client dismissal, capacity, focus routing, and inbox rendering
