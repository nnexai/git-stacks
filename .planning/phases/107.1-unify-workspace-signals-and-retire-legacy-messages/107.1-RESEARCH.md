## User Constraints (from CONTEXT.md)

### Locked Decisions

- No backward compatibility is required for `git-stacks message`, legacy message storage, legacy journal payloads, or the current duplicate attention publication endpoint.
- One versioned signal envelope accepts hooks, terminal OSC, app-owned ACP sessions, and CLI automation.
- Signals have an explicit `kind`: `activity` or `notification`.
- Common fields are: stable signal identity, source/provider, workspace identity, optional repository/surface/session identity, optional lifecycle state, optional title/detail, and occurrence time.
- Activity lifecycle states remain `working`, `waiting`, `failed`, `completed`, and `idle` unless research proves a smaller lossless mapping.
- Activity coalesces by provider session plus exact surface. Discrete notifications do not coalesce with lifecycle activity and remain independently dismissible.
- `attention` is a native/UI projection over signals needing user action; it is not a transport or persistence type.
- A completed/idle activity signal never clears an unrelated notification.
- Use one authenticated service publication endpoint, one journal/SSE event type, one native decoder/reducer input, and one capacity contract.
- Delete obsolete implementations and tests rather than preserving aliases or compatibility branches.
- ACP is accepted only for sessions the client actually owns as ACP sessions. Ordinary terminal CLIs use Phase 107 provider integrations/OSC.
- Git-stacks stays integration-neutral and does not own agent policy or general session orchestration.

### the agent's Discretion

The context defines no separate discretion section. Field spellings, identifier prefixes, dismissal representation, and internal file decomposition are implementation discretion as long as the locked semantics hold.

### Deferred Ideas (OUT OF SCOPE)

- Chat, agent-to-agent messaging, prompt orchestration, session ownership, and a generic event bus for unrelated product domains.
- Redesigning the Phase 107 provider installer/OSC mechanism.
- Phase 108 packaging before Phase 107.1 verification passes.

## Summary

The repository currently has two notification systems and a third local-only ingress. The original system persists per-workspace JSONL files through `src/lib/messages.ts`, exposes `git-stacks message send/list/clear`, pushes best-effort records to a TUI Unix socket, and additively republishes `{workspace_id,code,message}` into the service journal. The newer system exposes `git-stacks service attention publish` and `POST /v1/attention`, journals `StructuredAttentionEvent`, and decodes both structured and legacy variants into native `Attention`. Terminal OSC bypasses the service entirely and reduces directly into the native model. [VERIFIED: repository source]

Phase 107.1 should make `signal` the sole transport and persistence noun. Provider-specific hooks and wrappers should publish activity signals through the CLI/service boundary; terminal OSC should normalize to the same native signal action immediately and, where service publication is feasible, use the same envelope; app-owned ACP updates should enter through the same normalization seam. The UI may continue to call its derived collection “attention,” but transport types, journal records, ABI actions, service capabilities, and capacities should use signal terminology. [VERIFIED: phase context and repository source]

**Primary recommendation:** land one strict discriminated signal schema and reducer state machine first, then switch every producer/consumer to it in a single migration wave and delete both old systems before verification.

## Recommended Signal Contract

Use a strict discriminated union. A recommended v1 shape is:

```ts
type Signal =
  | {
      version: 1
      id: `sig_${string}`
      kind: "activity"
      source: "claude" | "copilot" | "codex" | "opencode" | "automation" | "other"
      workspace_id: UUID
      repository_id?: UUID
      surface_id: UUID
      session_id: string
      state: "working" | "waiting" | "failed" | "completed" | "idle"
      title?: string
      detail?: string
      occurred_at: ISO8601
    }
  | {
      version: 1
      id: `sig_${string}`
      kind: "notification"
      source: "claude" | "copilot" | "codex" | "opencode" | "automation" | "user" | "other"
      workspace_id: UUID
      repository_id?: UUID
      surface_id?: UUID
      session_id?: string
      title: string
      detail?: string
      occurred_at: ISO8601
    }
```

[VERIFIED: locked context plus current bounded contract patterns]

Prescriptive refinements:

- Require `surface_id` and `session_id` for activity. The locked coalescing key cannot be represented losslessly without both. A provider wrapper without a provider session identifier should derive a stable process/session token at launch; it must not collapse all activity on one surface forever.
- Preserve the current invariant that `surface_id` implies `repository_id`; exact terminal surfaces belong to workspace/repository pairs in the native model.
- Keep `occurred_at` supplied by the producer for display, but use journal `sequence` for ordering and last-write resolution. Wall clocks cannot decide freshness safely.
- Keep `journal_sequence` outside the publication body and attach it only in the journaled event.
- Use UTF-8 byte limits identical across Zod discovery and Zig storage. Rename capacity keys (`signal_items`, `signal_id`, `signal_title`, `signal_detail`, `signal_occurred_at`) rather than retaining transport-level attention labels.
- Do not add an `attention` boolean. Whether a signal needs action is derived: `waiting` and `failed` are unread; `completed`, `working`, and `idle` are history/presence rather than unread attention; undismissed notifications are actionable/unread. A producer that needs a completion badge publishes a discrete notification.

## State, Coalescing, and Dismissal Semantics

### Activity

- Logical key: `(source, session_id, surface_id)`; `id` may be deterministic from that key or a separate stable identifier, but every update for one activity must reuse it.
- Apply only a signal with a greater journal sequence than the stored instance. Duplicate sequence/id is idempotent; older updates are ignored.
- Update replaces state/title/detail/time for the logical key. It does not append another row.
- `working`, `idle`, and `completed` resolve unread attention while retaining history/presence. Only `waiting` and `failed` activity project unread attention.
- Selecting/focusing marks that projected activity read locally; a later attention-worthy lifecycle transition makes it unread again.

### Notification

- Logical key: `id` only. Every notification is discrete even if source/title/surface match.
- Dismissal applies only to that notification id. It never mutates an activity and is not inferred from a lifecycle transition.
- Dismissal is an authenticated service mutation represented in the authoritative journal and replayed in sequence to every client. Native and TUI reducers consume that state but create no client-local dismissal persistence.

### Capacity

- Evict resolved/read activity first, then dismissed notifications, then oldest non-actionable activity. Never silently evict unread `waiting`/`failed` activity or undismissed notifications; report `capacity_exceeded`/overflow diagnostics.
- Current capacity is 64 attention items. Preserve 64 as the initial `signal_items` limit unless fixtures prove both activity state and notifications require separate pools.

## Phase 107 Normalized Boundary

Phase 107 already provides a useful provider-neutral lifecycle vocabulary in `AgentLifecycleState` and `normalizeAgentSessionUpdate()`, plus terminal identity environment variables and OSC parsing. Shape current Phase 107 work toward 107.1 by:

1. Keeping provider adapters limited to `(source, provider session, lifecycle state, optional content)` normalization.
2. Avoiding new imports of `StructuredAttentionEvent`, `AttentionEventPayload`, `service attention publish`, or `/v1/attention` outside the existing migration surface.
3. Passing exact workspace/repository/surface identity at terminal provisioning; add provider session identity at the first event where the provider exposes it.
4. Treating OSC as an ingress adapter, not a second state model. Its parser should return normalized lifecycle input suitable for constructing a signal.
5. Keeping ACP limited to sessions owned by the client; ordinary typed commands remain hooks/OSC as locked.

Current Phase 107 code still hardcodes attention terminology in `structuredAttentionCommand()`, wrapper scripts, `GIT_STACKS_ATTENTION_*` environment variables, OSC payload marker, and native callback names. Do not broaden that surface. A small interim internal function named generically (for example `publishLifecycle`) may still call the old endpoint until 107.1, but cross-cutting renames belong in 107.1. 1. **Contract and state-machine tests first:** add strict signal schema fixtures, invalid combination tests, activity transition/coalescing tests, discrete notification/dismissal tests, and capacity policy tests.

## Architecture Patterns

### Data Flow

```text
provider hook ----\
terminal OSC ------> normalized signal ingress -> strict service/native Signal
owned ACP session -/             |                    |
CLI automation ------------------+                    +-> journal -> broker/SSE -> native reducer
                                                        |                 |
                                                        +-> coalesced state+-> attention projection -> GTK/TUI
```

### Pattern 1: Discriminated domain event

Use `kind` as the discriminator and make illegal combinations unrepresentable: activity requires state/session/surface; notification requires title and forbids lifecycle state. Zod `.discriminatedUnion()` plus strict objects matches the existing contract style. [VERIFIED: `src/lib/service/contract.ts`]

### Pattern 2: Sequence-driven last-write-wins activity

Journal sequence, not timestamp, decides whether an activity update replaces current state. Keep the latest state per activity key and discrete entries per notification id. [VERIFIED: existing ordered journal cursor contract]

### Pattern 3: Projection below presentation

Expose `aggregate()`/projection functions over signal state. GTK should never decide coalescing, lifecycle resolution, or dismissal semantics. The current `attention_view.project()` and `model.aggregate()` already provide the correct tier boundary and should be adapted rather than duplicated. [VERIFIED: native source]

### Anti-Patterns to Avoid

- Dual publication APIs or additive “legacy remains authoritative” behavior.
- Treating a notification as fake `waiting` activity (the current legacy decoder does this).
- Coalescing by source/workspace alone; it merges parallel sessions.
- Using wall-clock time for ordering.
- Letting `completed`/`idle` clear another signal.
- Parsing terminal output beyond the authenticated/tokenized OSC adapter.
- Keeping TUI file polling after the store is deleted.
- Calling the transport `attention`; attention is derived presentation state.

## Common Pitfalls

### Missing provider session identity

The current service attention id hashes source plus workspace/repository/surface, so two same-provider sessions on one surface collide. Phase 107.1 requires a real `session_id` in the activity key. Provider hooks should extract their session identifier where available; process wrappers need a generated launch/session token. [VERIFIED: `src/commands/service.ts`]

### OSC and service divergence

Current OSC reduces directly into native state while hook commands can publish through the service. If both paths fire, they may duplicate one lifecycle. Use the same logical id/key and idempotent reducer, or choose one ingress per provider session. [VERIFIED: native app and integration-manager source]

### Dismissal lost on replay

If dismissal only removes a row, journal replay resurrects it. Persist dismissed ids/read sequence in native UI state or journal a dedicated control mutation; do not encode dismissal as an unrelated lifecycle state.

### Exact capability validation breakage

The Zig client validates exact discovery keys and capacity keys. Contract/native changes must land together and incompatible clients must fail clearly rather than silently ignore signals. [VERIFIED: `native/linux/service_client.zig`]

### Unsafe provider-config rewrite

User-level hooks are live OS/user configuration. Upgrade only entries carrying the git-stacks ownership marker and preserve all unrelated JSON/TOML/plugin content. [VERIFIED: Phase 107 integration manager]

### Capacity eviction hides urgent state

The current reducer looks for read/non-actionable victims but can overflow when all items are urgent. Preserve explicit overflow diagnostics and test that urgent unread signals are not silently discarded. [VERIFIED: `native/core/reducer.zig`]
