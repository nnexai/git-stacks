<feature>
  <name>Strict unified signal schema and reference state machine</name>
  <files>src/lib/service/contract.ts, src/lib/service/signal-state.ts, tests/lib/service/contract.test.ts, tests/lib/service/signal-state.test.ts</files>
  <behavior>
    - Activity requires version 1, stable signal id, source, workspace, exact surface, provider session, lifecycle state, and occurred_at; notification requires title and forbids lifecycle state.
    - surface_id requires repository_id; all bounded text uses UTF-8 byte limits that native storage can mirror.
    - A higher journal sequence replaces activity with the same source/session/surface key; duplicate or older sequence is idempotently ignored.
    - Two provider sessions on one surface and one provider session on two surfaces remain distinct.
    - Notification identity is its id, dismissal affects only that id, and completed/idle activity never dismisses a notification.
    - The reference state accepts journaled dismissal mutations, replays them idempotently, and never relies on native-local dismissal persistence.
    - `completed`, `idle`, and `working` activity remain history/presence and are not unread; only waiting/failed activity and undismissed notification signals project unread attention.
    - Capacity evicts resolved/read state before urgent unread state and reports overflow instead of silently dropping urgent signals.
  </behavior>
  <implementation>Write failing fixtures first, then replace transport-level StructuredAttention schemas with a strict SignalSchema discriminated union and create a pure signal-state module usable as the executable semantic reference. Define a service-owned, journaled dismissal mutation keyed only by notification id; dismissal sequence participates in replay ordering and is not a second signal kind. Keep journal_sequence outside publication input but require it for ordered reduction. Encode the locked unread projection exactly: waiting/failed activity and undismissed notifications are unread; completed/idle/working are history or presence only. Use explicit source and activity-state enums, version 1, stable prefixed identifiers, exact identity invariants, and shared signal capacity labels. Do not add native-local persistence, a compatibility union, attention boolean, generic event bus, or policy beyond lifecycle/notification state.</implementation>
</feature>
