# Phase 107.1 Context: Unified Workspace Signals

## Intent

Replace the never-productively-used legacy workspace message system and the parallel structured-attention transport with one slim, provider-neutral signal system before Phase 108 packaging and macOS proof.

Phase 107 remains responsible for finishing Supacode-style provider integration: owned user-level provider installers, exact terminal identity, terminal-scoped OSC routing, and normalized lifecycle input. New Phase 107 code should shape its internal event boundary toward this contract but must not perform the cross-cutting migration concurrently.

## Locked Decisions

- No backward compatibility is required for `git-stacks message`, legacy message storage, legacy journal payloads, or the current duplicate attention publication endpoint.
- One versioned signal envelope accepts hooks, terminal OSC, app-owned ACP sessions, and CLI automation.
- Signals have an explicit `kind`: `activity` or `notification`.
- Common fields are: stable signal identity, source/provider, workspace identity, optional repository/surface/session identity, optional lifecycle state, optional title/detail, and occurrence time.
- Activity lifecycle states remain `working`, `waiting`, `failed`, `completed`, and `idle` unless research proves a smaller lossless mapping.
- Activity coalesces by provider session plus exact surface. Discrete notifications do not coalesce with lifecycle activity and remain independently dismissible.
- Notification dismissal is service-authoritative: the authenticated service records and replays dismissal state, so every client converges without a second native-only persistence schema.
- `completed` activity remains useful session history but is not unread/actionable by itself. A provider or automation that needs a completion badge publishes a discrete notification signal.
- `attention` is a native/UI projection over signals needing user action; it is not a transport or persistence type.
- A completed/idle activity signal never clears an unrelated notification.
- Use one authenticated service publication endpoint, one journal/SSE event type, one native decoder/reducer input, and one capacity contract.
- Producer cutover and removal of the old attention/message publication paths must be atomic at committed plan boundaries; no intermediate commit may leave installed hooks targeting a removed command or endpoint.
- Delete obsolete implementations and tests rather than preserving aliases or compatibility branches.
- ACP is accepted only for sessions the client actually owns as ACP sessions. Ordinary terminal CLIs use Phase 107 provider integrations/OSC.
- Git-stacks stays integration-neutral and does not own agent policy or general session orchestration.

## Required Removal/Replacement Audit

- `src/commands/message.ts` and message registration/help/tests.
- Legacy workspace message persistence and `src/lib/messages.ts` publication coupling.
- `service attention publish`, `/v1/attention`, `StructuredAttentionEvent`, duplicate attention journal/SSE variants, and legacy `{code,message}` decoder branches.
- Native `attention_received` ABI/decoder naming where it represents transport rather than UI projection.
- Duplicate fixtures, compatibility tests, documentation, and capacity labels.

## Verification Expectations

- Contract/schema tests for both signal kinds and invalid combinations.
- Coalescing/dismissal state-machine tests, including independent notifications.
- Real hook-to-OSC-to-service/native tests from Phase 107 integrations.
- ACP normalization contract tests without pretending ordinary PTY sessions are ACP-owned.
- Journal replay, reconnect/gap rebuild, ordering, idempotency, backpressure, and capacity coverage.
- Native ABI golden fixtures plus Linux inbox/focus UAT.
- Repository-wide search proves removed legacy message/attention transport symbols do not remain except historical planning artifacts.

## Scope Fence

- Do not add chat, agent-to-agent messaging, prompt orchestration, session ownership, or a generic event bus for unrelated product domains.
- Do not retain two publication APIs for migration convenience.
- Do not redesign the Phase 107 provider installer/OSC mechanism; consume its normalized boundary.
- Do not begin Phase 108 packaging until Phase 107.1 verification passes.
