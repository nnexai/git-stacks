## Typography

Phase 125 introduces exactly two weights: regular `400` and semibold `650`. Existing brand marks and pre-existing provider badges retain their current special weights.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Metadata | 11px | 400 | 1.4 |
| Body | 13px | 400 | 1.45 |
| Label / control | 14px | 650 | 1.3 |
| Heading | 17px | 650 | 1.2 |

Shortcut key caps use the Metadata size at weight `650`, tabular numerals, and the declared monospace stack. Do not encode hierarchy by adding more font sizes or weights; use color, spacing, and grouping.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `var(--bg)` (`#0c0d0f` dark / `#fafafa` light) | Application field, overlay input, and terminal-adjacent negative space |
| Secondary (30%) | `var(--panel)` and `var(--panel-2)` | Overlay, result rows, shortcut groups, settings rows, and toolbar controls |
| Accent (10%) | `var(--accent)` (`#3584e4`) | Active fuzzy result, primary `Customize shortcuts` CTA, active capture target, and selected toolbar state |
| Destructive | `var(--danger)` (`#ff747f` dark / `#c83d4d` light) | Close-terminal failure and existing destructive close affordance only |

Accent reserved for: the one active result, the one primary action in a surface, active capture/binding state, and existing selected workspace/organization state. Hover alone uses `--panel-hover`; focus always uses `--focus`; attention uses `--warning`, not accent.

All accent-backed text uses `--accent-text`. Warning, danger, success, and disabled states must include text or an icon/label and must never rely on color alone.

## Surface and Component Contract

### Toolbar Discoverability

- Add a visible `Next attention` control and a visible `Keyboard shortcuts` control to the existing topbar `.toolbar`; both invoke the same canonical action callbacks used by shortcuts.
- At widths above 640px, show glyph plus text. At 640px and below, retain an icon-sized 32px control with `aria-label`, `title`, and the same focus ring; neither action may disappear into a generic overflow palette.
- `Next attention` uses the warning color only when at least one resolvable attention candidate exists. The no-candidate state stays enabled so invoking it can announce the locked no-attention result without changing focus or selection.
- `Keyboard shortcuts` opens help. Help exposes `Customize shortcuts`; shortcut settings do not require a separate persistent toolbar button.

### Singleton Overlay Frame

- Workspace switcher, configured commands, keyboard help, shortcut settings, capture mode, and existing confirmations share one overlay controller, one backdrop, and one document-level containment handler.
- Retain the existing modal geometry: width `min(620px, calc(100vw - 28px))`, maximum height `78vh`, 12px radius, 1px `--line` border, and existing shadow/backdrop. Search results and grouped settings scroll inside the body; the header and search field remain visible.
- Opening the same surface again focuses its primary input/control. Opening a different surface replaces the body in place without briefly restoring terminal focus or adding another backdrop.
- Every surface has one close button in the header whose accessible label names its surface (`Close workspace switcher`, `Close commands`, `Close keyboard shortcuts`, or `Close shortcut settings`). Backdrop click and Escape cancel without mutation. Existing destructive/editor modals remain exclusive: unrelated global actions report unavailable and do not replace them.

### Primary Focal Points

- The terminal-focused main screen's primary focal point is the active xterm input and its active terminal tab. When there is no active terminal, the focal point becomes `Start a shell` for the selected repository; when there is no selected repository, it becomes the first available workspace/repository row.
- The workspace switcher and configured-commands overlays use the search input plus the single accent-backed active result as their primary focal point. The input retains DOM focus while the active result supplies the visual focus through `aria-activedescendant`.
- Keyboard help's primary focal point is the grouped current-binding list, with `Customize shortcuts` as its sole accent-backed CTA.
- Shortcut settings' primary focal point is the binding row currently focused, saving, or reporting an error. Capture mode narrows the focal point to the highlighted row's `Press shortcut‚Ä¶` control and inline instruction.
- Existing confirmation overlays keep their warning/title and explicit confirm action as the visual focal point while their safe cancel control retains initial keyboard focus. Existing editor/creation overlays use their first unresolved field as the focal point.

### Searchable Overlay Pattern

- Use a labelled native input with combobox semantics: `role="combobox"`, `aria-expanded="true"`, `aria-controls`, and `aria-activedescendant`. The result container uses `role="listbox"`; every result uses `role="option"` and `aria-selected`.
- Keep exactly one active result whenever results exist. The highest-ranked result begins active after each query change. ArrowUp/ArrowDown wrap; Home/End jump; Enter invokes the active result; Escape closes; Tab and Shift+Tab remain inside the dialog.
- Active rows use a 1px `--accent` border plus `color-mix(in srgb, var(--accent) 16%, var(--panel-2))`; pointer hover uses only `--panel-hover`. Pointer movement updates the active row so pointer and keyboard states cannot diverge.
- Scroll the active row into view with `block: nearest`. Do not animate result reordering. Existing `prefers-reduced-motion` handling remains authoritative.
- A workspace result is a two-line row: workspace name as the primary label; repository name and branch as muted secondary text. A command result is a two-line row: command name; workspace/repository scope and `Recent` marker as secondary text.
- Empty query shows the deterministic default order. Querying never hides the input or replaces the modal; an empty result renders the copy defined above in the result region and Enter becomes a no-op.

### Keyboard Help

- Title: `Keyboard shortcuts`. Render three ordered groups: `Navigation`, `Workspace`, and `Terminal`.
- Navigation contains `Switch workspace or repository` and `Next attention`; Workspace contains `Configured commands` and `New workspace`; Terminal contains `New terminal`, `Close terminal`, `Previous terminal`, and `Next terminal`.
- Each row shows action label on the left and the current effective chord as individual `<kbd>` caps on the right. An unbound action shows the muted literal `Unbound`, never a blank cell.
- The current platform is reflected in rendered modifier names (`Ctrl`, `Command`, `Alt`, `Shift`) rather than glyph-only notation. Help always reflects effective service-projected bindings, including optional aliases when enabled.
- Footer actions are `Close help` and primary `Customize shortcuts`. Help has no search box; initial focus is `Close help` only when no prior invoker exists, otherwise the first content action.

### Shortcut Settings and Capture

- Title: `Customize shortcuts`. Group rows identically to help. Each row contains action label, current binding button, `Reset shortcut to default`, and `Unbind shortcut`; reset is disabled when already at the platform default and unbind is disabled when already unbound.
- On open, render a compact busy state with `aria-busy="true"` and the loading copy. Do not flash defaults as though they were authoritative. A load failure renders the documented error plus a primary `Retry loading shortcuts` control; editing stays unavailable.
- Selecting a binding enters capture mode inside the same modal and highlights only that row. Replace its binding button with `Press shortcut‚Ä¶`, render the capture instruction directly beneath it, and expose an `aria-live="polite"` status. Escape exits capture and returns focus to that row without closing settings.
- Capture ignores pure modifier presses and composition/AltGraph events. A printable key without Ctrl, Alt, or Command renders the unsafe-capture error inline and leaves the old binding active.
- A conflict renders the documented warning inline beneath the row, names the other action, keeps both existing bindings unchanged, and leaves capture active. It must never silently swap, overwrite, or unbind another action.
- A valid save disables that row's controls, shows `Saving shortcut‚Ä¶`, and updates the key caps only after the typed service operation succeeds. Failure restores the prior caps and renders the save error with `Retry saving shortcut`. Success announces `{Action} shortcut updated.` through the existing polite toast/live region.
- `Reset shortcut to default` and `Unbind shortcut` are explicit one-click settings mutations, not destructive confirmations. Their in-flight, success, and failure treatment is identical to capture save.

### Focus and Terminal Input Boundary

- When an app overlay opens from xterm, record the active terminal ID before moving focus. On close or successful selection, restore that terminal if it still exists; otherwise focus the new active terminal, then the invoking toolbar control as the final safe fallback.
- When one overlay replaces another, preserve the original terminal return target. Never focus the removed overlay or the document body between replacements.
- Search/navigation keys, binding capture keys, Enter, Escape, and Tab used by an active overlay are consumed once and never reach xterm. Unmatched, rejected, composition, and AltGraph events outside capture continue to the PTY unchanged.
- A focus trap contains Tab and Shift+Tab across visible, enabled controls. Disabled controls are skipped. Result navigation uses active-descendant focus on the search input rather than moving DOM focus through every row.
- Repeated overlay shortcuts only refocus the current primary control. Repeated create, close, terminal traversal, or attention actions do nothing. No repeated key can produce a second modal, service operation, or toast storm.

### Action Feedback and Availability

- All pointer controls, toolbar controls, overlay selections, and shortcuts call the same canonical action registry callback and consume its current availability/disabled reason.
- New Terminal, Close Terminal, Previous/Next Terminal, and Next Attention do not open a modal for their empty cases; they preserve selection and focus and announce the exact copy above through the existing polite toast region.
- New Workspace reuses the existing creation modal. Configured Commands is unavailable without an active workspace/repository and announces `Select a repository first.` without opening an empty palette.
- Next Attention switches workspace/repository first, then the referenced live terminal when present. A valid repository-only target focuses its repository. Stale-surface targets are skipped without a warning toast or signal dismissal.

### Open UX and Accessibility Coverage

- Dialogs are labelled, modal, focus-contained, and closeable by Escape and the visible close button. Search uses combobox/listbox semantics and announces result count changes through a polite live region.
- All controls remain keyboard reachable with the established 2px focus-visible ring. Key caps are supplementary text, not the only action label.
- Light/dark/system themes use existing variables. No new animation is required; the repository-wide reduced-motion rule remains sufficient.
- Layout and shortcut capture do not assume a US-produced glyph. Display physical letter bindings by normalized action key while matching remains code-based as locked in CONTEXT.md.
