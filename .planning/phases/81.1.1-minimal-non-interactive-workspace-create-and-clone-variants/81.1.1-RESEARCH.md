## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Add an explicit `--non-interactive` flag to the existing `new` and `clone` commands rather than creating separate commands or implicit mode switches.
- **D-02:** `--non-interactive` is a strict mode: if required inputs are missing, the command must fail fast with a clear error listing what is missing.
- **D-03:** Ship `--non-interactive` for both `new` and `clone` in this phase.
- **D-04:** In `--non-interactive` mode, branch should keep the current defaulting behavior when omitted rather than becoming mandatory.
- **D-05:** Keep the input surface minimal: reuse existing args/flags plus optional `--branch`; do not add description/ports flags in this phase.
- **D-06:** `clone --non-interactive` should take the destination workspace name via `--name <new-name>`.
- **D-07:** Skip the integration override prompt and use the existing inherited/default integration cascade as-is.
- **D-08:** Do not open the workspace automatically in `--non-interactive` mode unless the user explicitly passes `--open`.
- **D-09:** When optional metadata is omitted in `new --non-interactive`, skip the prompts entirely: keep CLI labels if provided, otherwise no extra labels, empty description, and no extra ports.
- **D-10:** Phase 81.1.1 only needs to cover the paths required to unblock later E2E work: template-backed `new --non-interactive` and `clone --non-interactive`.
- **D-11:** Template-backed `new --non-interactive` must support both `--from <template>` and repeatable `--template` composition so Phase 82 can still cover template composition.

- Exact helper extraction for "missing required inputs" validation and prompt-skipping control flow.
- Exact error wording, as long as the mode clearly reports which inputs are required in non-interactive usage.
- Exact implementation of `--open` and whether prompt-skipping behavior is centralized or handled inline in each flow.

### Deferred Ideas (OUT OF SCOPE)
- Non-interactive ad-hoc repo-picking for `new`.
- Non-interactive `new --from <local-path>`.

## Summary

This phase adds `--non-interactive` mode to two existing CLI commands by threading new flags through the Commander.js command definitions (`workspace.ts`) into the TUI flow functions (`workspace-wizard.ts`, `workspace-clone.ts`). The core insight is that no new lifecycle infrastructure is needed — the existing `createWorkspace()` and inline worktree/write logic in `runWorkspaceClone` can already be called with all data resolved upfront. The non-interactive path simply pre-resolves the data that prompts currently gather, skips those prompts, and passes the resolved values straight to the creation/clone work.

The most surgical implementation threads `nonInteractive`, `branch`, and `open` as optional parameters into `runWorkspaceNew` and `runWorkspaceClone`. An early-return validation guard handles the fail-fast requirement (D-02). All downstream creation logic (worktree ops, `writeWorkspace`, integration generate, `openWorkspace`) is reused without modification.

**Primary recommendation:** Add a non-interactive guard at the top of each TUI function. Validate required inputs, compute defaults for optional ones (branch, labels), then fall through to the same creation/clone work the interactive path already uses — skipping only the prompt calls themselves.

### Pattern 1: Non-Interactive Guard at Function Top

**What:** Extend both TUI function signatures with optional `nonInteractive`, `branch`, and `open` parameters. Add an early-return guard block after the header setup that resolves all data and calls the work directly.

**When to use:** Best fit here because the interactive path stays completely unchanged — guard is an additive branch, not a refactor of the existing flow.

```typescript

// workspace-wizard.ts
export async function runWorkspaceNew(
  nameArg?: string,
  fromSource?: string,
  templateNames?: string[],
  cliLabels?: string[],
  opts?: { nonInteractive?: boolean; branch?: string; open?: boolean },
) {
  p.intro("New workspace")
  const config = readGlobalConfig()
  const tasksDir = getTasksDir(config.workspace_root)

  // === NON-INTERACTIVE PATH (D-01, D-02, D-04, D-07, D-08, D-09) ===
  if (opts?.nonInteractive) {
    // D-02: fail fast on missing required inputs
    const missing: string[] = []
    if (!nameArg) missing.push("<name> (positional argument)")
    if (!fromSource && (!templateNames || templateNames.length === 0))
      missing.push("--from <template> or --template <name>")
    if (missing.length > 0) {
      console.error(`[git-stacks] --non-interactive requires: ${missing.join(", ")}`)
      process.exit(1)
    }
    const wsName = nameArg!.trim()

    // D-09: labels from CLI only; description empty; no extra ports
    const labels = normalizeLabels(cliLabels ?? [])

    // D-04: compute branch default same way as interactive path
    const branch = opts.branch?.trim()
      || (() => {
        const tpl = (fromSource && templateExists(fromSource)) ? readTemplate(fromSource) : undefined
        const firstPattern = tpl?.repos.find(r => r.branch_pattern)?.branch_pattern
        return firstPattern ? expandBranchPattern(firstPattern, wsName) : `feature/${wsName}`
      })()

    // Resolve template (same logic as interactive path, refs existing functions)
    // ... resolve repos, wsHooks, wsEnv, wsEnvFile, wsFiles, wsIntegrationSettings, wsPorts ...
    // D-07: skip promptIntegrationOverrides — wsIntegrationSettings stays as template snapshot

    const result = await createWorkspace(
      { wsName, branch, repos, templateName, /* ... */ labels },
      (msg) => p.log.info(msg),
    )
    if (!result.ok) {
      if (result.rollbackErrors.length > 0) result.rollbackErrors.forEach(e => p.log.warn(e))
      console.error(`[git-stacks] Failed to create workspace: ${result.error}`)
      process.exit(1)
    }
    // D-08: only open if --open explicitly passed
    if (opts.open) {
      const openResult = await openWorkspace(wsName, {}, (msg) => p.log.info(msg))
      if (!openResult.ok) p.log.warn(`Open failed: ${openResult.error}`)
    }
    p.outro(`Workspace '${wsName}' ready.  Run \`git-stacks open ${wsName}\` to re-open.`)
    return
  }
  // === END NON-INTERACTIVE ===

  // ... existing interactive path unchanged ...
}
```

### Pattern 2: Commander.js Flag Threading

**What:** Declare new flags in `workspace.ts`, pass them into the TUI functions via the opts object.

```typescript
// workspace.ts — new command
program
  .command("new [name]")
  .description("Create a new workspace interactively")
  .option("--from <source>", "Create from template name or local repo path")
  .option("--template <name>", "Compose from template(s) — repeatable",
    (val: string, arr: string[]) => { arr.push(val); return arr }, [] as string[])
  .option("--label <tag>", "Set label on workspace (repeatable)",
    (val: string, arr: string[]) => { arr.push(val); return arr }, [] as string[])
  .option("--branch <branch>", "Branch name (non-interactive; defaults to feature/<name>)")
  .option("--non-interactive", "Skip all prompts; fail fast if required inputs are missing")
  .option("--open", "Open workspace immediately after creation (non-interactive mode)")
  .action(async (name, opts) => {
    if (name !== undefined) validateName(name)
    if (opts.from && opts.template && opts.template.length > 0) {
      console.error("[git-stacks] Error: --from and --template are mutually exclusive")
      process.exit(1)
    }
    if (opts.nonInteractive && !opts.from && (!opts.template || opts.template.length === 0)) {
      // Can catch this here too, but TUI function provides the canonical check
    }
    const templateNames = opts.template?.length > 0 ? opts.template : undefined
    await runWorkspaceNew(name, opts.from, templateNames, opts.label,
      { nonInteractive: opts.nonInteractive, branch: opts.branch, open: opts.open })
  })

// workspace.ts — clone command
program
  .command("clone [workspace]")
  .description("Clone a workspace with a new name and branch")
  .option("--name <new-name>", "New workspace name (required in --non-interactive)")
  .option("--branch <branch>", "Branch name (non-interactive; defaults to feature/<name>)")
  .option("--non-interactive", "Skip all prompts; fail fast if required inputs are missing")
  .option("--open", "Open workspace immediately after cloning (non-interactive mode)")
  .action(async (workspace, opts) => {
    await runWorkspaceClone(workspace,
      { nonInteractive: opts.nonInteractive, name: opts.name, branch: opts.branch, open: opts.open })
  })
```

### Anti-Patterns to Avoid

- **Forking into a completely separate function:** `runWorkspaceNewNonInteractive()` — duplicates all the template resolution / worktree build logic, creating maintenance surface.
- **Threading `nonInteractive` deep into every sub-call:** Spreading the flag down into `buildReposFromTemplate`, `composeTemplates`, etc. — they don't need it; only prompt calls and the open confirm need skipping.
- **Changing interactive flow:** Any refactor of the prompt sequence in the existing code path. This phase must not touch the interactive flow.
- **Skipping `p.intro()` / `p.outro()`:** These produce terminal output, not interactive prompts. They should run in non-interactive mode too for consistent log output.

**Key insight:** The creation/clone work is already separated from prompt collection in concept. The non-interactive path is purely a "collect without prompts" layer on top of the same downstream calls.

## Common Pitfalls

### Pitfall 1: `--from` and `--template` mutual exclusion in non-interactive
**What goes wrong:** User passes both `--from` and `--template`; code tries to satisfy both, producing confusing behavior.
**Why it happens:** The existing interactive path already rejects this combination in `workspace.ts` before calling `runWorkspaceNew`. But the check is in the commander action, not in `runWorkspaceNew` itself.
**How to avoid:** Keep the existing guard in `workspace.ts`. The TUI function's non-interactive path only sees one of the two cases.
**Warning signs:** Tests that pass both flags simultaneously.

### Pitfall 2: Missing `--name` in `clone --non-interactive` not caught clearly
**What goes wrong:** User runs `git-stacks clone source-ws --non-interactive` without `--name`; gets a generic prompt-based failure instead of a clear error.
**Why it happens:** Without the guard, `runWorkspaceClone` falls through to the interactive name prompt in non-interactive context.
**How to avoid:** The guard at the top of the non-interactive branch must validate `opts.name` presence before any other work.
**Warning signs:** Non-interactive tests that omit `--name` should exit(1) with a message naming the missing flag.

### Pitfall 3: Template resolution produces empty `repos` array silently
**What goes wrong:** `buildReposFromTemplate` skips repos whose registry entry is missing (emits a `p.log.warn`), resulting in a workspace created with zero repos — then `createWorkspace` is called with an empty repos array.
**Why it happens:** `buildReposFromTemplate` already has this behavior for interactive mode; non-interactive callers inherit it.
**How to avoid:** After calling `buildReposFromTemplate`, check `repos.length === 0` and fail fast with a clear error in non-interactive mode. [ASSUMED — the interactive flow has an existing guard `if (repos.length === 0) cancel()` that should also fire, but may not produce a helpful non-interactive message]
**Warning signs:** Non-interactive create succeeds with 0 repos in the workspace YAML.

### Pitfall 4: `--open` flag name collision with existing `open` subcommand
**What goes wrong:** Commander.js may misparse `git-stacks new myws --open` if `open` is both a subcommand and a flag.
**Why it happens:** Commander's top-level program has `open <workspace>` as a subcommand. Adding `--open` to `new` and `clone` is a flag on those subcommands, not the top-level — should be safe.
**How to avoid:** Add `--open` as a flag (`.option("--open", ...)`) on the `new` and `clone` sub-commands specifically, not on the top-level program. Confirm with a quick manual test.
**Warning signs:** `commander` emits "unknown command" or parses `--open` as a value to `new`.

### Pitfall 5: `p.intro()` call in non-interactive context writing to terminal
**What goes wrong:** CI/subprocess environments may not expect or want TUI intro/outro output.
**Why it happens:** `p.intro()` uses `@clack/prompts` which writes styled output to stdout/stderr. In a subprocess E2E test context, this pollutes assertion output.
**How to avoid:** This is acceptable per CONTEXT.md — E2E tests run in a subprocess anyway and capture stdout/stderr. `p.intro()`/`p.outro()` output is not interactive, just decorative. The E2E phase (82) can assert on `outro` text for success confirmation. [ASSUMED — the E2E test authors will need to account for this in their assertions]

## Open Questions

1. **Should `--open` be added to the interactive path too, or only to non-interactive?**
   - What we know: CONTEXT.md D-08 says "do not open in non-interactive unless `--open` is passed." The interactive path already has a `p.confirm` for opening.
   - What's unclear: Whether the flag should be wired to skip the confirm and force-open in interactive mode too.
   - Recommendation: Wire `--open` only to the non-interactive path for this phase. Leave the interactive open confirm unchanged. This matches the scope boundary.

2. **Should `--non-interactive` be rejected if `--from <local-path>` is supplied?**
   - What we know: D-10 says non-interactive `new` only covers template-backed creation. D-Deferred explicitly lists "Non-interactive `new --from <local-path>`" as out of scope.
   - What's unclear: Whether to silently ignore the local-path case or actively reject it.
   - Recommendation: In non-interactive mode, detect that `fromSource` resolves to a local path (check `existsSync(resolved)`) and exit with a clear "local-path --from is not supported in --non-interactive mode" error. This keeps the scope boundary explicit.
