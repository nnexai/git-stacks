# Phase 81.1.1: Minimal non-interactive workspace create and clone variants - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Add the smallest explicit non-interactive CLI variants needed on `git-stacks new` and `git-stacks clone` so later subprocess E2E phases can create template-backed workspaces and clone existing workspaces without driving TUI prompts.

This phase is intentionally narrow. It covers `new --non-interactive` for template-backed creation (`--from <template>` and repeatable `--template` composition) plus `clone --non-interactive` for cloning an existing workspace with supplied inputs. It does not broaden into ad-hoc repo-picking flows, local-path `--from` automation, full wizard-to-flag parity, or a general CI automation redesign.

</domain>

<decisions>
## Implementation Decisions

### CLI surface
- **D-01:** Add an explicit `--non-interactive` flag to the existing `new` and `clone` commands rather than creating separate commands or implicit mode switches.
- **D-02:** `--non-interactive` is a strict mode: if required inputs are missing, the command must fail fast with a clear error listing what is missing.
- **D-03:** Ship `--non-interactive` for both `new` and `clone` in this phase.

### Required inputs and defaults
- **D-04:** In `--non-interactive` mode, branch should keep the current defaulting behavior when omitted rather than becoming mandatory.
- **D-05:** Keep the input surface minimal: reuse existing args/flags plus optional `--branch`; do not add description/ports flags in this phase.
- **D-06:** `clone --non-interactive` should take the destination workspace name via `--name <new-name>`.

### Skipped prompt behavior
- **D-07:** Skip the integration override prompt and use the existing inherited/default integration cascade as-is.
- **D-08:** Do not open the workspace automatically in `--non-interactive` mode unless the user explicitly passes `--open`.
- **D-09:** When optional metadata is omitted in `new --non-interactive`, skip the prompts entirely: keep CLI labels if provided, otherwise no extra labels, empty description, and no extra ports.

### Scope boundary
- **D-10:** Phase 81.1.1 only needs to cover the paths required to unblock later E2E work: template-backed `new --non-interactive` and `clone --non-interactive`.
- **D-11:** Template-backed `new --non-interactive` must support both `--from <template>` and repeatable `--template` composition so Phase 82 can still cover template composition.

### the agent's Discretion
- Exact helper extraction for “missing required inputs” validation and prompt-skipping control flow.
- Exact error wording, as long as the mode clearly reports which inputs are required in non-interactive usage.
- Exact implementation of `--open` and whether prompt-skipping behavior is centralized or handled inline in each flow.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `workspace.ts` already exposes `new [name]`, `clone [workspace]`, `--from`, `--template`, and `--label`, so this phase can extend existing verbs instead of adding a parallel CLI.
- `workspace-wizard.ts` already has meaningful branch defaults (`branch_pattern` expansion or `feature/<name>`) that can be reused when `--branch` is omitted in non-interactive mode.
- `workspace-clone.ts` already defaults clone branches to `feature/<newName>`, which provides a sensible non-interactive fallback.

### Established Patterns
- Workspace creation/cloning currently funnels through prompt-driven TUI flows even when partial inputs are supplied on the CLI.
- Optional metadata and integration override prompts are layered on top of core create/clone logic, making them good candidates for skipping rather than rethinking.
- The final open step is already isolated as a prompt + subsequent `openWorkspace()` call, which makes an explicit `--open` opt-in feasible.

### Integration Points
- `src/commands/workspace.ts` will likely gain the new flags and required-input validation before handing off to the create/clone flows.
- `src/tui/workspace-wizard.ts` and `src/tui/workspace-clone.ts` are the core sites where prompt-skipping branches or helper extraction will be needed.
- `tests/commands/` will later consume this phase directly for subprocess E2E coverage of template-backed create/clone behavior.

</code_context>

<specifics>
## Specific Ideas

- The mode should feel like “same command, but fully specified,” not like a new automation subsystem.
- `--non-interactive` should be safe for E2E and automation by refusing to guess required data while still preserving today’s established branch defaults.
- Opening the workspace should become an explicit side effect (`--open`) instead of something implicitly prompted during automation-oriented runs.

</specifics>

<deferred>
## Deferred Ideas

- Non-interactive ad-hoc repo-picking for `new`.
- Non-interactive `new --from <local-path>`.
- Full flag parity for all optional wizard fields.

</deferred>
