# Phase 88: Functional Coverage Readiness Gate - Research

**Researched:** 2026-05-15
**Status:** Complete

## Research Question

What does Phase 88 need in order to turn the post-Phase 85-87 functional coverage work into a local readiness gate and release evidence without finalizing the milestone?

## Findings

### Existing Gate Architecture

- `bun run verify` is the maintainer-facing local release-prep command. It runs `verify:prereqs`, `coverage`, `verify:gates`, `test`, `test:deps`, and `typecheck` in order through `scripts/verify.ts`.
- `scripts/verify-gates.ts` already aggregates findings instead of failing fast. Its `VerifyGateReport` currently covers live command inventory drift, unmapped in-scope inventory items, broken mapped test paths, coverage artifact shape problems, and coverage sentinel problems.
- Phase 84.1 added sentinel hit checks against `.coverage/coverage-final.json`, proving targeted nonzero source-hit gates are practical and consistent with the existing gate style.
- `tests/lib/verify-gates.test.ts` already has fixture helpers for temp roots and minimal Istanbul-shaped coverage maps. Phase 88 should extend that style instead of running expensive coverage commands from unit tests.

### Readiness Evidence Shape

- The canonical source artifact remains `.coverage/coverage-final.json` generated by `bun run coverage`. Phase 88 should not require maintainers to manually stitch together `coverage:unit`, `coverage:integ`, or ad hoc scoped reports.
- A functional readiness result needs machine-readable categories for:
  - covered functional areas,
  - accepted gaps,
  - deferred external-environment coverage,
  - must-fix-before-release gaps.
- This category model should distinguish green tests, source files with coverage hits, and functional confidence. A passing test suite alone is not sufficient; the gate must also check the Phase 85-87 functional core areas that are expected to be covered after those phases complete.
- Raw `.coverage/` artifacts should remain generated and ignored. Committed evidence should be a script, tests, and a concise release-readiness document or checklist.

### Phase 85-87 Inputs

- Phase 85 targets core workspace/git/hooks/files/env/config behavior with real temp dirs, isolated config homes, and local git repositories.
- Phase 86 targets automation-safe workspace command workflows such as `open --recreate`, `clean --gone`, destructive safety, and wrapper commands through real CLI subprocess tests.
- Phase 87 targets integration source modules and injected executor contracts, while real external forge auth, browsers, IDE launches, desktop tools, and window managers remain deferred.
- Phase 88 can depend on these planned surfaces by naming source-area sentinels and categories, but execution should happen after Phase 87 so the final inventory reflects actual completed coverage.

### Scope Boundaries

- In scope: functional-only coverage readiness model, local gate integration, readiness docs, accepted/deferred/must-fix classification, and requirements/inventory clarity.
- Out of scope: broad TUI rendering coverage, live external integration environments, CI workflows, numeric threshold policy unless it is demonstrably more stable than sentinels, archive/tag/publish, and `$gsd-complete-milestone`.
- COVR-05 explicitly leaves future numeric coverage thresholds possible after coverage stabilizes. Phase 88 should preserve this by documenting that the readiness gate uses targeted sentinels now and does not introduce percentage thresholds.

## Recommended Implementation Shape

1. Add a small functional readiness module and machine-readable inventory:
   - `tests/functional-readiness-inventory.ts` for explicit area definitions and classifications.
   - `scripts/functional-coverage-readiness.ts` for parsing `.coverage/coverage-final.json`, calculating hit status, reporting accepted/deferred/must-fix categories, and exposing testable functions.
   - `tests/lib/functional-coverage-readiness.test.ts` for fixture-driven unit coverage.

2. Wire readiness into local verification:
   - Add a `functionalReadiness` section to `VerifyGateReport` in `scripts/verify-gates.ts`.
   - Keep `verify:gates` aggregated, with readiness problems printed alongside existing inventory and coverage findings.
   - Add or extend package scripts only if a direct debugging entrypoint is useful, e.g. `verify:functional`.

3. Update release-readiness evidence without finalization:
   - Add `.planning/v0.17.1-FUNCTIONAL-COVERAGE-READINESS.md` or an equivalent non-archive evidence file.
   - Update `README.md` and `CHANGELOG.md` only with scoped wording that explains functional confidence and the local readiness gate.
   - Do not touch `.planning/milestones/`, create tags, publish packages, or run milestone lifecycle commands.

## Validation Architecture

Phase 88 validation should prove:

- The functional readiness module is covered by focused fixture tests.
- `scripts/verify-gates.ts` aggregates readiness findings with existing gate findings.
- `bun run coverage` remains the canonical report producer.
- `bun run verify:gates` rejects must-fix readiness gaps after a fresh coverage report.
- `bun run verify` remains the local umbrella command and includes the readiness gate through `verify:gates`.
- Release-readiness docs explicitly separate green suite, covered source, functional confidence, accepted gaps, deferred external-environment coverage, and must-fix-before-release gaps.

## Research Constraints

- Use targeted sentinel gates unless numeric thresholds prove more stable and useful during execution.
- Do not add CI workflows or threshold policy.
- Do not commit raw `.coverage/` artifacts.
- Do not plan or execute milestone archive, tag, publish, cleanup, or `$gsd-complete-milestone`.
