**Part A — Delegation shim in `src/index.ts` (per Research §Pattern 3).**

Add an env-gated delegation block at the **very top** of `src/index.ts`, before the shebang's imports. This is the ONLY modification to production source in the entire phase. The block must:

1. Check `process.env.GS_COVERAGE_SRC`. If unset or empty, the block is a complete no-op — execution falls through to the normal Commander-based CLI below.
2. If set, **validate the path is within the project root** (per Research §Security Domain / T-83-04):
   ```typescript
   import { resolve, relative } from "path"
   const _gsCovSrc = process.env.GS_COVERAGE_SRC
   if (_gsCovSrc) {
     const _gsResolved = resolve(_gsCovSrc)
     const _gsRel = relative(resolve(import.meta.dir, ".."), _gsResolved)
     if (_gsRel.startsWith("..") || _gsRel.startsWith("/")) {
       console.error("GS_COVERAGE_SRC points outside the project root — aborting.")
       process.exit(1)
     }
     // Delegate to instrumented source
     await import(`${_gsResolved}/index.ts`)
     // The instrumented index.ts runs the full CLI and sets up exit handlers.
     // Do NOT call process.exit() here — let the instrumented code's exit handlers fire.
   }
   ```
3. Wrap the rest of the existing `src/index.ts` in an `else` block (or guard with `if (!_gsCovSrc) { ... }`) so the normal CLI path only runs when coverage delegation is not active. The cleanest approach: wrap everything after the delegation block in `if (!process.env.GS_COVERAGE_SRC) { <existing code> }`.

**CRITICAL:** The delegation shim must NOT call `process.exit(0)` after the dynamic import. The reason: the instrumented `index.ts` has a `process.on("exit")` shard writer that needs to fire. If we call `process.exit()` from the outer file, the exit handler fires but we need the instrumented code's Commander `process.exit()` to be what triggers the exit chain. Let the instrumented index.ts control the exit.

**CRITICAL:** The `resolve` and `relative` imports at the top are only used by the shim. Keep them as the very first non-shebang lines. The existing imports (`Command`, `basename`, etc.) go inside the else/guard block.

**Part B — Full implementation of `scripts/coverage-preload.ts`.**

Replace the stub from Plan 01 with the full Bun plugin (per Research §Pattern 4 + Q1 resolution):

```typescript
import { plugin } from "bun"
import { resolve, join } from "path"
import { writeFileSync } from "fs"

const instSrc = process.env.GS_COVERAGE_SRC_INST
const shardDir = process.env.GS_COVERAGE_SHARD_DIR

if (instSrc) {
  const resolvedInst = resolve(instSrc)

  plugin({
    name: "coverage-src-redirect",
    setup(build) {
      // Intercept @/* path alias imports and redirect to instrumented source
      build.onResolve({ filter: /^@\// }, (args) => {
        // Guard: don't re-redirect if already pointing to instrumented path
        if (args.importer?.includes(".coverage/src-inst")) {
          // Resolve relative to the instrumented tree instead
          const relative = args.path.replace(/^@\//, "")
          return { path: `${resolvedInst}/${relative}.ts` }
        }
        const relative = args.path.replace(/^@\//, "")
        return { path: `${resolvedInst}/${relative}.ts` }
      })
    },
  })
}

// Shard writer for unit test processes (per Q1 resolution).
// Unit tests don't go through src/index.ts, so the shard writer appended
// to .coverage/src-inst/index.ts never fires. This exit handler writes
// the __coverage__ data collected from instrumented @/* imports.
if (shardDir) {
  process.on("exit", () => {
    const cov = (globalThis as any).__coverage__
    if (!cov || Object.keys(cov).length === 0) return
    const shardPath = join(shardDir, `unit-${process.pid}-${Date.now()}-${Math.random().toString(36).slice(2)}.json`)
    writeFileSync(shardPath, JSON.stringify(cov), { flush: true })
  })
}
```

The plugin redirects all `@/lib/...`, `@/commands/...`, etc. imports to the instrumented copy in `.coverage/src-inst/`. Since unit tests use `@/*` aliases (per CLAUDE.md: "The `@/*` path alias resolves to `./src/*` but is test-only"), this makes unit test imports hit instrumented modules.

**Edge case per Research §Pitfall 4:** The instrumented files themselves contain `@/*` imports. The plugin must handle recursive resolution — when the importer is already inside `.coverage/src-inst/`, resolve `@/*` relative to the instrumented tree. The guard above handles this.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `GS_COVERAGE_SRC` env var → `src/index.ts` dynamic import | An arbitrary path from the environment crosses into a dynamic `import()` call in production source |
| Subprocess `__coverage__` shard → parent process | Coverage data written by instrumented subprocess is read by the coverage runner |
| `GS_COVERAGE_SRC_INST` env var → Bun plugin import redirect | An env var controls where `@/*` imports resolve during unit test coverage |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-83-03 | T (Tampering) | `src/index.ts` delegation shim | mitigate | Validate `GS_COVERAGE_SRC` resolves to a path within the project root using `resolve()` + `relative()` check. If path starts with `..` or `/`, abort with error. Prevents arbitrary-path code execution from env var injection. |
| T-83-04 | S (Spoofing) | `scripts/coverage-preload.ts` | accept | The preload plugin only activates when `GS_COVERAGE_SRC_INST` is set, which only happens inside `coverage-runner.ts`. If a malicious actor sets this env var, the worst case is unit test imports resolve to a different directory — no production impact since this is a test-only preload. |
| T-83-05 | D (Denial of Service) | Shard writer in instrumented index.ts | accept | The shard writer runs in `process.on('exit')` and writes one JSON file per subprocess. Bounded by test count (~30-50 integ files). No amplification vector. |
</threat_model>
