  <files>src/tui/dashboard/App.tsx</files>
  <action>
The root cause of tab switching freeze (UAT tests 2 and 3) is that the templates and repos Match branches on lines 536-550 use direct per-tab signal getters (e.g., `tabCursor.templates[0]()`) instead of the reactive memo accessors (`cursor()`, `filter()`, `filtering()`). When SolidJS Switch/Match swaps branches, the new branch mounts but doesn't detect these direct reads as dependency changes, so it never repaints. The help overlay toggle forces a full re-render which is why it "unsticks" the UI.

The workspaces branch (lines 527-534) already correctly uses `cursor()`, `filter()`, `filtering()`. Apply the same pattern to templates and repos branches.

**In the LIST PANE Switch/Match (around lines 526-552):**

Change the templates Match branch from:
```tsx
<TemplateList
  entries={filteredTemplates()}
  cursor={tabCursor.templates[0]()}
  filter={tabFiltering.templates[0]() ? tabFilter.templates[0]() : ""}
  height={listHeight()}
/>
```
To:
```tsx
<TemplateList
  entries={filteredTemplates()}
  cursor={cursor()}
  filter={filtering() ? filter() : ""}
  height={listHeight()}
/>
```

Change the repos Match branch from:
```tsx
<RepoList
  entries={filteredRepos()}
  cursor={tabCursor.repos[0]()}
  filter={tabFiltering.repos[0]() ? tabFilter.repos[0]() : ""}
  height={listHeight()}
/>
```
To:
```tsx
<RepoList
  entries={filteredRepos()}
  cursor={cursor()}
  filter={filtering() ? filter() : ""}
  height={listHeight()}
/>
```

This works because the `cursor()`, `filter()`, and `filtering()` memos already dereference through `tab()` (lines 68-73). When `tab()` changes, these memos re-evaluate to the correct per-tab signal value, and SolidJS properly tracks them as reactive dependencies within the Switch/Match rendering context.

Do NOT change the DETAIL PANE Switch/Match (lines 564-578) — those use `currentEntry()`, `currentTemplate()`, `currentRepo()` which are already memo-based and correctly reactive.

Do NOT change the `filteredEntries`, `filteredTemplates`, `filteredRepos` memos — they correctly use direct per-tab signal reads because they are App-level memos (not inside Match branches).

<task type="auto">
  <name>Task 2: Fix rename remnants — reset view to list after rename completes</name>
  <files>src/tui/dashboard/App.tsx</files>
  <action>
The root cause of rename remnants (UAT test 8) is in `handleInlineInputConfirm()` (line 296). After a rename completes, the function calls `setProgressDone(true)` then `reload()` on lines 311-312, but never calls `setView({ view: "list" })`. The view stays on 'progress' until the user presses a key (which triggers lines 423-425 in the progress handler). During this stale transition, InlineInput text artifacts persist on screen.

**Fix:** In the rename branch of `handleInlineInputConfirm()`, add `setView({ view: "list" })` after `reload()`.

Change lines 310-313 from:
```typescript
if (!result.ok) setProgressLines(prev => [...prev, `ERROR: ${result.error}`])
setProgressDone(true)
reload()
return
```
To:
```typescript
if (!result.ok) setProgressLines(prev => [...prev, `ERROR: ${result.error}`])
setProgressDone(true)
reload()
setView({ view: "list" })
return
```

This ensures the view immediately transitions back to 'list' after the rename operation and data reload complete, matching the pattern already used by `clone-template` (line 324) and `handleInlineInputCancel` (line 332).

Note: This means the user will NOT see intermediate progress lines for rename. This is acceptable because rename is near-instant (a file rename operation). If the rename fails, the error was already appended to progressLines but will not be visible since we skip the progress view. If error visibility is desired, a future enhancement could show a transient error toast instead.

Alternative (if error visibility during rename is important): Instead of skipping progress entirely, keep the progress view for errors only:
```typescript
if (!result.ok) {
  setProgressLines(prev => [...prev, `ERROR: ${result.error}`])
  setProgressDone(true)
  // Stay on progress view so user sees the error
} else {
  setProgressDone(true)
  reload()
  setView({ view: "list" })
}
```
Use this alternative approach — it preserves error visibility while fixing the clean-completion path.
