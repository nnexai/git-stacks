# Phase 78.1: Turn Capability Enums into TypeScript Interfaces — Context

**Gathered:** 2026-04-06
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the `Capability` string union and `ReadonlySet<Capability>` field on the `Integration` interface (introduced in Phase 76) with proper narrow TypeScript interfaces, where method presence on a plugin **is** the capability declaration. Remove the Capabilities column and `capabilities` JSON field from `git-stacks integration list`. This phase undoes the user-facing surface added by Phase 76 while preserving the underlying goal (gated dispatch in the runner) via stronger types instead of a parallel Set.

Out of scope: changing integration execution order, modifying the plugin registration pattern, removing optional methods themselves, or introducing third-party plugin support.

</domain>

<decisions>
## Implementation Decisions

### Interface shape
- **D-01:** Drop `capabilities: ReadonlySet<Capability>` from the base `Integration` interface entirely. Drop the `Capability` type alias.
- **D-02:** Decompose optional behaviors into narrow structural interfaces colocated in `src/lib/integrations/types.ts`. One interface per gated capability:
  - `Generates { generate(ctx: IntegrationContext): string | null }`
  - `Cleans { cleanup(ctx: IntegrationContext): Promise<void> }`
  - `HasCommands { commands(parent: Command): void }`
  - `HasConfigExample { configExample: string }`
  - `WindowDetecting { windowDetector: WindowDetector }`
  - `Conditional { applies(workspace: Workspace): boolean }`
- **D-03:** Base `Integration` keeps only the always-present fields: `id`, `label`, `hint`, `enabledByDefault`, `order`, `isEnabled`, `configurePrompt`, `open`. The optional method declarations (`generate?`, `cleanup?`, `commands?`, `configExample?`, `windowDetector?`, `applies?`) are removed from the base interface — they live exclusively on the narrow interfaces.
- **D-04:** Plugins compose via inline intersection at the export site. Example: `export const vscodeIntegration: Integration & Generates & HasCommands & HasConfigExample = { ... }`. Type-checked at the declaration site, no per-plugin named wrapper interfaces, no registry-level union.

### Runner gating mechanism
- **D-05:** `runner.ts` gates calls via type predicate helpers, not `'method' in i`, not non-null assertions, not optional chaining. One predicate per narrow interface:
  - `isGenerator(i: Integration): i is Integration & Generates`
  - `isCleaner(i: Integration): i is Integration & Cleans`
  - `isConditional(i: Integration): i is Integration & Conditional`
  - `isWindowDetecting(i: Integration): i is Integration & WindowDetecting`
  - (predicates for `HasCommands` / `HasConfigExample` only if the runner needs them — currently CLI-only, so probably not)
- **D-06:** Predicate implementation uses `typeof (i as Partial<X>).method === 'function'` (or property presence check for `windowDetector` / `configExample`). Single source of truth: method/property presence.
- **D-07:** Predicates live in `src/lib/integrations/types.ts` colocated with the interfaces they guard. No new `predicates.ts` file. No inlining in `runner.ts`.
- **D-08:** Runner reads as `if (isGenerator(integration)) { const path = integration.generate(ctx); ... }` — no `!` non-null assertions anywhere in `runner.ts` after this phase. The Phase 76 D-04 non-null-assertion contract is replaced.

### CLI `integration list` output
- **D-09:** Remove the `Capabilities` column from the human-readable table in `src/commands/integration.ts`. Drop the column header, the column data, and the divider.
- **D-10:** Remove the `capabilities` field from `--json` output. The JSON shape becomes `{ id, label, enabled, configured }` per row.
- **D-11:** Delete the `TAG_MAP` constant in `src/commands/integration.ts` (no longer referenced).
- **D-12:** No replacement field is added. The mental model "capabilities are a first-class introspectable thing" is removed entirely from the CLI surface — they remain a compile-time-only concept.

### Documentation & requirements cleanup
- **D-13:** `REQUIREMENTS.md` ENGN-07/08/09 are rewritten in place to reflect the new shape (e.g., "Each integration plugin declares its optional behaviors via narrow TypeScript interfaces composed at export"). They remain marked Complete, with Phase 76 → Phase 78.1 noted in the phase column. The audit trail is preserved without leaving stale text.
- **D-14:** Phase 76 artifacts (`76-CONTEXT.md`, `76-SUMMARY.md`, `76-RESEARCH.md`, `76-VERIFICATION.md`, `76-VALIDATION.md`, `76-SECURITY.md`, `76-DISCUSSION-LOG.md`) are NOT modified — left as historical record of the original design.
- **D-15:** STATE.md Phase 76 decision lines (the Phase 76 D-01 through D-10 entries in Accumulated Context) are NOT edited. A new Phase 78.1 decisions section is appended noting the supersession of Phase 76 D-01 through D-07.
- **D-16:** **CHANGELOG constraint for Phase 79 (release-prep):** Neither the Phase 76 capabilities column nor the Phase 78.1 removal land in the v0.17.0 CHANGELOG. Net user-visible change is zero — the feature was added and removed within the same unreleased milestone. Phase 79's release-prep planning must NOT mention "capabilities column", "integration capabilities", or "Phase 78.1" in CHANGELOG.md. The internal architectural change (narrow interface contracts replacing Set declaration) may be noted under an "Internal" section if Phase 79 has one, but is not user-facing copy.

### Test updates
- **D-17:** Tests that currently set `capabilities: new Set([...])` on fake integrations must be updated to drop that field and instead include/omit the relevant method to match what the test exercises:
  - `tests/lib/integrations/runner.test.ts` (multiple fakes — drop `capabilities`, ensure each fake has `generate` / lacks it / has `applies` / etc. as needed)
  - `tests/lib/integrations/window-detector.test.ts` (drop `capabilities`, keep/drop `windowDetector` per fake's intent)
  - `tests/lib/integration-commands.test.ts` lines 285-314: the entire "Phase 76-02: capabilities column" test block is deleted (the column no longer exists)
  - `tests/tui/workspace-wizard.test.ts` (drop `capabilities` from fakes)
- **D-18:** No backwards-compat shim. Phase 76 D-09 still applies — all 10 plugins are first-party, no third-party plugins exist, v0.17.0 hasn't shipped, so no `get capabilities()` getter or deprecation period is needed.

### Claude's Discretion
- Exact predicate function names (`isGenerator` vs `hasGenerate` vs `canGenerate`) — pick whichever reads cleanest in `runner.ts`
- Whether predicates use `typeof === 'function'` or `'method' in i` internally — both are sound for the type narrowing
- Order of properties in plugin object literals (the Phase 76 PRs put `capabilities:` after `order:`; after this phase, plugin objects just delete that line and keep existing key order)
- Whether to add JSDoc on the narrow interfaces explaining the "method presence is capability" contract
- Test reorganization beyond what D-17 mandates (e.g., consolidating fake-integration helpers if they grow)

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/lib/integrations/types.ts` already has `IntegrationContext`, `WindowDetector`, `ArtifactBag`, `IntegrationArtifact` types that the narrow interfaces will reference. No new supporting types needed.
- `resolveEnabled` and `resolveEnabledGlobally` helpers (lines 112-132 of types.ts) are independent of capabilities and stay unchanged.
- `Workspace`, `Command`, `IntegrationContext` imports already exist in types.ts — narrow interface signatures can reference them without new imports.

### Established Patterns
- Project convention is structural interfaces over discriminated unions (per CLAUDE.md "Conventions": `interface` for contracts, `type` for data structures). The narrow capability interfaces fit this convention exactly.
- Type predicates (`x is Y`) are not currently used in the codebase but are idiomatic TypeScript and align with the strict-mode posture.
- Plugin objects are exported with explicit type annotations (`export const xIntegration: Integration = {...}`), so adding intersection types is a one-line annotation change per plugin.

### Integration Points
- `src/lib/integrations/types.ts`: All structural changes land here — drop `Capability`, drop `capabilities` field, add 6 narrow interfaces, add 4 type predicates.
- `src/lib/integrations/runner.ts`: All three runner functions switch from `.has(...)` + `!` to predicate calls. No structural changes to control flow.
- `src/commands/integration.ts`: One column dropped from human output, one field dropped from JSON output, one constant deleted.
- All 10 plugin files: One-line `capabilities:` field removal + intersection type added to export annotation. No method body changes.

### Constraints / risks
- Phase 76 introduced this concept very recently (decisions D-01 through D-10 in STATE.md). Anyone reading STATE.md or REQUIREMENTS.md needs to understand why we're undoing it — Phase 78.1 STATE.md decisions section should explain the motivation briefly.
- The runner currently uses `i.capabilities.has('applies')` to gate window-detection filtering (line 27 of runner.ts). The detector list filter is the trickiest call site — make sure the type predicate composition still works there.
- v0.17.0 has not shipped — no external user has seen the capabilities column. This is the entire reason D-16 exists (no CHANGELOG mention needed).

</code_context>
