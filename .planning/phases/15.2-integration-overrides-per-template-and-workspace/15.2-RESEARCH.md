# Phase 15.2: Integration Overrides per Template and Workspace - Research

**Researched:** 2026-03-21
**Domain:** CLI wizard augmentation, TUI detail pane display, config cascade read-back
**Confidence:** HIGH

## Summary

This phase adds interactive UX to a config cascade that already exists in code. The schema
supports `template.integrations` and `workspace.settings.integrations`; `resolveEnabled()`
already implements the full workspace → global → default cascade. The work is purely UX:
prompts in CLI wizards and read-only display in TUI detail panes.

The integration system is small and well-bounded: four integrations, each implementing
`configurePrompt()`. The global config wizard in `commands/config.ts` already contains the
canonical pattern for multiselect → configure loop. That pattern is directly reusable in the
template and workspace wizards with one addition: a yes/no guard and a cascade-aware pre-selection.

The new `git-stacks edit <workspace>` command parallels `template edit` and reuses workspace-ops
patterns. Detail pane additions are pure SolidJS `<text>` additions — no new components needed.

**Primary recommendation:** Extract the multiselect-and-configure loop from `commands/config.ts`
into a shared helper, call it from template-wizard, workspace-wizard, workspace-clone, and the
new `edit` command. Detail panes read the cascade state directly using `resolveEnabled()` and
`resolveEnabledGlobally()`.

### Pattern 1: Shared Integration Override Helper

**What:** A shared async function that encapsulates the yes/no guard + multiselect + configure-each
loop, returning `Record<string, unknown> | undefined` (undefined = user said no to override).

**When to use:** Every CLI wizard entry point that needs an integration override prompt.

**Signature:**
```typescript
// src/lib/integrations/wizard-helpers.ts

/**
 * Prompts "Override integrations?", then runs multiselect + configurePrompt() for each.
 * Returns the resulting integrations record, or undefined if user declined/cancelled.
 *
 * @param initialEnabledIds  IDs pre-selected in multiselect (cascade-resolved by caller)
 * @param currentConfigs     Current raw config per integration (for configurePrompt pre-fill)
 */
export async function promptIntegrationOverrides(
  initialEnabledIds: string[],
  currentConfigs: Record<string, Record<string, unknown>>
): Promise<Record<string, unknown> | undefined>
```

This function:
1. Calls `p.confirm({ message: "Override integrations?", initialValue: false })`
2. Returns `undefined` if no or cancelled
3. Calls `p.multiselect` with `initialValues: initialEnabledIds`
4. For each selected integration, calls `integration.configurePrompt(currentConfigs[id] ?? {})`
5. For each deselected integration, stores `{ ...currentConfigs[id], enabled: false }` (preserves
   extra config in case user re-enables later — same pattern as `commands/config.ts` line 45)
6. Returns the resulting record

### Pattern 2: Cascade-Aware Pre-Selection

**What:** Computing `initialEnabledIds` correctly per call site, so pre-selection matches
the cascade source the workspace will inherit from.

**Template new/edit:** Pre-select from global config state.
```typescript
// In template-wizard.ts
import { resolveEnabledGlobally } from "../lib/integrations"
const config = readGlobalConfig()
const initialEnabledIds = integrations
  .filter(i => resolveEnabledGlobally(i.id, i.enabledByDefault, config))
  .map(i => i.id)
const currentConfigs = Object.fromEntries(
  integrations.map(i => [i.id, (config.integrations[i.id] ?? {}) as Record<string, unknown>])
)
const overrideResult = await promptIntegrationOverrides(initialEnabledIds, currentConfigs)
```

**Workspace new (template-based):** Pre-select from template's integrations if set, else global.
```typescript
// Pre-selection starts from template's overrides (D-06)
const templateIntegrations = template.integrations ?? {}
const initialEnabledIds = integrations.filter(i => {
  const override = templateIntegrations[i.id]
  if (override && typeof override === "object" && "enabled" in (override as object)) {
    return (override as { enabled: boolean }).enabled
  }
  return resolveEnabledGlobally(i.id, i.enabledByDefault, config)
}).map(i => i.id)
```

**Workspace new (ad-hoc):** Pre-select from global config (no template).

**Workspace clone:** Pre-select from cloned workspace's `settings.integrations` if set, else
template integrations, else global.

### Pattern 3: Cascade Source Determination for Display

**What:** Determining the source label (`[global]`, `[template]`, `[workspace]`, `[skipped]`)
for each integration in TUI detail panes.

**For workspace detail pane** — needs `Workspace`, `Template | undefined`, and `GlobalConfig`:

```typescript
// Source: local analysis of src/lib/integrations/types.ts resolveEnabled()
type IntegrationSource = "workspace" | "template" | "global" | "skipped"

function resolveIntegrationSource(
  integration: Integration,
  workspace: Workspace,
  templateIntegrations: Record<string, unknown> | undefined,
  globalConfig: GlobalConfig
): { enabled: boolean; source: IntegrationSource; detail: string } {
  // applies() check
  if (integration.applies && !integration.applies(workspace)) {
    return { enabled: false, source: "skipped", detail: "no matching repos" }
  }
  // workspace override
  const wsRaw = workspace.settings?.integrations?.[integration.id]
  if (wsRaw && typeof wsRaw === "object" && "enabled" in (wsRaw as object)) {
    return { enabled: (wsRaw as any).enabled, source: "workspace", detail: "" }
  }
  // template override (from snapshot stored in workspace.settings.integrations at creation)
  // NOTE: workspace.settings.integrations IS the template snapshot — there is no separate field
  // The distinction between template-sourced and user-set workspace overrides is not preserved
  // in the schema; label as [workspace] when ws settings exist, [template] is not distinguishable
  // at this level without a template field inspection.
  // template integrations on the template object itself (for TemplateDetail)
  if (templateIntegrations) {
    const tplRaw = templateIntegrations[integration.id]
    if (tplRaw && typeof tplRaw === "object" && "enabled" in (tplRaw as object)) {
      return { enabled: (tplRaw as any).enabled, source: "template", detail: "" }
    }
  }
  // global
  const globalEnabled = resolveEnabledGlobally(integration.id, integration.enabledByDefault, globalConfig)
  return { enabled: globalEnabled, source: "global", detail: "" }
}
```

**Key insight about workspace source annotation:** `workspace.settings.integrations` holds
the current per-workspace override (which may be a snapshot of the template's integrations from
creation, or user overrides from `edit`). There is no runtime way to distinguish "came from
template" vs "user edited" — the YAML just has a value. For workspace detail pane purposes,
if `settings.integrations[id]` is set, report `[workspace]`; otherwise walk to template then
global. For `TemplateDetail`, read `template.integrations[id]` directly.

### Pattern 4: Template Edit Integration Handling

**What:** `runTemplateEdit` uses a menu-based `action → done` pattern (not a linear flow).
Integration overrides should be one of the menu options, same as "Repos" and "Description".

**When to use:** Editing an existing template's integration overrides post-creation.

```typescript
// In runTemplateEdit, add to the select options:
{ value: "integrations", label: "Integrations", hint: "override per-integration settings" }

// Handler:
if (action === "integrations") {
  const currentOverrides = (template.integrations ?? {}) as Record<string, Record<string, unknown>>
  const initialEnabledIds = integrations.filter(i => {
    const override = currentOverrides[i.id]
    if (override && "enabled" in override) return override.enabled
    return resolveEnabledGlobally(i.id, i.enabledByDefault, config)
  }).map(i => i.id)
  const result = await promptIntegrationOverrides(initialEnabledIds, currentOverrides)
  if (result !== undefined) {
    template.integrations = result
    writeTemplate(template)
    p.outro(`Template '${name}' updated.`)
  }
  return
}
```

### Pattern 6: TUI Detail Pane Integration Display

**What:** Adding an "Integrations:" section to `WorkspaceDetail.tsx` and `TemplateDetail.tsx`.
These components receive only their domain object (Workspace, Template) — they need global
config access to determine cascade source.

**How to pass global config:** `readGlobalConfig()` can be called inline in the component
(it's a sync read). Alternatively, pass it as a prop from App.tsx. Reading inline is simpler
and consistent with how `commands/config.ts` uses it — no reactive signal needed since the
config doesn't change while the TUI is running.

**Display format (from CONTEXT.md specifics):**
```
Integrations:
  ✓ vscode  (cmd: code-insiders)  [template]
  ✗ tmux                          [workspace]
  ✓ cmux                          [global]
  - intellij                      [skipped: no java repos]
```

In OpenTUI SolidJS: one `<text>` per integration row. The label padding should be consistent.

### Anti-Patterns to Avoid

- **Duplicating the multiselect+configure loop:** Do not copy the `commands/config.ts` pattern
  into template-wizard, workspace-wizard, workspace-clone, and edit separately. Extract once
  into `wizard-helpers.ts`.
- **Reactive global config reads in TUI:** Do not create a signal for global config — it
  doesn't change during a TUI session. A one-time `readGlobalConfig()` call at render or
  passed as prop is correct.
- **Storing empty `integrations: {}` in YAML:** D-04 says store nothing if user said no.
  Check `Object.keys(result).length > 0` before assigning.
- **Modifying `resolveEnabled()` in types.ts:** The cascade is correct as-is. Display logic
  needs its own source-tracing function that does not modify the core resolver.
- **Using `nested <text>` elements in OpenTUI:** Memory file `feedback_opentui_no_nested_text.md`
  confirms OpenTUI crashes on nested `<text>`. Use sibling `<text>` inside `<box flexDirection="row">`
  for multi-part lines.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Integration enabled/disabled cascade | Custom cascade logic in display | `resolveEnabled()` / `resolveEnabledGlobally()` from `types.ts` | Already handles ws → global → default |
| Integration config collection | Custom prompts per integration | `integration.configurePrompt(current)` | Already implemented on every integration |
| Initial enabled state for multiselect | Inline enable checks | `resolveEnabledGlobally()` + template.integrations loop | Avoids reimplementing the cascade |
| YAML validation on edit command | Custom validator | `WorkspaceSchema.parse(parse(raw))` from `editWorkspaceYaml()` pattern | Already tested and correct |
| Extracting display config values | Custom config parsers per integration | Each integration's Zod configSchema on `integration.id` config | Each integration already knows its shape |

**Key insight:** Every integration already has `configurePrompt()` and the cascade functions
already handle all levels. Display code only needs to call what exists.

## Common Pitfalls

### Pitfall 1: workspace.settings.integrations vs template.integrations confusion

**What goes wrong:** `workspace.settings.integrations` holds the workspace-level snapshot
(originally a copy of `template.integrations` at creation, but may have been overridden). When
displaying workspace detail, reading `ws.settings?.integrations` is the workspace-level source.
But `template.integrations` (read from the template file) is the template-level source.

**Why it happens:** The schema stores both under `integrations` but at different nesting levels:
- `template.integrations` — the template's direct field
- `workspace.settings.integrations` — the workspace override (nested under `settings`)

**How to avoid:** Display functions receive both the workspace AND the template object (loaded
by name via `readTemplate(ws.template)` if `ws.template` is set). Walk: workspace settings first,
then template file's integrations, then global.

**Warning signs:** Source annotations all showing `[global]` when template has integrations set.

### Pitfall 2: `applies()` check visibility in display

**What goes wrong:** IntelliJ is `enabledByDefault: true` but its `applies()` returns false
for non-Java workspaces. Displaying `✓ intellij [global]` is misleading when it will never run.

**Why it happens:** `isEnabled()` only checks the enabled flag, not `applies()`.

**How to avoid:** In display functions, check `applies()` before checking enabled state.
If `applies` exists and returns false: show `- intellij [skipped: no java repos]` regardless
of the enabled flag. The `applies()` check takes precedence.

**Warning signs:** IntelliJ showing as "enabled" on a workspace with no Java repos.

### Pitfall 3: Template edit menu vs linear flow mismatch

**What goes wrong:** `runTemplateNew` is a linear flow (all steps in sequence).
`runTemplateEdit` is a menu-select flow (choose what to edit). Adding integration override
as a step to template edit requires a menu option, not a final linear step.

**Why it happens:** The two functions have different UX structures.

**How to avoid:** Add "Integrations" as a menu option in `runTemplateEdit`'s `p.select`.
For `runTemplateNew`, add the integration override prompt as the last step before `writeTemplate`.

**Warning signs:** Integration override only appearing in `new` but not in `edit`, or edit
prompting for integrations unconditionally (before the menu choice).

### Pitfall 4: config module mock in integration tests must include all exports

**What goes wrong:** Bun throws `Export not found` on first use if the config mock is
missing exports that other modules in the same test process have already imported.

**Why it happens:** Phase 15 STATE.md note: "workspace-ops mock must include all exports
(getWorkspaceStatus, editWorkspaceYaml, mergeEnv, writeEnvFiles, etc.) or Bun throws
'Export not found' on first use in integration tests"

**How to avoid:** Use `mock.module()` with a complete mock object including all exports,
or use inline fixture pattern from `integ-wizard.test.tsx`.

**Warning signs:** Test passes in isolation but fails when run in `bun test tests/`.

### Pitfall 5: `p.confirm` cancel handling

**What goes wrong:** `p.isCancel(addHooks)` pattern from template-wizard returns `true`
when user presses Ctrl+C, but `addHooks === false` when user says no. Both must be handled.

**Why it happens:** `@clack/prompts` returns a `Symbol` for cancel, not `false`.

**How to avoid:** Check `p.isCancel(result)` first, then check the value.
```typescript
const addOverrides = await p.confirm({ message: "Override integrations?", initialValue: false })
if (p.isCancel(addOverrides)) cancel()
if (!addOverrides) return undefined
```

**Warning signs:** Ctrl+C during integration prompt not exiting cleanly.

### Pitfall 6: OpenTUI nested `<text>` crash

**What goes wrong:** Attempting to render `<text fg="green">✓ {<text fg="gray">...</text>}</text>`
crashes OpenTUI.

**Why it happens:** Memory file `feedback_opentui_no_nested_text.md` confirms this.

**How to avoid:** Use sibling `<text>` elements inside `<box flexDirection="row">`.
```tsx
<box flexDirection="row">
  <text fg={enabled ? "green" : "red"}>{icon} {id.padEnd(10)}</text>
  <text fg="gray">  [{source}]</text>
</box>
```

**Warning signs:** TUI render error or blank output when adding integration display rows.

export function editWorkspaceYaml(name: string): {
  path: string
  validate: () => { ok: boolean; error?: string }
} {
  const path = workspacePath(name)
  return {
    path,
    validate: () => {
      try {
        const raw = readFileSync(path, "utf-8")
        WorkspaceSchema.parse(parse(raw))
        return { ok: true }
      } catch (err) {
        return { ok: false, error: String(err) }
      }
    },
  }
}
