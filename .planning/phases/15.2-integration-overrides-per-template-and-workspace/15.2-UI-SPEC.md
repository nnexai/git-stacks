## Design System

| Property | Value |

## Spacing Scale

Terminal spacing is measured in character cells, not pixels. The established pattern in the
codebase uses 2-space indentation blocks:

| Level | Characters | Usage |
|-------|-----------|-------|
| L0 | 0 chars | Section header labels in detail pane (`<text fg="white">`) |
| L1 | 2 chars (`"  "`) | Section headers and top-level fields in detail pane |
| L2 | 4 chars (`"    "`) | List items under a section (repos, integration rows) |
| blank | `<text>{""}` | Visual separator between sections |

**Source:** Extracted from WorkspaceDetail.tsx and TemplateDetail.tsx existing patterns.

Exceptions: none. Every detail pane field in the codebase uses exactly this 2-space/4-space rule.

## Typography

Terminal TUI has no configurable font sizes — all text is monospace at the terminal's cell size.
Typography is expressed through foreground color and character choice.

| Role | fg color | Usage |
|------|----------|-------|
| Section header | `"white"` | Section labels (`  Branch:`, `  Repos:`, `  Integrations:`) |
| Primary value | `"white"` | Values on the same line as a header, or standalone values |
| Subordinate / meta | `"gray"` | Descriptions, secondary info, source annotations (`[global]`) |
| Enabled / success | `"green"` | Enabled integrations (`✓`), clean repos |
| Disabled / error | `"red"` | Disabled integrations (`✗`), missing repos, errors |
| Warning / dirty | `"yellow"` | Dirty repos, age timestamps on stale messages |
| Skipped / inactive | `"gray"` | Skipped integrations (`-`), pending states |

**Source:** Extracted from WorkspaceDetail.tsx lines 39-43, 50-53, TemplateDetail.tsx lines 43-48.

## Color

Terminal color is fg-only (no background colors used in detail panes).

| Role | OpenTUI fg | Usage |
|------|-----------|-------|
| Dominant (surface) | default terminal bg | All text rows — no explicit background |
| Section headers | `"white"` | All `  Label:` lines |
| Enabled state | `"green"` | `✓` icon + integration id when enabled |
| Disabled state | `"red"` | `✗` icon + integration id when explicitly disabled |
| Skipped / inactive | `"gray"` | `-` icon + integration id when skipped; source annotations |
| Error | `"red"` | Error state copy in Loading/Error show blocks |

Accent (`"green"`) reserved for: enabled integration rows only. No other use of green in
integration display.

Destructive: `"red"` reserved for: disabled integration indicator (`✗`) and error states.
No destructive action in this phase (no delete flows).

## Interaction Contract

### CLI Wizard Flow (template new / template edit / workspace new / workspace clone / edit)

1. Existing wizard steps run first (unchanged).
2. `p.confirm` appears: "Override integrations?" with initialValue `false`.
3. If user answers `false` or presses Ctrl+C: skip — no `integrations` key stored in YAML.
4. If user answers `true`: `p.multiselect` appears pre-populated from cascade source.
5. For each selected integration: `integration.configurePrompt(currentConfig)` runs in order.
6. Result written to YAML only if `Object.keys(result).length > 0`.

**Template edit menu:** Integration override is one menu item alongside Repos, Description.
Selecting it triggers the same yes/no → multiselect → configure flow. Returns to menu after.

**`git-stacks edit <name>` command:** Menu-select flow paralleling `template edit`. Minimum
viable: "Override integrations?" as the first (and only required) menu option. May include
description editing if the executor deems it low-cost.

### TUI Detail Pane Behavior

- Integration section renders immediately when detail pane mounts — no async loading.
- `readGlobalConfig()` called once at render time (synchronous, no signal needed).
- `applies()` check runs first; if false: render skipped row regardless of enabled flag.
- No interactive elements — the pane is read-only. YAML editor (`editWorkspaceYaml`) is the
  existing edit path for post-creation changes.

### Keyboard Isolation (TUI)

No new keyboard handlers introduced in this phase. Detail panes are read-only. The existing
`useKeyboard` guard in App.tsx is not modified.

## Implementation Notes for Executor

### OpenTUI nested `<text>` rule (CRITICAL)

OpenTUI crashes on nested `<text>` elements. Integration display rows MUST use sibling `<text>`
inside `<box flexDirection="row">`, not nested text. See project memory
`feedback_opentui_no_nested_text.md`.

Correct pattern:
```tsx
<box flexDirection="row" height={1}>
  <text fg={iconColor}>    {icon}  {id.padEnd(10)}  {configSummary}</text>
  <text fg="gray">  [{source}]</text>
</box>
```

Wrong (crashes):
```tsx
<text fg="green">    ✓  vscode  <text fg="gray">[global]</text></text>
```

### Column alignment

`integration.id.padEnd(10)` aligns all source annotations to the same column (longest current
id is `intellij` at 8 chars; 10 gives 2 chars clearance). Config summary `(key: value)` is
optional — only shown when at least one meaningful config value exists for the integration.

### Blank separator between sections

Use `<text>{""}</text>` to insert a blank line between sections, consistent with existing
WorkspaceDetail.tsx lines 34, 54 and TemplateDetail.tsx line 49.
