# Phase 82.1: Support Commands and Error-Path E2E Coverage - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Prove the remaining non-TUI, non-integration support commands through real CLI subprocess tests, and add representative negative-path coverage for in-scope command families. This phase covers success/output contracts for `config show`, `doctor`, `completion`, `version`, `install --hooks`, `env`, `paths`, `edit --yaml`, `integration list`, and `integration <id> config show/example`, plus a minimal pre-Phase-83 Istanbul subprocess proof.

This phase is about automation-safe support flows and representative failures, not prompt-driving the `config` wizard or interactive `install` selection flows, not expanding into external integration behavior, and not pulling full coverage tooling forward from Phase 83.

</domain>

<decisions>
## Implementation Decisions

### Support command boundary
- **D-01:** Phase 82.1 should stay on automation-safe, non-interactive support flows only; prompt-driven `config` and interactive `install` selection flows stay out of scope.
- **D-02:** Success-path support coverage should target `config show`, `completion`, `version`, `env`, `paths`, `doctor`, `integration list`, `integration <id> config show/example`, flag-driven `install --hooks ...`, and the YAML edit commands.

### YAML edit proof strategy
- **D-03:** `edit --yaml` coverage should use a fake `$EDITOR` script that records the target file and performs a small valid edit so the suite proves launch, correct target selection, and persisted round-trip behavior.
- **D-04:** Validation-failure and editor-error variants belong in the representative failure slice, not in the success-only edit contract path.

### Completion contract
- **D-05:** Completion coverage should use shell-specific invariant assertions against generated output rather than full golden snapshots.
- **D-06:** The same invariant contract should run across bash, zsh, and fish so all shipped shells stay covered.

### Representative failure coverage
- **D-07:** Negative coverage should stay representative: one meaningful failure path per in-scope command family, spanning the roadmap's failure classes without turning into a full cross-product matrix.
- **D-08:** Across the whole phase, those representative failures should cover malformed input, missing entity, missing path, validation failure, dirty repo, environment-sensitive behavior, and permission/error cases.

### Phase 83 handoff proof
- **D-09:** The pre-Phase-83 Istanbul check should be a tiny dedicated proof, separate from the main support-command suites.
- **D-10:** That proof should live as a committed automated smoke/verification artifact in the repo so Phase 83 can build directly on it.

### the agent's Discretion
- Exact test file boundaries across the support command families, as long as the suites stay focused and automation-safe instead of becoming one omnibus support file.
- Exact mapping from command family to representative failure case, as long as the combined phase spans the required failure classes.
- Exact shape and naming of the fake-editor harness and the Istanbul smoke artifact, as long as both stay deterministic and committed in-repo.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `tests/helpers.ts` already provides isolated config, temp directory, and git-environment primitives that can support the new support-command suites without inventing a second harness.
- Existing subprocess tests such as `tests/commands/env.test.ts` and `tests/commands/doctor-json.test.ts` already show the intended `bun run src/index.ts ...` pattern for hermetic command assertions.
- `tests/commands/workspace-edit.test.ts` already exercises lower-level edit helpers, which means Phase 82.1 can add a fake-editor E2E layer instead of redesigning edit behavior.
- `src/commands/install.ts` already exposes a fully flag-driven path (`--hooks`, `--copilot`, `--claude`, `--remove`) that fits the non-interactive scope choice.
- `src/commands/completion.ts` delegates to `src/lib/completion-generator.ts`, so all three shell outputs derive from one command tree and can share the same invariant assertions.

### Established Patterns
- Command-level E2E coverage belongs in `tests/commands/` and runs in isolated per-file subprocesses via the custom runner.
- The shared E2E layer should stay thin: helper plumbing in `tests/helpers.ts`, with assertions remaining local to each test file.
- Support-command coverage should stay non-TUI and non-integration; real external CLI dependencies are only acceptable when explicitly enabled inside fixtures.
- Representative failures belong in this phase, but they should stay focused and command-family-scoped rather than becoming an exhaustive matrix.

### Integration Points
- `tests/helpers.ts` is the natural home for any fake-editor helper, workspace-cwd shaping helper, or permission/error fixture utilities needed across support suites.
- `tests/commands/` is the landing zone for new support-command success suites, negative-path suites, and the dedicated Istanbul smoke artifact.
- `src/commands/config.ts`, `doctor.ts`, `completion.ts`, `install.ts`, `integration.ts`, and `workspace.ts` are the primary support command entrypoints under test.
- Because Phases 80 and 82 are planned but not yet executed on this branch, downstream planning should begin with a prerequisite verification task that confirms the expected harness and command-family surfaces actually exist before Phase 82.1 implementation assumes them.

</code_context>

<specifics>
## Specific Ideas

- Keep the phase automation-safe: prove the support surfaces users can run in scripts and deterministic subprocess tests, not the remaining prompt flows.
- Make `edit --yaml` feel like a real user contract by actually launching a fake editor and persisting a valid mutation.
- Treat bash, zsh, and fish as first-class shells for completion coverage rather than giving zsh/fish only smoke treatment.
- Keep the Phase 83 handoff minimal and clean: one tiny committed Istanbul subprocess proof instead of prematurely building the full coverage toolchain here.

</specifics>

<deferred>
## Deferred Ideas

- Prompt-driven `config` wizard E2E coverage — remains out of scope for this milestone's non-interactive subprocess coverage.
- Interactive `install` workspace/plugin selection coverage — same harness problem as other prompt-driven flows; not part of 82.1.
- Full golden snapshots for completion output — intentionally avoided in favor of invariant checks tied to the live command tree.
- Exhaustive negative-path matrices per command family — beyond the representative-failure scope of this phase.
- General subprocess coverage tooling and report generation — belongs in Phase 83 after the dedicated proof-of-concept lands.

</deferred>
