## Summary

Phase 82.1 should be planned as three distinct slices rather than one omnibus suite: **support-command success/output contracts**, **representative failure coverage**, and a **tiny dedicated Istanbul subprocess proof**. That split matches the locked context, fits the existing `tests/commands/*.test.ts` organization, and makes failures actionable by command family instead of by giant mixed suites.

The strongest reusable pattern is already in the repo: small command-focused subprocess tests built around `Bun.spawnSync(["bun", "run", "src/index.ts", ...])` with isolated config or git env, local assertions, and no second harness runtime. `tests/commands/env.test.ts` and `tests/commands/doctor-json.test.ts` are the clearest support-command examples, while `tests/lib/completion-generator.test.ts` already demonstrates the right completion-testing philosophy: invariant assertions against generated content rather than brittle snapshots.

The biggest planning risk is **prerequisite drift**. The current branch still lacks visible Phase 80 inventory artifacts (`tests/e2e-inventory.ts`) and lacks the planned Phase 81.1.1 non-interactive workspace surface (`--non-interactive` is absent from `src/commands/workspace.ts`). Phase 82 support-command planning should therefore begin with an explicit prerequisite verification task rather than assuming earlier planned outputs already exist on this branch.

## Key Findings

### 1. Best file split is domain-focused, not omnibus

Recommended planning split:

1. **Support read-only contracts** — `config show`, `--version`, and `completion`
2. **Workspace-adjacent support surfaces** — `env`, `paths`, and `edit --yaml`
3. **Doctor, install, and integration config support** — `doctor`, flag-driven `install --hooks`, `integration list`, and `integration <id> config show/example`
4. **Representative failures** — one failure per command family
5. **Istanbul smoke** — one tiny dedicated subprocess proof

This keeps each suite aligned to a user-facing behavior domain and follows the existing focused-suite pattern in `tests/commands/`.

### 2. `edit --yaml` needs a real executable fake editor

`openYamlInEditor()` runs:

```ts
const editor = process.env.VISUAL || process.env.EDITOR || "vi"
const proc = _exec.spawnEditor(editor, path)
```

and `_exec.spawnEditor()` calls:

```ts
Bun.spawn([editor, path], ...)
```

So the fake editor cannot be a shell snippet like `sh -c ...`; it needs to be a **real executable path** that Bun can launch directly. The cleanest plan is one committed stub script under a repo-controlled test-support path, with env-driven modes such as:

- `record` / `mutate-valid`
- `mutate-invalid`
- `fail`

That gives the phase one reusable harness for both success-path edit round-trips and representative validation/editor-error cases.

### 3. Completion should use invariants across all three shells

`src/commands/completion.ts` is intentionally thin and delegates to `generateBash`, `generateZsh`, and `generateFish`. The higher-value assertions already exist in `tests/lib/completion-generator.test.ts`, where the test suite checks:

- command presence
- option presence
- argument-specific completion behavior
- negative inheritance boundaries
- shell-specific structural markers

Phase 82.1 should reuse that philosophy in CLI subprocess form:

- call `git-stacks completion bash|zsh|fish`
- assert shell-specific markers plus current command-tree invariants
- avoid full-file snapshots

This is especially important because the completion output derives from the live Commander tree and will grow over time.

### 4. Doctor can stay local-only if fixtures keep integrations disabled

`src/commands/doctor.ts` only checks forge binaries when the integration is globally enabled:

```ts
if (!resolveEnabledGlobally(integrationId, false, config)) continue
const available = await checkBinary(binary)
```

So baseline doctor tests should use an empty or default-disabled integration config and avoid machine-specific assertions about optional binaries. Separate explicit-fixture tests can enable one integration when the phase wants to prove the gated behavior.

### 5. `completion bash` is the smallest subprocess target for the Istanbul handoff proof

`src/index.ts` skips the git-version gate for the `completion` subcommand:

```ts
const subcommand = process.argv[2]
if (subcommand !== "completion") {
  await checkGitVersion()
}
```

That makes `completion bash` the lowest-fixture subprocess path for the pre-Phase-83 proof. It exercises the real CLI entrypoint, avoids extra workspace/repo fixtures, and is a clean candidate for a tiny instrumented subprocess smoke that emits mergeable Istanbul coverage data.

### 6. Existing tests already show the right subprocess style

`tests/commands/env.test.ts` and `tests/commands/doctor-json.test.ts` both:

- create isolated temp dirs
- point `GIT_STACKS_CONFIG_DIR` at fixture config
- call the real CLI with `bun run src/index.ts ...`
- assert on stdout/stderr/JSON rather than mocking command wiring

Phase 82.1 should extend that exact pattern instead of creating a second E2E subsystem.

### 7. Prerequisite drift is visible and must be planned around

Research confirmed current-branch drift against planned prerequisites:

- `tests/e2e-inventory.ts` is absent
- `src/commands/workspace.ts` has no visible `--non-interactive` support
- no Phase 82 support suites exist yet under `tests/commands/`

This does **not** mean Phase 82.1 should absorb those earlier phases. It means the first planning task should explicitly verify those prerequisites and block early if they are still missing.

## Architecture Patterns

### Pattern 1: Focused support suites
Keep readonly contract checks, workspace-adjacent support surfaces, and doctor/install/integration config coverage in separate suites. This keeps failure localization sharp and matches the existing `tests/commands/` style.

### Pattern 2: Stub external side effects with explicit executable helpers
For `$EDITOR` and similar process-launch surfaces, prefer deterministic committed helper executables over ad hoc shell snippets. That matches Bun's spawn model and keeps failure modes easy to control.

### Pattern 3: Invariant-first generated output assertions
For completion output, assert presence/absence of meaningful command-tree invariants across bash/zsh/fish rather than exact generated text snapshots.

### Pattern 4: Fixture-gated optional binary checks
For doctor, keep baseline suites local-only by leaving integrations disabled, and enable them only in explicit fixtures when the test intends to prove optional binary behavior.

### Pattern 5: Minimal dedicated proof before general tooling
The Istanbul subprocess smoke should stay a tiny, committed artifact focused on one real CLI subprocess invocation. General report generation, merging, and full-suite tooling remain Phase 83 work.

## Command-Specific Notes

- **`config show`**: assert stable sections and key rows, not full snapshots.
- **`--version`**: test the program-level Commander version surface, not a subcommand.
- **`install --hooks`**: avoid prompt paths by running inside a workspace repo so cwd detection succeeds and by passing explicit `--claude` / `--copilot` flags.
- **`integration <id> config example`**: plan for both real-example integrations and fallback-message integrations.
- **`paths`**: likely best paired with workspace-adjacent support surfaces rather than readonly support.
- **`edit --yaml`**: success path should prove valid persisted edits; invalid-YAML and editor-fail branches belong in representative failures.

## Common Pitfalls

### Pitfall 1: Smuggling prompt-driven config/install work into 82.1
The locked context explicitly excludes those prompt branches. Keep the phase on automation-safe paths only.

### Pitfall 2: Treating completion like a snapshot artifact
Because completion is generated from the live command tree, exact snapshots will create unnecessary churn. Favor invariants.

### Pitfall 3: Making doctor assertions machine-specific
Optional integration binary checks depend on fixture enablement. Baseline doctor tests should stay deterministic and local-only.

### Pitfall 4: Turning representative failures into a matrix
The requirements call for representative failure classes, not exhaustive permutations per command family.

### Pitfall 5: Pulling Phase 83 coverage tooling into Phase 82.1
The handoff proof should stay minimal. Full instrumentation/report plumbing belongs in Phase 83.

## Validation Architecture

- **Fast loop:** targeted `bun test tests/commands/<touched-file>.test.ts`
- **Wave loop:** `bun run test`
- **Plan gate:** `bun run typecheck`
- **Proof artifact:** one dedicated automated Istanbul smoke that exercises a single isolated subprocess path and writes mergeable coverage output

## Open Questions (RESOLVED)

- **How should `edit --yaml` fake the editor?** Use a committed executable stub, not a shell snippet.
- **Which CLI path is smallest for the Istanbul smoke?** `completion bash`, because it skips the git-version preflight and has minimal fixture needs.
- **Should prerequisite drift be silently absorbed?** No. Start the phase with explicit prerequisite verification and stop if Phase 80 / 81.1.1 surfaces are still missing.

## Recommendation

Plan Phase 82.1 as **three plan files**:

1. support success/output contracts
2. doctor/install/integration support plus remaining success surfaces
3. representative failures plus the dedicated Istanbul smoke

Each plan should begin by respecting prerequisite verification rather than assuming planned-but-unmerged earlier artifacts already exist on this branch.
