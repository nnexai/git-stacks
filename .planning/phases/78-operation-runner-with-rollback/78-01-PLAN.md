## Locked decisions from CONTEXT.md (verbatim — these MUST be implemented)

- **D-04:** Imperative API. Callers use `runner.do(name, forwardFn, undoFn)`. Each call executes `forwardFn` immediately and pushes `undoFn` onto an internal LIFO stack.
- **D-05:** On any throw from a forward step, the runner invokes `rollback()` which pops and executes undos in reverse order. Each undo is wrapped in its own try/catch — a failing undo does not abort the remaining undos (ENGN-03).
- **D-06:** The runner never throws for expected failures. It returns a discriminated union `{ ok: true } | { ok: false; error: string; rollbackErrors: string[] }`. `rollbackErrors` is empty on the happy path and on successful full rollbacks; non-empty only when individual undos failed.
- **D-07:** The runner accepts an optional `onProgress: (message: string) => void` callback and reuses the existing `ProgressCallback` type from `src/lib/workspace-ops.ts:38`. No new progress channel is introduced.
- **D-14:** Rollback progress messages flow through the same `onProgress` callback the caller already supplies. No `console.error` side writes.
- **D-15:** Message format conventions:
  - Forward-step progress: free-form (caller emits its own strings if it wants — runner does NOT auto-emit on successful forwards)
  - Rollback-step progress: prefixed `Rollback: ` (e.g. `"Rollback: removed worktree for foo"`)
  - Undo failures: prefixed `Rollback error: ` (e.g. `"Rollback error: removeWorktree foo failed (permission denied)"`)
- **D-16:** `rollbackErrors[]` in the return value carries the same undo-failure strings that were streamed through `onProgress`, so programmatic callers can inspect them without re-parsing stdout.
- **D-17:** The original forward-step error is the value of the `error` field in the returned `{ ok: false, error, rollbackErrors }`. Rollback errors never replace it — they're supplementary.

Required public exports (named exports only):

```typescript
import type { ProgressCallback } from "./workspace-ops"

export type RunnerResult =
  | { ok: true }
  | { ok: false; error: string; rollbackErrors: string[] }

export type Runner = {
  /**
   * Execute `forward` immediately. If it succeeds, push `undo` onto the LIFO stack.
   * If it throws, automatically run `rollback()` (LIFO undo of all previously-pushed
   * steps), then re-throw the original error so the caller's surrounding try/catch
   * can convert it into a RunnerResult via `result()`.
   *
   * The `name` is used in rollback progress messages ("Rollback: {name}") and in
   * undo-failure messages ("Rollback error: {name} failed ({err})").
   */
  do(name: string, forward: () => Promise<void>, undo: () => Promise<void>): Promise<void>

  /**
   * Returns the runner's accumulated state as a discriminated union. Caller invokes
   * this after the orchestrating try/catch — on success returns { ok: true }, on
   * failure returns { ok: false, error, rollbackErrors }.
   *
   * Implementation note: result() is the single source of truth for the discriminated
   * union; the do() method only manages the stack and rolls back on throw. This split
   * lets callers structure their orchestration as a normal try/catch without having
   * to reason about whether each `do` call returned ok or not.
   */
  result(): RunnerResult
}

export function createRunner(onProgress?: ProgressCallback): Runner
```

Internal data structure (Claude's discretion per CONTEXT.md):

```typescript
type Step = { name: string; undo: () => Promise<void> }
const stack: Step[] = []
let forwardError: string | null = null
const rollbackErrors: string[] = []
```

Implementation rules (verbatim from D-04, D-05, D-06, D-14, D-15, D-16, D-17):

1. `do(name, forward, undo)`:
   - `await forward()` — if it throws:
     - Capture the error message: `forwardError = err instanceof Error ? err.message : String(err)`
     - Run rollback (see step 2)
     - Re-throw the original error so the orchestrating caller's try/catch fires (caller then calls `result()` to get the discriminated union)
   - On success: `stack.push({ name, undo })` (push AFTER forward succeeds, never before — see D-04)

2. Rollback (internal helper, runs from `do` on throw, also exposed implicitly through `result()` — your call):
   - While `stack.length > 0`: `const step = stack.pop()!`
   - `onProgress?.(\`Rollback: \${step.name}\`)` — emit the rollback message FIRST (so users see the attempt even if it fails)
   - Wrap the undo in its own try/catch:
     - On success: continue to next step
     - On failure: build message `\`Rollback error: \${step.name} failed (\${err instanceof Error ? err.message : String(err)})\``, push to `rollbackErrors`, AND emit the same string through `onProgress?.(...)` (per D-16 — the array and the stream mirror each other)
   - Continue popping until stack is empty (best-effort: a failing undo does not break the loop)

3. `result()`:
   - If `forwardError === null` AND `rollbackErrors.length === 0`: return `{ ok: true }`
   - Otherwise: return `{ ok: false, error: forwardError ?? "unknown forward error", rollbackErrors: [...rollbackErrors] }`
   - (D-17: `error` is always the forward error — never replaced by a rollback error)

4. The runner does NOT auto-emit on successful forward steps. The caller emits its own free-form forward messages (D-15). The runner only owns the `Rollback: ...` and `Rollback error: ...` namespace.

5. The runner does NOT introduce a new `_exec` seam. All side effects are inside the caller's forward/undo closures, so the existing Phase 75 seams (`workspace-git._exec`, `lifecycle._exec`) cover failure injection in Plan 02's tests.

6. ESM module structure: imports at top, type definitions, then `createRunner` factory. No default export.

| Boundary | Description |
|----------|-------------|
| Caller→Runner | Internal in-process call. Caller controls forward/undo closures, step names, and the optional onProgress callback. No untrusted input crosses this boundary — all callers (Plan 02's createWorkspace, eventually wizard/dashboard) are first-party code. |
| Runner→onProgress | Runner emits strings into a caller-supplied callback. Caller controls how those strings are surfaced (TUI, log, test spy). |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-78-01 | Tampering | step name in rollback messages | accept | Step names originate from first-party code (Plan 02 hard-codes them). Even if a future caller passed user input, the message is informational only — not a control-flow signal. Low severity. |
| T-78-02 | DoS / Denial of Service | infinite rollback loop | mitigate | Rollback iterates by popping from a finite stack. Each pop is O(1); the loop terminates when `stack.length === 0`. No recursion. Verified by Test 4 (multiple failures all complete). |

| T-78-05 | Elevation of Privilege | runner running with elevated forward closure | accept | Runner does not execute shell, does not spawn processes, does not touch the filesystem on its own.
