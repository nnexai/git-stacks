# Phase 84.1: Coverage Report Accuracy and TUI Instrumentation Follow-up - Context

**Gathered:** 2026-05-14
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the Phase 83 coverage workflow so the reports are trustworthy enough to close v0.17.1. Existing tests that import `src/` files, including regular TypeScript modules and TUI TSX component tests, must contribute real Istanbul hits to the canonical coverage report instead of leaving source files at zero through blank-template-only accounting.

This phase repairs coverage reporting accuracy and the local gate that proves it. It does not add CI, introduce numeric coverage thresholds, replace the public coverage command surface, or expand into broad TUI E2E behavior coverage.

</domain>

<decisions>
## Implementation Decisions

### Instrumentation Strategy
- **D-01:** Planning must start with a small implementation spike comparing two repair paths: fixing the current preload redirect pipeline versus moving to load-time instrumentation.
- **D-02:** After the spike, choose the smallest reliable path that preserves the existing `coverage`, `coverage:unit`, and `coverage:integ` commands and the stable `.coverage/` artifact contract.
- **D-03:** The chosen approach must handle direct `tests/** -> src/**` imports, source-to-source relative imports, TypeScript, TSX, and CLI subprocess propagation.

### Report Shape
- **D-04:** `bun run coverage` remains the canonical full-suite coverage report.
- **D-05:** `bun run coverage:unit` and `bun run coverage:integ` remain scoped debugging aids only; maintainers should not have to manually piece together unit and integration reports to understand release readiness.
- **D-06:** Final outputs continue to be written to `.coverage/` using the Phase 83 stable artifact names.

### Accuracy Gates
- **D-07:** Local verification must fail when the report is structurally valid but known exercised source files still have zero hits.
- **D-08:** Add sentinel hit checks against `coverage-final.json`, including at least one regular source module such as `src/lib/messages.ts` and one representative TUI TSX file under `src/tui/dashboard/`.
- **D-09:** Artifact existence/parseability remains necessary but is not sufficient for Phase 84.1 release readiness.

### Scope Boundary
- **D-10:** In scope: coverage accounting for current unit/integration tests that already import `src/`, including existing TUI TSX component tests.
- **D-11:** Tiny dedicated coverage sentinel tests may be added if existing tests are too indirect or brittle for a reliable gate.
- **D-12:** Out of scope: broad TUI E2E coverage, prompt-driving new TUI behavior, CI workflow work, and mandatory numeric coverage thresholds.

### the agent's Discretion
- Exact spike structure and comparison evidence, as long as both viable instrumentation paths are tested against the observed zero-hit failure.
- Exact sentinel file choices, as long as they prove both normal TypeScript source imports and TUI TSX imports produce nonzero hits.
- Exact helper/module boundaries for the repaired coverage runner and gate checks, as long as the public command and artifact contracts stay stable.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `scripts/coverage-runner.ts` already instruments every `src/**/*.{ts,tsx}` file, builds blank templates, runs test files under coverage, merges shards, and writes text/HTML/JSON/summary/LCOV outputs.
- `scripts/coverage-preload.ts` already registers a Bun runtime plugin, rewrites selected source imports to `.coverage/src-inst`, rewrites CLI subprocess entrypoints, and writes per-process coverage shards on exit.
- `scripts/verify-gates.ts` already parses coverage artifacts during `bun run verify:gates`, making it the natural place for sentinel hit checks.
- Existing tests such as `tests/lib/messages.test.ts` and `tests/tui/dashboard/ActionMenu.test.tsx` are useful sentinel candidates because they pass while their corresponding source files currently remain zero-hit in generated coverage.

### Established Patterns
- The repo uses Bun and a custom test runner; normal `bun run test`, `test:unit`, and `test:integ` flows must stay non-coverage and avoid coverage overhead.
- Unit tests and integration tests are classified the same way in `scripts/test-runner.ts` and `scripts/coverage-runner.ts`; coverage fixes should keep these classifications aligned.
- Final coverage artifacts live in `.coverage/` and are ignored from git; scratch instrumented trees and shards are disposable implementation details.
- Gates aggregate actionable findings instead of fail-fast single-problem output.

### Integration Points
- The coverage spike should exercise `scripts/coverage-runner.ts` and `scripts/coverage-preload.ts` directly against a regular TS module and a TUI TSX module.
- The final gate should read `.coverage/coverage-final.json` and fail when sentinel files have zero statement/function hits after `bun run coverage`.
- `bun run verify` should continue to run coverage before gates, so the sentinel checks validate freshly generated artifacts.

</code_context>

<specifics>
## Specific Ideas

- Research confirmed the failure mode with focused probes: `bun run coverage:unit tests/tui/dashboard/ActionMenu.test.tsx` passes but leaves `src/tui/dashboard/ActionMenu.tsx` at zero hits, and `tests/lib/messages.test.ts` leaves `src/lib/messages.ts` at zero hits.
- The report can include all 100 source files through blank templates while still showing zero real execution, so “artifact exists and parses” is not enough.
- Bun official docs support the intended mechanism in principle: test preloads run before test files, and runtime plugins can intercept module resolution/loading. The current implementation needs a repo-specific proof that the hook actually redirects all relevant source imports.
- The user explicitly wants to avoid having to piece together coverage reports manually; the canonical `bun run coverage` report must be fixed instead.

</specifics>
