## Concrete Existing Patterns

### Coverage shard writer

`scripts/coverage-preload.ts` already writes `globalThis.__coverage__` to `${GS_COVERAGE_SHARD_DIR}/${pid}-${timestamp}-${random}.json` on process exit. Keep this pattern so every covered test or child process remains a mergeable Istanbul shard.

### Stable report artifact generation

`scripts/coverage-runner.ts` uses `istanbul-reports` to write:

- `.coverage/coverage-final.json`
- `.coverage/coverage-summary.json`
- `.coverage/lcov.info`
- `.coverage/index.html`

Do not rename these outputs. Phase 84 gates and docs already consume them.

### Aggregated gate reports

`scripts/verify-gates.ts` collects inventory drift, unmapped in-scope items, broken mapped tests, and coverage artifact problems into one `VerifyGateReport`. Add sentinel hit failures as another collection in this same shape instead of failing early while reading coverage JSON.

### Test fixture style

`tests/lib/verify-gates.test.ts` writes temporary `.coverage/` files under a temp root and calls `collectVerifyGateReport({ root, inventory, liveCommands })`. Reuse this pattern for sentinel tests by writing tiny Istanbul-shaped `coverage-final.json` fixtures with zero and nonzero counters.

## Landmines

- Do not run two `coverage-runner` processes at the same time; they share `.coverage/src-inst`, `.coverage/shards`, and `blank-templates.json`.
- Do not add CI or numeric threshold behavior.
- Do not make normal `bun run test*` commands use coverage.
- Do not intercept tests or scripts with instrumentation hooks; only `root/src/**` should be instrumented.
- For TSX source, return the `tsx` loader or dashboard tests can fail.
