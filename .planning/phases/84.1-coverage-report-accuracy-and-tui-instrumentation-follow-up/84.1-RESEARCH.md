# Phase 84.1: Coverage Report Accuracy and TUI Instrumentation Follow-up - Research

**Researched:** 2026-05-14
**Status:** Ready for planning

## Executive Summary

Phase 84.1 should be planned as a narrow coverage-accuracy repair over the existing Phase 83/84 surfaces, not as a new coverage policy or test runner rewrite. The current report can include all `src/**/*.{ts,tsx}` files through blank Istanbul templates while still recording zero real hits for tests that definitely execute source. Focused probes confirmed the failure:

- `bun run coverage:unit tests/lib/messages.test.ts --timeout-ms=30000` passes 13 tests but leaves `src/lib/messages.ts` with zero statement/function hits in `.coverage/coverage-final.json`.
- `bun run coverage:unit tests/tui/dashboard/ActionMenu.test.tsx --timeout-ms=30000` passes 11 tests but leaves `src/tui/dashboard/ActionMenu.tsx` with zero statement/function hits.

The likely cause is the preload redirect model. `scripts/coverage-preload.ts` depends on `build.onResolve()` patterns for source import specifiers, but direct test imports include aliases such as `@/lib/messages`, deeper relative paths such as `../../../src/tui/dashboard/ActionMenu`, and source-to-source imports such as `./paths`. Those are not reliably redirected into `.coverage/src-inst`, so the tests execute original `src/**` modules while the report merges blank templates.

The implementation should begin with a spike comparing two repair paths required by D-01:

1. **Repair current redirect pipeline:** keep pre-instrumenting into `.coverage/src-inst`, but use a resolved-file `onLoad` hook for real `root/src/**` modules so aliases, deep relative imports, and source-to-source imports load instrumented contents.
2. **Move to load-time instrumentation:** use a preload `onLoad` hook to instrument real `root/src/**` modules as Bun loads them, keep blank templates for untouched files, and keep the existing shard merge/report writers.

Both paths preserve `coverage`, `coverage:unit`, `coverage:integ`, and the stable `.coverage/coverage-final.json`, `.coverage/coverage-summary.json`, `.coverage/lcov.info`, and `.coverage/index.html` artifact contract.

## Current Implementation Findings

### Working surfaces to preserve

- `package.json` already exposes `coverage`, `coverage:unit`, `coverage:integ`, `verify`, `verify:prereqs`, and `verify:gates`.
- `scripts/coverage-runner.ts` already discovers tests, builds blank Istanbul templates for all source files, runs coverage-specific test subprocesses, merges shards, and writes stable report names.
- `scripts/coverage-preload.ts` already writes per-process coverage shards and rewrites CLI subprocess entrypoints.
- `scripts/verify-gates.ts` already parses `.coverage/coverage-final.json`, `.coverage/coverage-summary.json`, and `.coverage/lcov.info`; it is the natural home for sentinel hit checks.
- `tests/lib/coverage-surface.test.ts` already locks the public coverage script/artifact surface.

### Observed failure mode

The runner reports all source files at 0% when focused tests pass. For the current probes, `.coverage/coverage-final.json` contains entries for `src/lib/messages.ts` and `src/tui/dashboard/ActionMenu.tsx`, but their statement/function counters remain zero. This means artifact existence and parseability are insufficient release gates for Phase 84.1.

### Likely root cause

- The preload uses `build.onResolve()` redirects for import specifiers, but the relevant test/source imports are broader than the current filters.
- `tests/tui/dashboard/ActionMenu.test.tsx` imports `../../../src/tui/dashboard/ActionMenu`, while the preload only handles `../../src/` and `../src/`.
- `tests/lib/messages.test.ts` imports `@/lib/messages`; this alias path is not reliably caught by the current runtime resolver path.
- Source modules then import other source modules through `./...` paths, which are not generally redirected when the original real source file is loaded.
- The report merges blank template entries for files that were never loaded through instrumented code, so the final JSON is structurally valid but behaviorally false.

## Candidate Repair Paths

### Path A: resolved-file onLoad over pre-instrumented source

Keep `scripts/coverage-runner.ts` pre-instrumenting `src/**` into `.coverage/src-inst`, but change `scripts/coverage-preload.ts` so resolved real source files under `${GS_COVERAGE_ROOT}/src` load corresponding instrumented contents from `${GS_COVERAGE_SRC_INST}`.

Expected shape:

- Register a Bun plugin `onLoad` hook that only matches source file extensions.
- In the callback, normalize the resolved real path, verify it is inside `root/src`, map it to `.coverage/src-inst/<rel>`, and return the instrumented contents.
- Return loader `ts` for `.ts` files and `tsx` for `.tsx` files.
- Keep current shard writing and CLI entrypoint rewrite.

Strengths:

- Smallest change to the current architecture.
- Preserves the existing blank-template and merge/report pipeline.
- Avoids import-specifier depth/alias matching because Bun has already resolved the source file.

Risks:

- The callback must avoid intercepting tests, scripts, and non-source files.
- Returning `undefined` from overly broad `onLoad` hooks can break Bun module loading, so filter design matters.
- It still maintains a parallel `.coverage/src-inst` module graph.

### Path B: load-time instrumentation

Move instrumentation into `scripts/coverage-preload.ts` by instrumenting real `root/src/**/*.{ts,tsx}` files as Bun loads them. Keep `scripts/coverage-runner.ts` responsible for blank templates, test orchestration, shard merge, and final report generation.

Expected shape:

- Register a Bun plugin `onLoad` hook for resolved files under `root/src`.
- Read the original source file, instrument it with `istanbul-lib-instrument`, and return canonical Istanbul paths such as `src/lib/messages.ts`.
- Use parser plugins `["typescript"]` for TS and `["typescript", "jsx"]` for TSX.
- Preserve shard writes on process exit.
- Propagate the preload into CLI subprocesses rather than rewriting CLI execution into `.coverage/src-inst/index.ts`, if the spike proves this is reliable.

Strengths:

- Lets Bun resolve aliases, deep relative imports, and source-to-source imports normally.
- Avoids duplicated real-vs-instrumented module identities for most execution.
- Conceptually simpler once the preload hook is proven.

Risks:

- More internal change than Path A.
- Must confirm exact Bun preload injection for child `bun run src/index.ts` processes.
- Must handle cache-busted or query-string imports.
- Shell-string child process invocations may still need explicit handling if they exist.

## Recommended Planning Shape

Start with one blocking spike task that produces written evidence and chooses the smallest reliable path. The implementation task should then apply only the chosen approach. If neither candidate proves real hits for the sentinel files, execution should stop and write spike findings instead of guessing.

The likely winner is a resolved-file `onLoad` repair, either returning pre-instrumented contents from `.coverage/src-inst` or instrumenting at load time. The decisive criterion is not conceptual elegance; it is whether the focused probes produce nonzero counters for regular TS, TUI TSX, source-to-source imports, and CLI subprocess execution while preserving the existing coverage command and artifact contract.

## Validation Architecture

### Sentinel hit checks

Phase 84.1 needs a gate that distinguishes structurally valid coverage artifacts from trustworthy coverage artifacts. Add sentinel checks against `.coverage/coverage-final.json` for:

- `src/lib/messages.ts` after `tests/lib/messages.test.ts` or the full `bun run coverage` run.
- `src/tui/dashboard/ActionMenu.tsx` after `tests/tui/dashboard/ActionMenu.test.tsx` or the full `bun run coverage` run.
- At least one source-to-source dependency if the spike identifies a stable candidate, such as `src/lib/paths.ts`.
- At least one CLI subprocess-loaded source file if the final repair touches subprocess preload propagation.

Sentinel pass condition: the file exists in the coverage map and has a positive sum across statement counters or function counters. Do not add numeric percentage thresholds.

### Automated commands

- Focused spike probes:
  - `bun run coverage:unit tests/lib/messages.test.ts --timeout-ms=30000`
  - `bun run coverage:unit tests/tui/dashboard/ActionMenu.test.tsx --timeout-ms=30000`
- Sentinel inspections:
  - `bun --print 'const cov=await Bun.file(".coverage/coverage-final.json").json(); for (const f of ["src/lib/messages.ts","src/tui/dashboard/ActionMenu.tsx"]) { const e=cov[f]; const hits=e && (Object.values(e.s).some(Boolean) || Object.values(e.f).some(Boolean)); console.log(`${f}: ${hits ? "hit" : "zero"}`) }'`
- Unit regressions:
  - `bun test tests/lib/coverage-surface.test.ts -x`
  - `bun test tests/lib/verify-gates.test.ts -x`
- Gate/full verification:
  - `bun run coverage`
  - `bun run verify:gates`
  - `bun run verify`

## Threats and Pitfalls

- **False positive release gate:** Artifact JSON exists and parses but sentinel files stay at zero. Mitigate by adding hit checks in `scripts/verify-gates.ts`.
- **Concurrent coverage runs:** Multiple `coverage-runner` processes race on shared `.coverage/src-inst`, `.coverage/shards`, and `blank-templates.json`. Spike and verification commands must run serially.
- **Scope creep:** Do not add CI, mandatory thresholds, broad TUI E2E behavior coverage, or a replacement test runner.
- **TSX loader breakage:** Any preload load hook must return `tsx` for `.tsx` source or dashboard tests can fail.
- **Subprocess coverage regression:** Preserve or explicitly replace current CLI subprocess coverage propagation; do not fix unit imports while losing command/E2E subprocess accounting.

## Open Questions Resolved for Planning

- **Should the plan require a spike?** Yes. D-01 explicitly requires a small implementation spike because the repair path is not fully verified.
- **Can the plan pick a default path now?** It can identify likely candidates, but the executor should choose only after spike evidence proves sentinel hits.
- **Should `coverage:unit` and `coverage:integ` become separate release gates?** No. They remain debugging aids; `bun run coverage` remains canonical.
- **Should Phase 84.1 add thresholds?** No. Sentinel nonzero-hit checks are enough for this phase and preserve the deferred COVR-05 threshold policy.
