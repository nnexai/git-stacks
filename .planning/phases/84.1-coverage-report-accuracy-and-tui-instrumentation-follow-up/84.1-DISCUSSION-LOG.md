## Research First

The user asked to research first because the original plan did not account for the uncovered issue of piecing together coverage reports. The focused research found:

- `scripts/coverage-runner.ts` pre-instruments `src/**/*.{ts,tsx}` and generates valid Istanbul artifacts with blank templates.
- `scripts/coverage-preload.ts` tries to redirect source imports and CLI subprocesses to instrumented source.
- Focused probes showed tests can pass while their corresponding source files remain zero-hit:
  - `tests/tui/dashboard/ActionMenu.test.tsx` passed while `src/tui/dashboard/ActionMenu.tsx` stayed at zero hits.
  - `tests/lib/messages.test.ts` passed while `src/lib/messages.ts` stayed at zero hits.
- `scripts/verify-gates.ts` currently checks coverage artifact existence/parseability, not whether known exercised files received hits.

## Instrumentation Strategy

| Option | Description | Selected |
|--------|-------------|----------|
| Repair current preload redirect first | Keep the Phase 83 architecture, but broaden/fix Bun plugin resolution and subprocess propagation until direct `tests/** -> src/**`, source-to-source relative imports, TS, and TSX all hit instrumented files. | |
| Replace with load-time instrumentation | Move away from prewritten `.coverage/src-inst` redirects and instrument source at module load time through a broader Bun plugin hook, if research confirms it is more reliable. | |
| Planner decides after a spike | Require an implementation spike comparing both approaches, then choose the smallest reliable path that preserves the existing `coverage*` commands and artifacts. | ✓ |

**User's choice:** Planner decides after a spike.
**Notes:** The spike must compare current preload redirect repair against load-time instrumentation and preserve the existing public command/artifact contract.

## Report Shape and Piece-Together UX

| Option | Description | Selected |
|--------|-------------|----------|
| One merged report remains canonical | `bun run coverage` must produce the trustworthy full-suite report; `coverage:unit` and `coverage:integ` are only debugging aids, not things maintainers have to manually merge or compare. | ✓ |
| Split reports plus merged report | Keep the merged report, but also preserve separate unit and integration report summaries so maintainers can see where coverage came from. | |
| Explicit provenance in one report | Keep one merged report, but include lightweight provenance/check output showing which test buckets produced shards and which sentinel files received hits. | |

**User's choice:** One merged report remains canonical.
**Notes:** The user specifically called out not wanting manual report piecing. The full-suite report is the release-readiness surface.

## Trust Gates for Accuracy

| Option | Description | Selected |
|--------|-------------|----------|
| Sentinel hit checks | Add focused gate checks that fail when known exercised files, like `src/lib/messages.ts` and a representative `src/tui/dashboard/*.tsx`, remain zero-hit in `coverage-final.json`. | ✓ |
| Broad nonzero source check | Fail if the merged report has too few nonzero source files overall, without naming specific sentinel files. | |
| Artifact shape only after fixing | Fix the instrumentation pipeline but leave gates at parseability/existence only. | |

**User's choice:** Sentinel hit checks.
**Notes:** Phase 84.1 must prevent structurally valid but behaviorally false coverage reports from passing release verification.
