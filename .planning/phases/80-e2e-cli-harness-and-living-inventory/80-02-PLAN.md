<threat_model>
## Trust Boundaries

| Boundary | Description |
|---|---|
| command surface → inventory data | Manual inventory drift can hide required E2E scope |
| inventory module → later gates/phases | Duplicate IDs or bad mappings would corrupt later automation |
| exclusion metadata → planning scope | Missing exclusions can silently re-expand banned scope |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|---|---|---|---|---|
| T-80-05 | T | `tests/e2e-inventory.ts` ids and mappings | mitigate | Use stable string ids, export `getDuplicateInventoryIds()`, and enforce uniqueness in `tests/lib/e2e-inventory.test.ts`. |
| T-80-06 | T/R | exclusion metadata | mitigate | Ship first-class excluded entries for `exclude.tui.manage`, `exclude.integration.behavior`, all editor-launching exclusions, wizard-only exclusions, and `exclude.audit.rollback-visibility`, each with explicit rationale. |
| T-80-07 | T | unmapped in-scope detection | mitigate | Export `getUnmappedInScopeItems()` and `validateE2EInventory()`; test that only in-scope items with empty `mappedTests` are reported. |
| T-80-08 | R | inventory-to-test traceability | mitigate | Require `mappedTests` to contain repo-relative `tests/...` paths and test this invariant so later phases can update mappings deterministically. |

No high-severity threat remains after these mitigations.
</threat_model>
