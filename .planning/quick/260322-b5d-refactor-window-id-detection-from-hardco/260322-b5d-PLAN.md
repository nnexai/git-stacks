       - Update `WindowArtifact`: replace `niriWindowIds?: number[]` with `windowIds?: Record<string, number[]>`

       - In `runIntegrations()`, before the integration loop, collect all detectors: `const detectors = sorted.filter(i => i.windowDetector && i.isEnabled(ctx)).map(i => i.windowDetector!)`
       - Wrap the integration open() call: before calling `integration.open()`, call `begin()` on all detectors to capture pre-spawn snapshots. After `open()` returns, if the result is a `WindowArtifact` (kind === "window"), call `resolve()` on each detector with the snapshot and hints `{ pid: artifact.pid, app_id: artifact.app_id }`, then merge results into `artifact.windowIds` keyed by detector.id.
       - This way vscode/intellij remain unaware of detectors entirely -- the runner handles detection externally.

    3. In `src/lib/integrations/niri.ts`:
       - Add `windowDetector` property to `niriIntegration` that implements `WindowDetector`:
         - `id: "niri"`
         - `begin()`: calls `isNiriRunning()` -- if false, return a snapshot with empty set. If true, calls `listNiriWindows()` to get current window IDs, returns snapshot containing `Set<number>` of IDs.
         - `resolve(snapshot, hints?)`: calls `listNiriWindows()` again, diffs against snapshot set, returns new IDs. Uses the same exponential backoff polling loop that `snapshotWindowIds` currently uses (inline it or call snapshotWindowIds internally with a _listWindows wrapper).
         - Actually simpler: `begin()` returns a snapshot, and `resolve()` does the polling loop internally. The `snapshotWindowIds` function in `niri.ts` remains as-is (it's the low-level primitive). The detector's `begin()` captures state, then `resolve()` polls until new windows appear.
       - Update `open()` method: change `artifact.niriWindowIds` references to `artifact.windowIds?.["niri"]` when reading from the ArtifactBag for window moves (Step 2 of open).
       - For column config source windows: change `artifact.niriWindowIds` to `artifact.windowIds?.["niri"]`.
       - Remove direct `snapshotWindowIds` calls from open() for column app/command windows. Those still need detection. Since the column windows are spawned INSIDE niri's own open() (tier 3), the runner's detector wrapping won't help here. For column-spawned windows, niri's open() should call its own detector's begin/resolve directly (or continue using snapshotWindowIds directly since it IS the niri integration doing the spawning).

    IMPORTANT: The runner's detector wrapping handles detection for tier-1 integrations (vscode, intellij) -- they just spawn and the runner detects. For windows spawned inside niri's own open() (column app/command windows), niri's open() uses snapshotWindowIds directly as before -- this is internal to the niri integration and doesn't need the generic detector interface.
