# Quick Task 260403-x04: Make keychain resolver attribute names configurable - Context

**Gathered:** 2026-04-03
**Status:** Ready for planning

<domain>
## Task Boundary

Make the keychain resolver's attribute names configurable instead of hardcoded `service` and `account`. Currently `secret-tool lookup service X account Y` is hardcoded in `src/lib/secrets.ts:98`. Users should be able to specify arbitrary key-value attributes.

</domain>

<decisions>
## Implementation Decisions

### Attribute count
- Support arbitrary N key-value pairs, matching what `secret-tool` actually supports natively

### Path syntax
- Use `key=value` with comma delimiter: `${{ keychain:key1=val1,key2=val2 }}`
- Example: `${{ keychain:service=myapp,account=db }}` or `${{ keychain:app=myapp,env=prod,key=db }}`
- Backward compatibility: old `service/account` format (no `=` in value) should still work, mapping to `service=<first>,account=<second>`

### macOS parity
- macOS `security find-generic-password` only supports `-s` (service) and `-a` (account)
- Error on >2 attributes when running on macOS — fail fast with a clear message
- On macOS, the first pair maps to `-s` and the second to `-a`

</decisions>

<specifics>
## Specific Ideas

- `parseKeychainPath()` in `src/lib/secrets.ts` is the function to refactor
- Detect new vs legacy syntax by checking for `=` in the path string
- Legacy path `myapp/db` → `[{key: "service", value: "myapp"}, {key: "account", value: "db"}]`
- New path `service=myapp,account=db` → same result
- Linux: `secret-tool lookup` accepts any number of key-value pairs as positional args
- macOS: `security find-generic-password` needs exactly `-s service -a account`

</specifics>
