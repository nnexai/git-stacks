<objective>
Add concurrency limiting and re-entrancy guard to the useStaleness TUI hook to prevent spawning unbounded parallel git processes when a workspace has many repos.

Purpose: Workspaces with many repos currently fire all fetchOrigin + getCommitsBehind calls simultaneously via Promise.allSettled with no concurrency cap. This can overwhelm the system with dozens of concurrent git processes. A concurrency limiter (max 3) and a re-entrancy guard (skip if already fetching) solve this.

Output: A reusable `createLimiter` utility in `src/lib/concurrency.ts`, updated `useStaleness` hook using it, and unit tests for the limiter.
</objective>

Create `src/lib/concurrency.ts` exporting:

```typescript
export type LimiterResult<T> = PromiseSettledResult<T>

/**
 * Run an async function over each item with at most `limit` concurrent executions.
 * Returns results in input order, same shape as Promise.allSettled.
 */
export async function mapLimited<T, R>(
  items: T[],
  fn: (item: T) => Promise<R>,
  limit: number
): Promise<LimiterResult<R>[]>
```

Implementation: Use a simple slot counter + queue approach. Maintain a `running` count. For each item, if `running < limit`, start immediately and increment `running`. Otherwise, push to a queue. When a task finishes (success or failure), decrement `running` and dequeue the next if any. Collect results into a pre-allocated array indexed by input position so order is preserved.

Do NOT use a third-party library — this is a ~30-line utility. Do NOT use `p-limit` or similar. Keep it simple: no class, just a single exported async function.

Write tests in `tests/lib/concurrency-limiter.test.ts` using `bun:test`. Use a controllable async task (resolve/reject via deferred promises or setTimeout) to assert concurrency behavior:
- Track peak concurrency by incrementing/decrementing a counter around each task.
- Assert peak never exceeds limit.
- Assert all results returned in order.
- Assert a rejecting task does not stall the queue.
