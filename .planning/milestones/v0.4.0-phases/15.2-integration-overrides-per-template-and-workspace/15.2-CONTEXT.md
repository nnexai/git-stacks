# Phase 15.2: Integration Overrides per Template and Workspace - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Add interactive UX for configuring integration overrides at the template and workspace level. The schema already supports overrides (`template.integrations`, `workspace.settings.integrations`, `resolveEnabled()` cascade) — this phase makes them accessible via CLI wizards and visible in detail panes. No TUI dashboard editing — the TUI's existing YAML editor is sufficient for post-creation changes.

</domain>

<decisions>
## Implementation Decisions

### Template integration config
- **D-01:** Add "Override integrations?" yes/no prompt to both `template new` and `template edit` CLI wizards
- **D-02:** If user says yes, show multi-select pre-populated from current global config state (not blank)
- **D-03:** For each enabled integration, run the full `configurePrompt()` flow (same as `git-stacks config`)
- **D-04:** Only store overrides if user opted in — no integrations key in template YAML if they said no

### Workspace integration config
- **D-05:** Add "Override integrations?" yes/no prompt to `git-stacks new` and `git-stacks clone` CLI wizards
- **D-06:** Pre-selection starts from template's overrides (if template-based) or global config (if ad-hoc)
- **D-07:** Full `configurePrompt()` flow for each enabled integration, same as templates
- **D-08:** Add `git-stacks edit <workspace>` CLI command (paralleling `template edit`) for post-creation changes including integration overrides

### Display in detail panes
- **D-09:** TUI detail panes for templates and workspaces show all integrations with resolved enabled state
- **D-10:** Each integration shows source annotation: `[global]`, `[template]`, `[workspace]`, or `[skipped: reason]`
- **D-11:** Config values shown inline (e.g., `✓ vscode (cmd: code-insiders) [template]`)
- **D-12:** List views (CLI and TUI) show nothing about integrations — detail only

### Claude's Discretion
- Exact prompt wording and ordering within wizard flows
- How to handle the `applies()` check in display (e.g., IntelliJ skipped when no Java repos)
- Whether `edit <workspace>` reuses template-wizard patterns or has its own flow
- Error handling for invalid override combinations

</decisions>

<specifics>
## Specific Ideas

- Override prompt pattern should match the existing hooks prompt ("Override integrations? → yes → multi-select + configure") — consistent with how the template wizard already asks about hooks
- Cascade display format:
  ```
  Integrations:
    ✓ vscode  (cmd: code-insiders)  [template]
    ✗ tmux                          [workspace]
    ✓ cmux                          [global]
    - intellij                      [skipped: no java repos]
  ```

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `resolveEnabled()` / `resolveEnabledGlobally()` — already implement the full cascade; display code can use these to determine source
- `configurePrompt()` on each integration — already exists; template/workspace wizards just need to call the same methods
- `config.ts` multiselect pattern — `git-stacks config` already does "select integrations → configure each" which can be extracted/reused

### Established Patterns
- Template wizard hooks prompt — "add hooks?" → yes → configure. Integration override prompt should follow the same ask-first pattern
- Template integrations are deep-copied to workspace on creation (`JSON.parse(JSON.stringify(template.integrations))`)
- `editWorkspaceYaml()` in `workspace-ops.ts` — pattern for the new `edit` command to follow

### Integration Points
- `template-wizard.ts` — add integration override step after existing steps
- `workspace-wizard.ts` / `workspace-clone.ts` — add integration override step after template selection
- `commands/workspace.ts` — register new `edit` subcommand
- TUI detail pane components — add integration display section (read-only)

</code_context>
