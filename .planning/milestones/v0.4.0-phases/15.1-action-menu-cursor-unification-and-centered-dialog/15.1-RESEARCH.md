# Phase 15.1: Action Menu Cursor Unification and Centered Dialog - Research

**Researched:** 2026-03-21
**Domain:** OpenTUI/SolidJS terminal dialog layout and cursor navigation patterns
**Confidence:** HIGH

## Summary

This phase has two tightly coupled deliverables: (1) unify keyboard cursor navigation across all three action menus, and (2) convert every overlay/wizard/progress view from inline-in-detail-pane rendering to centered modal dialog boxes with dimmed backgrounds.

The cursor unification work is a direct copy-and-adapt of the pattern already proven in `ActionMenu.tsx`. The centering work exploits OpenTUI's flex layout system (`justifyContent="center"` + `alignItems="center"` on a full-screen container box with a `backgroundColor` dim layer) — the same flex primitives the renderer already uses for the split-pane layout. OpenTUI has no built-in modal/overlay component, but `<box>` with `backgroundColor`, `justifyContent`, and `alignItems` provides everything needed. The key architectural question is how centering boxes compose with App.tsx's height-based render pattern — the answer is that overlays must be rendered at the App root level (not inside the detail pane), parallel to the existing top/bottom pane structure, using `Show` conditions.

**Primary recommendation:** Create a shared `CenteredDialog` wrapper component that wraps any content in a full-screen dim container with a centered bordered box; use this for all 11 components being converted. Do not hand-roll per-component layering logic.

### Pattern 1: CenteredDialog Wrapper

**What:** A full-screen container that dims the background and centers a bordered box child. The dim layer is a full-screen `<box>` with `backgroundColor="#1a1a1a"` (or `#333333`). The centered dialog is a `<box>` with `justifyContent="center"` + `alignItems="center"` dimensions set per size variant.

**When to use:** Wrap every component that was previously rendered inline in the detail pane.

**Example:**
```typescript
// Source: @opentui/core Box.d.ts + Renderable.d.ts (confirmed available)
/** @jsxImportSource @opentui/solid */
type CenteredDialogProps = {
  title: string
  size?: "small" | "medium" | "large"
  children: JSX.Element
}

export function CenteredDialog(props: CenteredDialogProps) {
  const widthMap = { small: "50%", medium: "70%", large: "90%" }
  const heightMap = { small: "auto", medium: "60%", large: "90%" }
  const size = props.size ?? "small"

  return (
    // Full-screen dim layer — sits on top of everything
    <box
      height="100%"
      width="100%"
      backgroundColor="#1a1a1a"
      justifyContent="center"
      alignItems="center"
    >
      <box
        border
        title={` ${props.title} `}
        flexDirection="column"
        width={widthMap[size]}
        minWidth={40}
        backgroundColor="#000000"
      >
        {props.children}
      </box>
    </box>
  )
}
```

**Key insight on `shouldFill`:** `BoxRenderable`'s default options include `shouldFill: true`, meaning the box paints its `backgroundColor` across all its cells. The dim box fills the full terminal area at `height="100%"` `width="100%"` with the dark color.

### Pattern 2: Cursor Navigation for Action Menus

**What:** Add `createSignal(0)` cursor + arrow key handling in useKeyboard, then prefix each menu item with `"> "` (cyan) or `"  "` depending on cursor position.

**When to use:** RepoActionMenu and TemplateActionMenu need this added; ActionMenu already has it.

**Example (RepoActionMenu after update):**
```typescript
// Source: ActionMenu.tsx existing pattern — replicate exactly
const items = [
  { key: "w", action: "create-workspace" as const, label: () => wsLabel() },
  { key: "t", action: "create-template" as const, label: () => tplLabel() },
  { key: "r", action: "remove" as const, label: () => removeLabel() },
]
const [cursor, setCursor] = createSignal(0)

useKeyboard((key) => {
  if (key.name === "escape") { props.onCancel(); return }
  if (key.name === "down") { setCursor(c => Math.min(c + 1, items.length - 1)); return }
  if (key.name === "up") { setCursor(c => Math.max(c - 1, 0)); return }
  if (key.name === "return") {
    props.onAction(items[cursor()].action)
    return
  }
  // Letter shortcuts stay
  if (key.name === "w") { props.onAction("create-workspace"); return }
  if (key.name === "t") { props.onAction("create-template"); return }
  if (key.name === "r") { props.onAction("remove"); return }
})

// Render: cyan + "> " for cursor item
<For each={items}>
  {(item, i) => (
    <text fg={i() === cursor() ? "cyan" : "white"}>
      {i() === cursor() ? "> " : "  "}{item.label()}
    </text>
  )}
</For>
```

**Note on reactive labels in RepoActionMenu:** The existing `wsLabel()`, `tplLabel()`, `removeLabel()` are already arrow functions that depend on `props.selectionCount`. When placing them in a `For` items array, the label must remain a function (not a value) to stay reactive. The array must be defined INSIDE the component (not at module scope) so it closes over reactive signals.

### Pattern 3: App.tsx Overlay Promotion

**What:** Dialogs must escape the detail pane box and render at the App root level to achieve true full-screen coverage. The height-based visibility pattern still applies — `Show` conditions, not Switch/Match.

**Current state:** All overlays (except HelpOverlay and MessageOverlay) render inside the bottom detail pane `<box>`. After conversion, they must render at the root level, overlapping both panes.

**How to promote:** In App.tsx, add `Show` blocks at the root level (after the help/messages Shows, before the split-pane Show) for each centered dialog state.

```typescript
// Source: App.tsx current structure — extend this pattern
return (
  <box flexDirection="column" height="100%">
    {/* Layer 1: Help overlay (existing) */}
    <Show when={helpOpen()}>
      <HelpOverlay ... />
    </Show>

    {/* Layer 2: Message overlay (existing) */}
    <Show when={!helpOpen() && messagesOpen()}>
      <MessageOverlay ... />
      ...
    </Show>

    {/* Layer 3: NEW — Centered dialogs that overlay the split pane */}
    <Show when={!helpOpen() && !messagesOpen() && view().view === "action-menu"}>
      {/* ActionMenu as centered dialog */}
    </Show>
    <Show when={!helpOpen() && !messagesOpen() && view().view === "confirm"}>
      {/* ConfirmDialog as centered dialog */}
    </Show>
    {/* ...etc for all promoted views */}

    {/* Layer 4: Split pane (existing — always visible, dialogs render on top) */}
    <Show when={!helpOpen() && !messagesOpen()}>
      {/* Top box: lists */}
      {/* Bottom box: detail panes ONLY — remove promoted views from here */}
      {/* Help bar */}
    </Show>
  </box>
)
```

**Critical:** The height-based pattern means the split pane stays in the DOM at all times. Centered dialogs sit "on top" by rendering after the split pane in the component tree — OpenTUI's z-ordering respects DOM order (later children paint over earlier children).

### Anti-Patterns to Avoid

- **Rendering centered dialog INSIDE the detail pane box:** The dialog inherits the detail pane's height constraint and cannot cover the full screen. Move to root level.
- **Using `Switch`/`Match` to show/hide the split pane when dialog is active:** OpenTUI renderer doesn't repaint conditional DOM swaps (known project constraint). Use `Show` with the split pane always visible underneath.
- **Positioning with `position="absolute"` or `top`/`left` offset calculations:** Not needed. Flex centering with `justifyContent="center"` + `alignItems="center"` is simpler and reliable.
- **Per-component dim layers:** Extract to `CenteredDialog` wrapper so dim color/size is consistent and changed in one place.
- **Reactive labels in module-scope `items` arrays:** Any menu items array that references reactive props/signals must be defined inside the component function, not at module scope.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Modal/overlay centering | Custom position math with `top`/`left` offsets | `<box justifyContent="center" alignItems="center">` on a `height="100%"` container | Flex handles terminal resize automatically; position-based approach requires `useTerminalDimensions` recalculation |
| Background dimming | Drawing spaces over the existing content | `backgroundColor="#1a1a1a"` on a full-screen `<box>` with `shouldFill` (default true) | OpenTUI Box paints bg color across all cells by default |
| Per-dialog z-ordering | `zIndex` manipulation | DOM render order (later child = front) | OpenTUI renders children in declaration order; last child paints on top |
| Cursor in action menus | Custom selection state beyond a simple index signal | `createSignal(0)` + clamped arrow key handlers | ActionMenu already proves this; no new primitives needed |

**Key insight:** OpenTUI's `<box>` with flex alignment IS the modal system. There is no separate overlay or portal API — you simply render a full-screen colored box containing a smaller centered box, and it covers everything beneath it.

## Common Pitfalls

### Pitfall 1: Dialog Renders Inside Detail Pane — Clipped to Pane Height

**What goes wrong:** Component renders inside the bottom `<box flexGrow={2} minHeight={10}>` in App.tsx, so the "full screen" overlay is actually full-detail-pane-height, not full-terminal-height. Background dim doesn't cover the top list pane.

**Why it happens:** Currently ALL views (ActionMenu, ConfirmDialog, WizardView, etc.) are in the detail pane. This works for non-centering content but breaks the full-screen dim effect.

**How to avoid:** Promote all centered dialogs to the App root level, outside the split-pane `Show` block. The split pane remains visible underneath — the dim layer covers it visually.

**Warning signs:** Background color appears only in bottom half of terminal; top list pane is still visible/interactive.

### Pitfall 2: Reactive Labels in Module-Scope Items Array

**What goes wrong:** `const items = [{ label: props.selectionCount > 0 ? ... : ... }]` at module scope captures `props.selectionCount` at module load time, not at render time. Labels never update reactively.

**Why it happens:** `props` in SolidJS are reactive getters, but only when accessed inside reactive contexts (effects, memos, JSX). Module-scope array definitions run once at import, not on render.

**How to avoid:** Define items arrays inside the component function body (as done in ActionMenu.tsx for `fullActions`). For RepoActionMenu, the existing `wsLabel()` etc. arrow functions already solve this — move the items array inside the component.

**Warning signs:** Label text shows stale value (e.g., old selectionCount) when component re-renders.

### Pitfall 3: useKeyboard Global Broadcast — Dialog Not Blocking Background Keys

**What goes wrong:** When a centered dialog is "open" (via `Show`), the underlying list view's `useKeyboard` handlers are still active. Arrow keys meant for dialog cursor navigation also fire list navigation.

**Why it happens:** `useKeyboard` is a global broadcast — ALL registered handlers see ALL keys simultaneously. Components are not isolated just because they are shown/hidden.

**How to avoid:** The existing pattern in App.tsx uses view-state guards:
```typescript
if (v.view === "action-menu") return  // ActionMenu handles its own keyboard
if (v.view === "confirm") return       // ConfirmDialog handles its own keyboard
```
These guards must remain in App.tsx's main `useKeyboard` handler for every promoted dialog state. When promoting views out of the detail pane, verify the corresponding guard is present in App.tsx.

**Warning signs:** Pressing arrow keys in an action menu also moves the list cursor behind the dialog.

### Pitfall 4: Dim Layer Background Color Inheritance

**What goes wrong:** The dim `<box>` renders but the terminal area behind it shows through because the inner dialog box inherits the dim color and `shouldFill` paints over it.

**Why it happens:** Both the outer dim box and inner dialog box have `shouldFill: true` by default. The inner dialog needs an explicit `backgroundColor` (e.g., `"#000000"` or the terminal default) so it doesn't show the dim color as its background.

**How to avoid:** Set `backgroundColor="#000000"` (or a dark neutral) explicitly on the inner centered dialog box.

**Warning signs:** Dialog content appears on a dark gray background that matches the dim layer, rather than on a distinct panel.

### Pitfall 5: HelpOverlay and MessageOverlay Already Use height="100%"/width="100%" — Structural Mismatch

**What goes wrong:** HelpOverlay currently renders as a full-screen bordered box directly. If it gets wrapped in a CenteredDialog, you get a box inside a box inside a box, with the outer dim covering the inner dialog's title/border.

**Why it happens:** HelpOverlay was designed as a direct full-screen replacement, not as content inside a centered container.

**How to avoid:** For HelpOverlay and MessageOverlay, use `size="large"` in CenteredDialog (90% width/height). Strip the existing `height="100%"` `width="100%"` from their outer `<box>` — let CenteredDialog control sizing. The inner content box in HelpOverlay can use `flexGrow={1}` to fill the CenteredDialog container.

**Warning signs:** Title appears twice (once from inner component's border, once from CenteredDialog); or content is clipped because two nested "full height" boxes conflict.

### Pitfall 6: Title Prop Routing for HelpOverlay/MessageOverlay

**What goes wrong:** D-07 requires titles like "Keybindings" for HelpOverlay and `${workspaceName} — Messages` for MessageOverlay. If these components render their own `border title={...}` AND CenteredDialog adds another border with its own title, the border is doubled.

**How to avoid:** CenteredDialog owns the border and title. Components pass their title via a `title` prop to CenteredDialog and remove their own `border` attribute. Components render only their internal content (text lines, lists) without an outer bordered box.

### Pitfall 7: InlineInput in Centered Dialog — Input Focus Timing

**What goes wrong:** When InlineInput becomes a centered dialog, the `focused={props.focused ?? true}` default means the underlying input is focused immediately, but the deferred-focus pattern (setTimeout 0) used in WizardView may not apply if the dialog is activated by a keypress.

**Why it happens:** The activating keypress (e.g., `n` for rename) can leak into the input as the first character if focus is set synchronously.

**How to avoid:** When App.tsx transitions to `{ view: "inline-input" }`, apply the same deferred focus pattern used for WizardView:
```typescript
// In App.tsx action handler:
setInlineInputFocused(false)
setView({ view: "inline-input", ... })
setTimeout(() => setInlineInputFocused(true), 0)
```
InlineInput already has a `focused` prop for this purpose.

## Open Questions

1. **Split-pane always-visible during dialogs**
   - What we know: Height-based visibility (not Switch/Match) is required due to OpenTUI renderer repaint constraint. The split pane will be visible behind the dim layer.
   - What's unclear: Whether OpenTUI's z-ordering ensures later DOM children always visually cover earlier ones, or if there's a zIndex prop needed.
   - Recommendation: Trust DOM order first (confirmed by existing HelpOverlay behavior which already overlays the split pane using a full-screen box rendered before the split pane's Show block). If z-ordering issues appear, use `zIndex={10}` on the CenteredDialog outer box (confirmed available in `Renderable.d.ts`).

2. **`minWidth` on inner dialog box**
   - What we know: `minWidth` is a confirmed `LayoutOption` in `Renderable.d.ts`.
   - What's unclear: Whether `50%` width on a narrow terminal leaves the inner dialog too small for content.
   - Recommendation: Planner should choose `minWidth={40}` (characters) for small dialogs as a floor. Exact value is Claude's discretion per CONTEXT.md.
