# Phase 15.1: Action Menu Cursor Unification and Centered Dialog - Context

**Gathered:** 2026-03-21
**Status:** Ready for planning

<domain>
## Phase Boundary

Unify selection cursor across all three tabs' action menus and convert all overlay/dialog/wizard/progress views to centered dialog boxes with dimmed backgrounds. List views, detail panes, filter input, and BatchBar stay as-is.

</domain>

<decisions>
## Implementation Decisions

### Cursor unification
- **D-01:** RepoActionMenu and TemplateActionMenu get `cursor`/`setCursor` with arrow key navigation + Enter to select, matching ActionMenu's existing pattern
- **D-02:** Letter-key shortcuts remain as accelerators alongside cursor navigation
- **D-03:** Cursor clamps at top/bottom (no wrap) ‚Äî same behavior as list navigation without multi-select
- **D-04:** Highlight style: cyan fg + `"> "` prefix for focused item, same across all three menus
- **D-05:** Default cursor position is always index 0

### Centered dialog layout
- **D-06:** All dialogs render as bordered boxes with title, centered on screen using `justifyContent`/`alignItems` on a full-screen container
- **D-07:** Title text is the entity name ‚Äî e.g., workspace name for ActionMenu, template name for TemplateActionMenu, repo name for RepoActionMenu, "Confirm" for ConfirmDialog, "Keybindings" for HelpOverlay
- **D-08:** Background dimming via full-screen `backgroundColor` box (dark gray, e.g., `#333333`) rendered behind the dialog
- **D-09:** Small dialogs (action menus, confirm, remove-blocked, inline rename/clone) use percentage-based width with a minimum size, auto-sized to content where practical
- **D-10:** Large overlays (HelpOverlay, MessageOverlay) use near-full-screen centered box ‚Äî inset slightly from edges but use most of the available space

### Component scope ‚Äî what becomes a centered dialog
- **D-11:** Action menus: ActionMenu, RepoActionMenu, TemplateActionMenu (small)
- **D-12:** Dialogs: ConfirmDialog, RemoveBlockedView (small)
- **D-13:** Large overlays: HelpOverlay, MessageOverlay (near-full-screen)
- **D-14:** Wizards: WizardView (small-to-medium)
- **D-15:** Progress views: SyncProgressView, CreateProgressView, ProgressView (small-to-medium)
- **D-16:** Inline input: InlineInput rename/clone becomes a small centered dialog

### What stays as-is
- **D-17:** List views (WorkspaceList, TemplateList, RepoList) ‚Äî no change
- **D-18:** Detail panes (WorkspaceDetail, TemplateDetail, RepoDetail) ‚Äî no change
- **D-19:** Filter input ‚Äî stays inline in footer area
- **D-20:** BatchBar ‚Äî stays as footer bar

### Claude's Discretion
- Exact percentage/min-width values for small vs large dialogs
- Whether to extract a shared `CenteredDialog` wrapper component or inline the layout per component
- Dim color exact hex value
- Transition/mount behavior details

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ActionMenu.tsx` cursor pattern (createSignal + useKeyboard + arrow nav + Enter) ‚Äî replicate in Repo/TemplateActionMenu
- `useTerminalDimensions()` ‚Äî already used in App.tsx and MessageOverlay for reactive width/height
- OpenTUI `justifyContent`/`alignItems` flex properties ‚Äî available for centering

### Established Patterns
- Height-based tab visibility over Switch/Match (OpenTUI renderer doesn't repaint conditional DOM swaps)
- `useKeyboard` global broadcast ‚Äî input-mode guard must be above navigation handlers
- Deferred focus via `setTimeout(0)` for inputs activated by keypress

### Integration Points
- App.tsx orchestrates all view states via `UIView` union ‚Äî dialog rendering likely wraps existing view branches
- Each component currently renders inline in the detail pane ‚Äî needs to render in a full-screen centered container instead
