## Spacing Scale

Terminal UIs measure spacing in character cells (columns and rows), not pixels. The following character-cell scale maps the 8-point concept to terminal conventions already in use across the codebase.

| Token | Value | Terminal Unit | Usage |
|-------|-------|---------------|-------|
| xs | 1 col | `paddingLeft={1}` | Glyph-to-text gap within a row |
| sm | 2 col | `paddingLeft={2}` | Standard left indent — matches ConfirmDialog, ActionMenu |
| md | 1 row | `paddingTop={1}` | Vertical gap between sections within a view |
| lg | 2 row | `paddingTop={2}` | Major section separation — matches ConfirmDialog top padding |

Source: Directly read from `ConfirmDialog.tsx` (`paddingTop={2} paddingLeft={2}`), `ActionMenu.tsx` (`paddingTop={1} paddingLeft={2}`), `ProgressView.tsx` (inline padding via leading spaces `"  "`).

Exceptions: Status table rows are always `height={1}` (single character row) — no row-height exceptions needed.

## Typography

Terminal UIs have one monospace font. "Typography" in this context means character weight and color — not font sizes, which the terminal host controls.

| Role | OpenTUI Color | Weight Equivalent | Usage |
|------|--------------|-------------------|-------|
| Primary text | `fg="white"` | Normal | Repo names, workspace names, branch names, active label text |
| Secondary / metadata | `fg="gray"` | Dimmed | Skip reasons, commit counts, inactive hints, `[Esc] Back` lines |
| Heading / prompt | `fg="yellow"` | Emphasized | Confirm dialog prompt message (matches ConfirmDialog pattern) |
| Accent / active | `fg="cyan"` | Emphasized | Spinner color, active cursor highlight, primary action indicators |
| Success | `fg="green"` | Emphasized | Done message, synced repo checkmark |
| Warning | `fg="yellow"` | Emphasized | Skipped repo indicator, warning glyph |
| Error / failure | `fg="red"` | Emphasized | Failed repo indicator |

Source: `ProgressView.tsx`, `ConfirmDialog.tsx`, `ActionMenu.tsx`, `StatusIndicator.tsx`, `WorkspaceRow.tsx` — all read directly.

Line height: not configurable in terminal — always 1 row per `<text>` element.

## Status Glyphs

Phase 12 introduces a per-repo status table with in-place updates. The glyph set must be consistent with `StatusIndicator.tsx`.

| SyncRow State | Glyph | Color | Meaning |
|---------------|-------|-------|---------|
| `pending` | `·` | `fg="gray"` | Waiting — matches StatusIndicator pending glyph |

| `rebasing` | spinner `dots` | `color="cyan"` | Active rebase in progress |

| `skipped` | `⚠` | `fg="yellow"` | Conflict or fetch failure — skipped with best-effort |

## Interaction Contract

### Confirm step (D-07, D-08)

- Trigger: User presses `s` in ActionMenu
- Component: `ConfirmDialog`
- Message: `Sync {workspaceName}? (rebase from upstream)`
- Keys: `[y] Yes  [n/Esc] No`
- Color: `fg="yellow"` for prompt (matches existing pattern)

### Sync progress view

- Trigger: User presses `y` in confirm dialog
- Component: `SyncProgressView`
- View state: `{ view: "sync-progress"; message: "Syncing..." }`
- Keyboard: All keys blocked during sync (D-11). Guard: `if (v.view === "sync-progress") return` in App.tsx keyboard handler, placed above all navigation handlers.
- Post-sync dismiss: Any key press when `done === true` → reload workspace list + `setView({ view: "list" })`

### SyncProgressView layout

```
[spinner dots cyan]  Syncing...          ← shown while done === false

  · repo-a                               ← pending
  [spinner] repo-b  fetching...          ← active (fetching or rebasing)
  ✓ repo-c  +3 commits                   ← synced: show commit count
  ⚠ repo-d  conflict: src/lib/git.ts     ← skipped: show first conflict file
             src/index.ts                ← additional conflict files, indented 5 cols
  ✗ repo-e  fetch failed (timeout)       ← failed: show reason

  3 synced, 1 skipped, 1 failed. Press any key to continue.   ← shown when done === true
```

Layout rules:
- Each repo row: `height={1}`, `flexDirection="row"`
- Status glyph column: 2 characters wide (glyph + 1 space) — `paddingLeft={2}` from left edge
- Repo name: left-aligned, no padding — `fg="white"`
- Detail text: 2 spaces after repo name — `fg="gray"`
- Conflict file list: indented 5 columns from left edge (`"     "` prefix) — `fg="gray"` — one file per row below the skipped repo row
- Summary line: blank line separator + summary text — `fg="green"` on success, `fg="yellow"` if any skipped, `fg="red"` if any failed

### Summary line logic

- All synced (0 skipped, 0 failed): `fg="green"` — `"{N} synced. Press any key to continue."`
- Some skipped, none failed: `fg="yellow"` — `"{N} synced, {N} skipped. Press any key to continue."`
- Any failed: `fg="red"` — `"{N} synced, {N} skipped, {N} failed. Press any key to continue."`

## Interaction States Required

From WS-01 through WS-04 requirements:

| State | Trigger | Visual |
|-------|---------|--------|
| ActionMenu with Sync visible | User opens action menu on any workspace | `[s] Sync` entry at bottom of action list |
| Confirm dialog | User presses `s` | ConfirmDialog with sync-specific message |
| Sync progress — in-flight | User confirms | SyncProgressView with spinner + per-repo rows updating in-place |
| Sync progress — repo active | syncWorkspace onProgress callback fires | Row transitions from `·` (pending) to spinner (fetching/rebasing) |
| Sync progress — repo done | Repo completes | Row transitions to `✓` (synced) or `⚠` (skipped) |
| Sync progress — done | syncWorkspace resolves | Spinner hidden; summary line appears; any key dismisses |
| Sync error — timeout | fetchOrigin times out after 30s | Repo row shows `✗ fetch failed (timeout)` — D-10, WS-04 |
| Sync error — hard fail | syncWorkspace throws | Full-view error message — `fg="red"` |

## Implementation Notes for Executor

1. `SyncProgressView` receives `rows: SyncRow[]` and `done: boolean` and `summary: string` as props. The parent (`App.tsx`) owns the signals.

2. `SyncRow` type:
   ```typescript
   type SyncRow = {
     repo: string
     status: "pending" | "fetching" | "rebasing" | "synced" | "skipped" | "failed"
     detail: string        // commit count "+N commits", skip reason "conflict: file.ts", or ""
     conflicts: string[]   // additional conflict files to list beneath the row
   }
   ```

3. The `onProgress` callback from `syncWorkspace` currently receives `string` messages. For structured in-place updates, `App.tsx` must maintain a `SyncRow[]` signal and update individual rows by matching `repo` name from the progress payload. Alternatively, a structured callback `onSyncProgress: (update: SyncRow) => void` can replace the string callback — preferred for type safety.

4. Keyboard guard placement in `App.tsx` `useKeyboard`:
   ```typescript
   if (v.view === "sync-progress" && syncDone()) { /* dismiss on any key */ return }
   if (v.view === "sync-progress") return   // block all keys during sync
   ```
   This guard goes above ALL navigation handlers — same pattern as existing `v.view === "progress"` guard.

5. The `{ view: "sync-progress"; message: string }` UIView variant requires a `<Show>` branch in `App.tsx` render tree, rendering `<SyncProgressView>` instead of `<ProgressView>`.

6. Do not use `createStore` for `SyncRow[]` — simple `createSignal` with array replacement is sufficient for sequential sync. Rows update one at a time.
