<objective>
Fix the "open now?" lifecycle bypass in both workspace-wizard.ts and workspace-clone.ts, and fix the `{} as Workspace` type contract gap in the new flow's file ops path.

Purpose: When users create a workspace and answer "yes" to "Open workspace now?", the immediate open currently only runs integration.open() directly -- skipping post_open hooks, per-repo pre_open hooks, file-ops re-application, writeEnvFiles, trunk branch checks, and last_opened timestamp. This means the first open after creation is incomplete compared to subsequent `ws open` calls. The type contract fix ensures workspace-level file ops at creation time receive a properly-typed workspace object instead of `{} as Workspace`.

Output: Two fixed TUI files that delegate to openWorkspace() for the full lifecycle.
</objective>

   The fix: Move the workspace object construction ABOVE the file ops block. Build it as a proper typed object instead of `Record<string, unknown>`. Then pass it to `applyFileOpsForWorkspace`. The write to YAML stays in its current position.

   Specifically:
   a. Move the `settingsIntegrations` computation (lines 401-403) above the file ops spinner block.
   b. Build the workspace object before file ops:
   ```typescript
   const workspaceObj: Workspace = {

     branch,
     created: new Date().toISOString().split("T")[0],
     ...(templateName ? { template: templateName } : {}),
     ...(wsHooks ? { hooks: wsHooks } : {}),
     repos,
     ...(wsEnv ? { env: wsEnv } : {}),
     ...(wsEnvFile ? { env_file: wsEnvFile } : {}),
     ...(wsFiles ? { files: wsFiles } : {}),
     ...settingsIntegrations,
   } as Workspace
   ```
   c. Replace the `{} as Workspace` call:

   d. Replace the later `const workspace: Record<string, unknown> = { ... }` block and `writeWorkspace(workspace as Workspace)` with:
