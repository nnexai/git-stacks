# Milestone v0.3.0: Dashboard UI Overhaul

**Status:** ✅ SHIPPED 2026-03-20
**Phases:** 6–9
**Total Plans:** 13

## Overview

Transform the minimal OpenTUI dashboard into management central — full CRUD for all entities, a workspace notification system for AI agent hooks, and comprehensive shell completions. Introduced the `git-stacks message` subcommand family (JSONL-backed, IPC-pushed), rewrote the dashboard as a tabbed split-pane layout, and completed shell completion coverage across all shells.

## Milestone Summary

**Key Decisions:**

- JSONL files (one per workspace) as message store — not a WorkspaceSchema field; avoids concurrent write corruption from agents
- Single global Unix socket at `/tmp/git-stacks.sock` (not per-workspace); all messages carry `workspace` field for routing
- OPTION_ENUMS static table in completion-generator.ts (not Commander `.choices()`) — avoids unintended runtime validation behavior change
- `tab` signal is independent of `UIView` — existing view-state components (ActionMenu, ConfirmDialog, ProgressView, DetailStatus) untouched
- Use Switch/Match (then height-based visibility) for mutually-exclusive tab content — prevents SolidJS retaining inactive tab DOM nodes
- Two-box layout with flexGrow ratios (3:2) replaces manual innerHeight/detailHeight memos
- No nested `<text>` in OpenTUI — TextRenderable.add() rejects TextRenderable children; use sibling `<text>` in `<box flexDirection="row">`
- useMessages creates new Map on every IPC push (not mutate-in-place) — SolidJS identity check requires reference change to trigger reactivity
- Always unlink stale socket on startup — eliminates hang-prone Bun.connect probe

**Issues Resolved:**

- Tab switching freeze root-caused to nested `<text>` elements in TemplateList/RepoList (OpenTUI crash, not SolidJS rendering issue)
- SolidJS reactivity gaps in `<For>` callbacks — resolved with reactive function pattern `const fn = () => expr`
- Socket client data handler required by Bun API (even for write-only clients)

**Issues Deferred:**

- Branch completions for `new --from <branch>` (CMPL-F01) — requires design spike on repo-context resolution at completion time
- Windows IPC support (MSG-F01) — TCP fallback deferred to v0.4.0+
- In-TUI template creation wizard (DASH-F01) — renderer bridging complexity

**Technical Debt Incurred:**

- Dashboard component test coverage is thin — integration tests rely on TUI running; unit tests for pure utilities (messageUtils) exist but component render tests absent
- `git-stacks manage` keyboard help text describes `r` as reload but actual key is lowercase `r` (OpenTUI provides lowercase names)
