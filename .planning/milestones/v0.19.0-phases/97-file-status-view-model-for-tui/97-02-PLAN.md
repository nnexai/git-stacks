<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dashboard hook -> shared lib helper | TUI code consumes the grouped file-status model and must not bypass it with subprocesses or duplicated logic. |
| selected workspace state -> async loader | User selection and workspace lifecycle changes can invalidate the requested workspace while a load is in flight. |
| dashboard startup -> detail load path | File-status work must remain outside the initial workspace-list loading loop. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-97-03 | Denial of Service | `useWorkspaceFileStatus.ts` | mitigate | Keep the hook selected-workspace-only and independent from `useWorkspaces()` so Phase 97 does not add N-workspace filesystem scans to dashboard startup. |
| T-97-04 | Elevation of Privilege | dashboard file-status loading path | mitigate | Import the shared lib helper directly and forbid CLI/subprocess execution in production code, preventing command injection or shell-behavior drift from entering the TUI load path. |
| T-97-05 | Repudiation | dashboard error state | mitigate | Return explicit idle/loading/loaded/error states with clear messages so later UI rendering can surface failure cause instead of swallowing it. |
| T-97-SC | Tampering | npm/pip/cargo installs | mitigate | No package-manager installs are part of this plan; rely on existing Solid/OpenTUI test utilities already present in the repo. |
</threat_model>
