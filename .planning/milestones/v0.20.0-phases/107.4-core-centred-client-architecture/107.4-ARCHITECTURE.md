# Core-Centred Client Architecture

```text
workspace YAML / Git / filesystem / processes / signals
                         |
                  Bun local service
             +-----------+-----------+
             |                       |
   trusted official client      browser BFF projection
             |                       |
        OpenTUI renderer          web renderer + xterm
```

## Authority

The service owns all durable configuration reads/writes, Git and file inspection, workspace orchestration, configured commands, signal state, and browser PTYs. Both clients submit intent and render projections.

## Contracts

- `service/contract.ts`: protocol identities, snapshots, operations, events, signals, and launch plans.
- `service/core-contract.ts`: complete trusted local projection and mutation request schemas.
- `service/web/contract.ts`: browser-safe projection and browser principal requests.
- `service/client.ts`: official authenticated client, operation waiting, and SSE subscription.

## Deliberate client-local handoffs

The TUI must temporarily yield its controlling terminal for `$EDITOR` and for a foreground workspace shell. The service resolves edit targets and all workspace context first; only terminal attachment remains local. These handoffs are isolated and excluded explicitly by the architecture gate.

## Incremental web completion

Future web features should add service capabilities once, then expose a browser-safe adapter and browser rendering. They must not reimplement machine behavior in the web client or reintroduce direct machine behavior in the OpenTUI dashboard.
