# Phase 107.4 Summary

The service is now the single machine-side authority for both interactive clients.

- `/v1/core` provides trusted first-party clients with workspace definitions, authoritative projections, templates, repository registrations, and global configuration.
- One typed official client owns authentication, core queries, signals, SSE events, idempotent operations, progress, file status, notes, and edit-target resolution.
- The OpenTUI dashboard no longer reads persisted configuration, inspects Git/files, or executes domain mutations directly.
- All existing dashboard actions run through the operation registry, including structured sync, push, configured-command, and issue output.
- Browser and OpenTUI code share contract types, signal presentation rules, provider deduplication, relative time, and priority ordering.
- Browser configured commands now execute complete pre/main/post plans rather than rejecting plans with multiple steps.
- Operation progress uses SSE with durable-operation fallback on disconnect, and revision-aware invalidation avoids duplicate full-state reloads.
- Trusted clients validate discovery and launch a detached shared service; TUI teardown aborts its own event stream without owning the service lifetime.
- TUI detail data is cached by workspace/revision, signal events update locally, integrations remain reactive to the core projection, and operation SSE readiness is immediate.
- The TUI owns an adaptive list/detail split and native detail scrolling; the service continues to provide complete data rather than presentation-sized pages.
- Signal dismissal applies to notifications and activity, is persisted by shared signal state, and is filtered consistently by both clients.
- Foreground editor and attached-shell execution remain explicit, isolated presentation handoffs.
- Architecture tests enforce the authority boundary and cross-client contract reuse.
