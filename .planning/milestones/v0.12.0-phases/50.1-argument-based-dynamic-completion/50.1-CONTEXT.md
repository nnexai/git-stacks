# Phase 50.1: Argument-Based Dynamic Completion - Context

**Gathered:** 2026-04-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the manually maintained `DYNAMIC_COMPLETIONS` static map with convention-based inference from Commander.js argument names. The completion generator should automatically derive completion types from argument names (`<workspace>`, `[template]`, `<repo>`, etc.), support per-argument-position completions, and extract `.choices()` from Commander arguments. New completion types added: integration IDs. The existing `DYNAMIC_COMPLETIONS` map shrinks to an override layer for edge cases only.

</domain>

<decisions>
## Implementation Decisions

### Argument-name inference
- **D-01:** Convention-based inference — `buildNode()` reads all `registeredArguments` from Commander.js commands and maps argument names to `DynamicCompletion` types via naming convention: `workspace` → workspace, `template` → template, `repo` → repo, `shell` → shells, `integration` → integration.
- **D-02:** `DYNAMIC_COMPLETIONS` becomes an override layer. Priority order: (1) explicit `DYNAMIC_COMPLETIONS[path]` override, (2) convention inference from argument name, (3) Commander `.choices()` extraction, (4) null (no completion). Most entries (~37 of ~40) get deleted since inference handles them.

### New completion types
- **D-03:** Add `"integration"` as a new `DynamicCompletion` type. Completes from the list of registered integration IDs (vscode, tmux, aerospace, niri, etc.) read from the integration registry.
- **D-04:** When Commander arguments have `.choices()` or `.argChoices()`, extract those as fixed-value completion lists. This generalizes beyond the fixed type enum — any argument with choices auto-completes.

### Multi-position completion
- **D-05:** Replace `dynamic: DynamicCompletion | null` with `argCompletions: ArgCompletion[]` on `CommandNode`. Each entry carries: `{ name: string, type: DynamicCompletion, required: boolean }`. `buildNode()` iterates ALL `registeredArguments`, not just `[0]`.
- **D-06:** Shell generators (bash, zsh, fish) updated to emit position-aware completion — complete different types based on which argument position the cursor is at.

### Migration strategy
- **D-07:** Override layer approach — convention inference runs first, `DYNAMIC_COMPLETIONS` overrides for edge cases (e.g., `completion` command where arg is `<shell>` → mapped to `shells` type). Most entries deleted.
- **D-08:** `OPTION_ENUMS`, `FLAG_COMPLETIONS`, and `COMMAND_FLAG_COMPLETIONS` remain unchanged — they handle flag-value completions, orthogonal to positional argument inference.

### Claude's Discretion
- Whether to rename `DynamicCompletion` type or keep the existing name
- Exact convention matching logic (exact match vs contains vs regex)
- How integration ID completions are generated in shell scripts (grep/hardcoded/function)
- Test approach for validating generated completion scripts

</decisions>

## Existing Code Insights

### Reusable Assets
- `buildNode()` in completion-generator.ts already reads `cmd.registeredArguments[0]` for `firstArgRequired` — extend to read all args and their names
- `bashDynamicLookup()`, `_${id}_workspaces/repos/templates` zsh helpers, `__${id}_workspaces/repos/templates` fish functions — extend pattern for new `integration` type
- Integration registry in `src/lib/integrations/index.ts` — source of IDs for completion generation

### Established Patterns
- Shell-specific dynamic lookup helpers follow consistent pattern across bash/zsh/fish — new types follow same structure
- `OPTION_ENUMS` handles fixed-value flag completions — Commander `.choices()` on arguments is analogous for positional args
- `COMMAND_FLAG_COMPLETIONS` for per-command flag overrides — similar override concept for `DYNAMIC_COMPLETIONS`

### Integration Points
- `buildNode()` — primary inference point; reads Commander command metadata
- `CommandNode` interface — structural change from `dynamic` to `argCompletions`
- All three shell generators (bash, zsh, fish) — must handle positional array instead of single dynamic type
- `DYNAMIC_COMPLETIONS` — shrinks to 2-3 override entries
