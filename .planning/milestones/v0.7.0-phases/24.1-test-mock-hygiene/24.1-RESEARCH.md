**Assessment for files without `@/tui/utils` mock (`integration-commands.test.ts`, `niri.test.ts`, `tmux.test.ts`):**

- `integration-commands.test.ts` tests `src/commands/integration.ts` ‚Äî which only imports from `@/lib/integrations/index`. At test run time, `@/lib/integrations/tmux.ts` is imported (it uses `prompts as p from @/tui/utils`), but the tests only exercise command structure (names, subcommand existence) ‚Äî no prompt calls are triggered. Dead mock causes no issues because no prompt path is exercised.

- `niri.test.ts` tests `niriIntegration` ‚Äî it calls `open()` and `cleanup()` paths, which use `p.spinner()` and `p.log` from `@/tui/utils`. Since there is no `@/tui/utils` mock, Bun will use the real `@/tui/utils` module, which in turn imports `@clack/prompts`. **The dead `@clack/prompts` mock does intercept these calls indirectly** ‚Äî when `niriIntegration.open()` calls `p.spinner()`, it routes through `@/tui/utils.prompts.spinner`, which is loaded from the real module, which loaded `@clack/prompts`'s mock. So the mock chain IS functioning, but only incidentally. If someone removes the `@clack/prompts` mock without adding a `@/tui/utils` mock, `p.spinner()` would call the real clack spinner, which might behave unexpectedly in tests.

- `tmux.test.ts` (integrations) tests `tmuxIntegration` ‚Äî same situation as niri. `open()` calls `p.spinner()` from `@/tui/utils.prompts`. Same indirect mock chain applies.

### Category 2: Incomplete `@/tui/utils` mock in `artifacts.test.ts`

**Verified: Confirmed.** `artifacts.test.ts` line 68-70 mocks `@/tui/utils` with only `safeText`. No `prompts` object.

```typescript
mock.module("@/tui/utils", () => ({
  safeText: mock(() => Promise.resolve("")),
}))
```

**Does this cause actual test failures?** No. Here is why:

The tests in `artifacts.test.ts` exercise only `open()` on each integration. Looking at the production code:
- `vscodeIntegration.open()` ‚Äî uses `p.spinner()` and `p.isCancel()` only in `configurePrompt()`, not `open()`. The `open()` path uses no prompts.
- `cmuxIntegration.open()` ‚Äî uses `p.spinner()` and `p.log.warn()`. BUT the spinner is gated: `const spinner = ctx.silent ? null : p.spinner()`. The test's `fakeCtx` does not set `silent`, so `ctx.silent` is undefined (falsy on a plain object), meaning... `p.spinner()` WILL be called.

Wait ‚Äî `cmuxIntegration.open()` calls `p.spinner()` from `@/tui/utils`, but `artifacts.test.ts` mocks `@/tui/utils` without a `prompts` object. When the test runs `cmuxIntegration.open(fakeCtx, null, emptyBag)`, it will try to call `prompts.spinner()` from the mocked `@/tui/utils` ‚Äî but `prompts` is `undefined` in the mock, causing `undefined.spinner is not a function`.

**Yet the tests pass.** The explanation: the tests use cache-busting imports (`?artifacts-test` query params). Bun's module cache means the `@/lib/integrations/cmux?artifacts-test` module gets the mocked `@/tui/utils`. But `@clack/prompts` is also mocked with a spinner (`spinner: () => mockSpinner`). The `@/tui/utils` module itself was likely cached earlier when `@clack/prompts` was already mocked ‚Äî so `prompts.spinner` in the already-loaded `@/tui/utils` module object points to the mocked clack spinner, not through the `@/tui/utils` mock. This is a mock-cache ordering artifact.

**The latent bug is real**: if test isolation changes or run order changes, `prompts` being missing from the `@/tui/utils` mock could cause `p.spinner()` failures in `cmuxIntegration.open()` and `tmuxIntegration.open()` tests.

## Real Impact Assessment

### Test correctness risk: LOW-MEDIUM

Dead `@clack/prompts` mocks do not cause false positives (tests passing when they shouldn't). They also don't mask real failures. However:

- `niri.test.ts` and `tests/lib/integrations/tmux.test.ts` have a genuine **fragile mock chain**: their `p.spinner()` calls work only because the module load order happens to bind the clack spinner mock into the `@/tui/utils` `prompts` object before the integration module loads. If the cache-busting import forces a fresh `@/tui/utils` load after the `@clack/prompts` mock is registered, it will pick up the mock. But if `@/tui/utils` loads first (before `@clack/prompts` is mocked), it will bind the real clack functions. This is **order-dependent and fragile**.

- `artifacts.test.ts` has the confirmed incomplete mock for `@/tui/utils`. The missing `prompts` object creates a latent failure risk as described above.

### Future breakage risk: MEDIUM

If someone adds a test that runs these files in isolation (without the implicit `@clack/prompts` mock chain established by another test), `niri.test.ts` and `integrations/tmux.test.ts` could break on `p.spinner()` calls. This is a real but non-immediate risk.

The dead `@clack/prompts` mocks also create confusion: a future developer looking at `niri.test.ts` will see `mock.module("@clack/prompts")` and reasonably believe it is the active mock for prompts calls in the niri integration ‚Äî but it isn't. The real interception happens through `@/tui/utils`.

### Maintenance cost: LOW

7 files √ó ~15 lines each of dead mock code = ~105 lines of misleading noise. Not catastrophic, but anyone who needs to debug a test failure in these files will be misled about how prompts are mocked.

### Effort required: SMALL

Based on actual file inspection:

- **Category 1 (dead mocks):** Remove 1 `mock.module("@clack/prompts", ...)` block per file = 7 deletions. For `niri.test.ts`, `integration-commands.test.ts`, and `integrations/tmux.test.ts` ‚Äî which have no `@/tui/utils` mock ‚Äî a replacement mock for `@/tui/utils` with `prompts` object should be added (to make the mock chain explicit). Estimated: 30 minutes.
- **Category 2 (incomplete mock):** Add `prompts` object to `artifacts.test.ts` `@/tui/utils` mock. 15 lines. Estimated: 5 minutes.
- **Category 3:** Does not exist. Zero effort needed.

Total estimated effort: ~35-40 minutes of careful editing.

### Overall verdict: PHASE IS WORTH DOING ‚Äî but scope needs correction

**Do it, but discard Category 3.** The phase is small, the risk of the dead mocks increasing is real (future tests could rely on the fragile mock chain), and the incomplete `artifacts.test.ts` mock is a genuine latent bug. The effort is small enough that deferring indefinitely is not a defensible choice.

However, the CONTEXT.md should not be trusted verbatim. Category 3 is entirely fabricated (or was fixed during Phase 24 execution). The planner must re-verify actual file state rather than executing against the CONTEXT.md claims.
