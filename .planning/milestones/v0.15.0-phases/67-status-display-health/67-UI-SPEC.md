## Spacing Scale

Terminal UIs use character-cell spacing, not pixel spacing. The 8-point scale maps to character units as follows for this phase:

| Token | Char Value | Usage |
|-------|------------|-------|
| xs | 1 char | Icon-to-text gap (e.g. `✓  name`) |
| sm | 2 chars | Column separator (`  `) between fields in a row |
| md | 4 chars | Indent level for repo rows under workspace header |
| lg | 8 chars | Not used in this phase |
| xl | — | Not used in this phase |

**CLI stdout layout (status command):**

```
\n  {workspace_name}  [{branch}]  {created}
    {icon}  {name.padEnd(28)} {modeLabel}{abStr}
```

- Workspace header: 2-char left indent
- Repo row: 4-char left indent, 2-char gap between icon and name
- `name.padEnd(28)` — name field fixed at 28 chars to align modeLabel column
- `abStr` appended with 2-char leading spaces when present

**TUI WorkspaceRow layout (pre-existing, no new columns):**

- Columns: `prefix(5)` + `space(1)` + `StatusIndicator(1)` + `space(1)` + `name(dynamic)` + `space(1)` + `branch(dynamic)` + `a/b indicators(dynamic)` + `labels(dynamic)` + `  countsText` + `  messagePreview`
- `drCount` appended to `countsText` only when `> 0`, format: ` {N}dir`

**TUI WorkspaceDetail layout (pre-existing indent pattern):**

- Repo rows: 4-char indent, same `{icon}  {name.padEnd(28)} {modeLabel}` pattern as CLI

Exceptions: none.

## Typography

Terminal typography is determined by the user's terminal. The app controls only text content and ANSI color via OpenTUI's `fg` prop (CLI uses `chalk` zero-dependency ANSI). The following defines the semantic text roles and their rendering in this phase:

| Role | Char Width | Weight/Style | Usage |
|------|-----------|--------------|-------|
| Section label | any | bold (fg white) | `  Repos:`, `  Messages:`, `  Integrations:` — pre-existing |
| Body / data | any | normal (fg gray or white) | Repo names, branch names, paths |
| Success indicator | 1 | normal (fg green) | `✓` icon — repo exists and clean |
| Warning indicator | 1 | normal (fg yellow) | `~` icon — repo dirty |
| Error indicator | 1 | normal (fg red) | `✗` icon — repo missing |
| Mode label | varies | normal (fg same as body) | `[worktree]`, `[trunk]`, `[dir]` — inline with repo name |

**Line height:** 1 terminal row per element. OpenTUI `height={1}` is mandatory for all `<box flexDirection="row">` elements (existing pattern; do not change).

## Color

Terminal color palette via OpenTUI `fg` prop and ANSI codes. Existing palette inherited from the codebase — this phase extends it with one new semantic usage.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | terminal default bg | All surfaces (no explicit background except focused row) |
| Secondary (30%) | `#333333` | Focused row background in WorkspaceRow — pre-existing |
| Success (semantic) | `fg="green"` | `✓` icon, ahead indicator, enabled integration checkmark |
| Warning (semantic) | `fg="yellow"` | `~` dirty icon, behind indicator |
| Error (semantic) | `fg="red"` | `✗` missing icon, doctor fail icon, error messages |
| Muted / secondary text | `fg="gray"` | Unfocused rows, secondary info, timestamps, mode labels when not the primary focus |
| Accent / highlight | `fg="cyan"` | Branch name column in WorkspaceRow, linked issue tracker IDs |
| Neutral body | `fg="white"` | Section labels, workspace name, repo names |

**New in Phase 67 — `[dir]` mode label color:**

The `[dir]` label uses `fg="gray"` (matching the visual weight of `[trunk]`). Do not use a distinct accent color — dir repos are not a special call-to-action, just a different mode type. Consistency with `[trunk]` weight is the goal.

**Accent reserved for:**
- Branch name column in `WorkspaceRow` (pre-existing `fg="cyan"`)
- Linked issue tracker IDs in `WorkspaceDetail` (pre-existing `fg="cyan"`)
- Loading spinner in `StatusIndicator` (pre-existing `color="cyan"`)

No new accent usages are added in this phase.

**Destructive color:** `fg="red"` — used only for missing/error states and doctor `fail` icon. Not used for `[dir]` label.

## Copywriting Contract

### CLI: `status` command output

| Element | Copy |
|---------|------|
| Dir repo mode label | `[dir]` |
| Dir repo: icon when path exists | `✓` (same as other modes) |
| Dir repo: icon when path missing | `✗` (same as other modes) |
| Dir repo: ahead/behind columns | omitted entirely — no trailing whitespace placeholder, no dash |
| Dir repo: dirty indicator | `✗` on missing path; `✓` if path exists (dir has no dirty state — always clean if present) |

**Status human-readable row format for dir repo:**

```
    ✓  {repo_name.padEnd(28)} [dir]
```

No `abStr` appended for dir repos. The `abStr` is built from `repo.ahead > 0 || repo.behind > 0` checks — for dir repos both are always 0 (already zeroed in `getWorkspaceStatus`), so `abStr` will be `""` with no code change needed beyond the modeLabel fix.

### CLI: `list` command output

| Element | Copy |
|---------|------|
| `repoCount` field | Unchanged — dir repos already counted in `info.repoCount` via `getWorkspaceListInfo` |
| No new dir-specific columns in list | DISP-02 only requires no git aggregation errors — data layer already correct |

Empty state (list with no workspaces): `No workspaces. Run \`git-stacks new\` to create one.` — pre-existing, unchanged.

Empty state (list filtered by label with no matches): `No workspaces match labels: {labels}` — pre-existing, unchanged.

### TUI: `WorkspaceDetail` — repo mode label

| Element | Copy |
|---------|------|
| Dir repo mode label | `[dir]` |
| Dir repo: no ahead/behind badges | Ahead/behind badges already gated on `repo.mode === "worktree"` — dir repos show no badges |

### TUI: `WorkspaceRow` — counts text

| Element | Copy / Format |
|---------|--------------|
| countsText with dir repos (when drCount > 0) | `{wtCount}wt {trCount}tr {drCount}dir` |
| countsText with dir repos (when drCount === 0) | `{wtCount}wt {trCount}tr` (unchanged) |
| countsText with dirty repos | `{wtCount}wt {trCount}tr {drCount}dir ~{dirty}` (dir count before dirty) |
| countsText with no dir repos | `{wtCount}wt {trCount}tr ~{dirty}` (unchanged) |

### Doctor: new check function

| Element | Copy |
|---------|------|
| Issue message: dir path missing | `dir repo '{repo.name}' path missing: {repo.main_path}` |
| Issue fix message: dir path missing | `Check that the directory exists: {repo.main_path}` |
| Issue message: path exists but is a file | `dir repo '{repo.name}' path is not a directory: {repo.main_path}` |
| Fix action for both | `{ action: "info", message: "..." }` — no automated fix (mirrors existing pattern for `findMissingMainClones`) |
| Doctor icon for both | `"fail"` (existing `icon` values: `"fail"` / `"warn"` / `"ok"`) |

### `hasMissing` — `useWorkspaces.ts`

| Element | Decision |
|---------|----------|
| Dir repos with missing paths | Included in `hasMissing` — a missing dir path is as broken as a missing worktree |
| Updated predicate | `!r.exists && (r.mode === "worktree" \|\| r.mode === "dir")` |

Source: RESEARCH.md Recommendation for open question 1. Consistent with HLTH-02 semantics.

### Error states

| State | Message |
|-------|---------|
| TUI status load error | `Error: {message}` — pre-existing pattern in `WorkspaceDetail`, unchanged |
| Doctor check — `findMissingMainClones` guards dir repos | Mode guard added: dir repos route exclusively to `findInvalidDirRepos`, not to the clone-missing check |

### Destructive actions in this phase

None. Phase 67 is read-only display and health checks. No destructive operations are added.

## Interaction Contracts

### CLI `status` — dir repo rows

- Dir repos render on their own row in the per-workspace repo list
- Row format: `    {icon}  {name.padEnd(28)} [dir]` — no trailing a/b fields
- `icon` is `✓` if `repo.exists === true`, `✗` if `repo.exists === false`
- `dirty` is always `false` for dir repos (already zeroed in data layer) — no `~` icon will appear
- `--fetch` flag: dir repos are excluded from the `fetchOrigin` call via `r.mode !== "dir"` guard added to the allRepos filter at workspace.ts:363

### TUI WorkspaceDetail — dir repo rendering

- Extend ternary at line 66: `repo.mode === "worktree" ? \`[\${repo.branch}]\` : repo.mode === "dir" ? "[dir]" : "[trunk]"`
- Ahead/behind `<text>` badges already gated on `repo.mode === "worktree"` — no change needed, dir repos will produce no badges by default
- `fg` color for icon follows the same logic as other modes: red for missing, green for clean (dir repos are always "clean" if they exist)

### TUI WorkspaceRow — countsText

- Add reactive `const drCount = () => ws().repos.filter((r) => r.mode === "dir").length`
- Extend `countsText` memo: append ` {N}dir` only when `drCount() > 0`
- `countsWidth` memo derives from `countsText().length + 2` — no change needed, it reads the text dynamically

### TUI StatusIndicator — no change

- `hasMissing` now includes dir repos with missing paths — the red `✗` indicator will appear for workspaces where a dir repo's path has been deleted
- No code change in `StatusIndicator.tsx` itself; the behavior change comes from the updated `hasMissing` predicate in `useWorkspaces.ts`

### Doctor — `findInvalidDirRepos`

- Checks: (1) `existsSync(repo.main_path)` — path exists, (2) `statSync(repo.main_path).isDirectory()` — is actually a directory
- Two-step check mandatory (statSync throws ENOENT if path missing — check existsSync first)
- Issues returned as `{ icon: "fail", entity: ws.name, message: "...", fix: { action: "info", message: "..." } }`
- `statSync` must be imported in `doctor.ts` (currently only `existsSync` is imported)
- Function is called alongside existing checks in the main doctor action handler

### Doctor — `findMissingMainClones` guard

- Add `repo.mode !== "dir"` condition: `if (!existsSync(repo.main_path) && repo.mode !== "dir")`
- Prevents dir repos from generating a misleading "run `git-stacks repo show`" remediation message
