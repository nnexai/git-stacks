  <action>
    Add `ensureUpstreamTracking` to the import from `./git` in workspace-ops.ts. Then add upstream tracking for ALL worktree repos (not just missing ones) inside `openWorkspace()`, between worktree recreation (line 735) and the pre_open hooks (line 739).

    **Location:** After the "Recreate missing worktrees" block (ends at line 735) and before `const baseEnv = buildBaseEnv(...)` (line 737):

    ```typescript
    // Ensure upstream tracking for all worktree repos (parallel)
    const worktreeReposForTracking = workspace.repos.filter(
      (r) => r.mode === "worktree" && existsSync(r.task_path)
    )
    if (worktreeReposForTracking.length > 0) {
      const trackingResults = await Promise.all(
        worktreeReposForTracking.map(repo =>
          ensureUpstreamTracking(repo.main_path, workspace.branch)
        )
      )
      const tracked = trackingResults.filter(r => r.tracked)
      if (tracked.length > 0) {
        onProgress?.(`Upstream tracking set for ${tracked.length} repo(s)`)
      }
    }
    ```

    Key details:
    - Filter for `mode === "worktree"` AND `existsSync(r.task_path)` — skip trunk repos and repos whose worktrees don't exist yet (edge case: missing worktree recreation failed silently)
    - Use `repo.main_path` (not `task_path`) as the repoPath argument, consistent with createWorktree pattern
    - Use `workspace.branch` for the branch name
    - Run in parallel with `Promise.all()` per locked decision
    - Use `onProgress?.()` for feedback — this integrates with both CLI (`console.log`) and TUI (`runHooksCaptured`) callers
    - Only log when tracking was actually set (skip noise for brand-new branches or already-tracked repos)

    **Do NOT add tracking to `renameWorkspace()`** — the research notes this as a stretch goal since the branch was already tracked before the rename. The rename recreates worktrees but the branch tracking configuration is stored in the main repo's `.git/config`, which is unaffected by worktree removal/recreation.
